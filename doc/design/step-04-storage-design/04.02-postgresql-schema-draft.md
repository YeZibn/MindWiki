# 4.2 PostgreSQL 表结构草案

## 任务目标

为个人知识库 RAG 项目第一阶段设计 PostgreSQL 主数据库中的核心表结构草案，明确主要表、字段、类型建议、主键与外键方向，为后续 SQL DDL、迁移脚本和实现落地提供基础。

本子任务对应路线图中的 `4.2`：

- 设计 documents、chunks、tags、links、jobs 等核心表草案

## 用户确认结论

基于当前讨论，`4.2` 采用以下正式原则：

- 表结构粒度采用 `设计级`
- 第一阶段正式列出以下 10 张核心表：
  - `sources`
  - `documents`
  - `sections`
  - `chunks`
  - `tags`
  - `document_tags`
  - `import_jobs`
  - `query_records`
  - `generated_artifacts`
  - `citations`
- `chunks` 表中保留完整文本
- `generated_artifacts` 与 `citations` 采用通用表设计，不按类型拆表

## 设计原则

### 1. PostgreSQL 只承载结构化主数据

PostgreSQL 作为第一阶段主数据源，负责：

- 结构化业务实体
- 关系映射
- 状态管理
- 引用链
- 查询记录
- 生成结果

### 2. 不在本步骤中写死完整 SQL DDL

本文件提供表结构设计草案、字段类型建议和约束方向，但不直接替代最终迁移脚本。

### 3. 字段命名保持统一

建议统一采用：

- 主键：`id`
- 创建时间：`created_at`
- 更新时间：`updated_at`
- 删除标记：后续如需要，可统一采用 `deleted_at`

## 核心表清单

第一阶段 PostgreSQL 主库建议包含以下 10 张表：

1. `sources`
2. `documents`
3. `sections`
4. `chunks`
5. `tags`
6. `document_tags`
7. `import_jobs`
8. `query_records`
9. `generated_artifacts`
10. `citations`

## 1. `sources`

### 表职责

保存文档来源上下文信息，用于描述资料从哪里来、如何进入系统。

### 主要字段草案

| 字段名 | 类型建议 | 说明 |
| --- | --- | --- |
| `id` | `uuid` | 主键 |
| `source_type` | `varchar(50)` | 来源类型，如 `markdown`、`pdf` |
| `source_uri` | `text` | 来源 URI 或标准来源标识 |
| `file_path` | `text` | 本地文件路径 |
| `import_method` | `varchar(50)` | 导入方式，如 `upload`、`directory_scan` |
| `source_note` | `text` | 来源备注 |
| `is_active` | `boolean` | 来源是否有效 |
| `created_at` | `timestamp` | 创建时间 |
| `updated_at` | `timestamp` | 更新时间 |

### 约束方向

- 主键：`id`
- 不建议按单字段做强唯一
- 可在后续对 `source_type + file_path + import_method` 做业务唯一约束评估

## 2. `documents`

### 表职责

保存知识库中的文档主记录，是文档级元数据和状态的核心表。

### 主要字段草案

| 字段名 | 类型建议 | 说明 |
| --- | --- | --- |
| `id` | `uuid` | 主键 |
| `source_id` | `uuid` | 关联 `sources.id` |
| `title` | `text` | 文档标题 |
| `document_type` | `varchar(50)` | 文档类型，如 `markdown`、`pdf` |
| `content_hash` | `varchar(128)` | 文档内容哈希 |
| `source_native_key` | `text` | 来源侧原生标识，可空 |
| `file_path` | `text` | 文件路径快照 |
| `summary` | `text` | 文档级摘要，可空 |
| `status` | `varchar(50)` | 文档状态，如 `active`、`failed`、`deleted` |
| `imported_at` | `timestamp` | 导入时间 |
| `created_at` | `timestamp` | 创建时间 |
| `updated_at` | `timestamp` | 更新时间 |
| `deleted_at` | `timestamp` | 逻辑删除时间，可空 |

### 约束方向

- 主键：`id`
- 外键：`source_id -> sources.id`
- 建议后续评估唯一键：
  - `source_id + content_hash`
  - `source_id + source_native_key`

## 3. `sections`

### 表职责

保存文档结构层级信息，用于表达标题分段和内容组织关系。

### 主要字段草案

| 字段名 | 类型建议 | 说明 |
| --- | --- | --- |
| `id` | `uuid` | 主键 |
| `document_id` | `uuid` | 关联 `documents.id` |
| `parent_section_id` | `uuid` | 父 section，可空 |
| `title` | `text` | section 标题 |
| `level` | `integer` | 标题层级 |
| `order_index` | `integer` | 在文档中的顺序 |
| `start_offset` | `integer` | 起始偏移，可空 |
| `end_offset` | `integer` | 结束偏移，可空 |
| `created_at` | `timestamp` | 创建时间 |
| `updated_at` | `timestamp` | 更新时间 |

### 约束方向

- 主键：`id`
- 外键：
  - `document_id -> documents.id`
  - `parent_section_id -> sections.id`（可空）

## 4. `chunks`

### 表职责

保存最小检索单元、引用单元和上下文构建单元。

### 主要字段草案

| 字段名 | 类型建议 | 说明 |
| --- | --- | --- |
| `id` | `uuid` | 主键 |
| `document_id` | `uuid` | 关联 `documents.id` |
| `section_id` | `uuid` | 关联 `sections.id`，可空 |
| `chunk_index` | `integer` | 在文档中的顺序编号 |
| `content_text` | `text` | chunk 完整文本 |
| `token_count` | `integer` | token 数，可空 |
| `start_offset` | `integer` | 起始偏移，可空 |
| `end_offset` | `integer` | 结束偏移，可空 |
| `page_number` | `integer` | PDF 页码，可空 |
| `embedding_ref` | `text` | 对应 Qdrant 向量记录引用 |
| `created_at` | `timestamp` | 创建时间 |
| `updated_at` | `timestamp` | 更新时间 |
| `deleted_at` | `timestamp` | 逻辑删除时间，可空 |

### 约束方向

- 主键：`id`
- 外键：
  - `document_id -> documents.id`
  - `section_id -> sections.id`（可空）
- `document_id` 必填，不允许孤立 chunk

## 5. `tags`

### 表职责

保存标签主数据。

### 主要字段草案

| 字段名 | 类型建议 | 说明 |
| --- | --- | --- |
| `id` | `uuid` | 主键 |
| `name` | `varchar(100)` | 标签名 |
| `tag_source` | `varchar(50)` | 标签来源，如 `manual`、`auto` |
| `description` | `text` | 标签描述，可空 |
| `created_at` | `timestamp` | 创建时间 |
| `updated_at` | `timestamp` | 更新时间 |

### 约束方向

- 主键：`id`
- 唯一键：`name`

## 6. `document_tags`

### 表职责

保存文档与标签之间的关联关系。

### 主要字段草案

| 字段名 | 类型建议 | 说明 |
| --- | --- | --- |
| `id` | `uuid` | 主键 |
| `document_id` | `uuid` | 关联 `documents.id` |
| `tag_id` | `uuid` | 关联 `tags.id` |
| `created_at` | `timestamp` | 创建时间 |

### 约束方向

- 主键：`id`
- 外键：
  - `document_id -> documents.id`
  - `tag_id -> tags.id`
- 后续可评估联合唯一：
  - `document_id + tag_id`

## 7. `import_jobs`

### 表职责

保存导入任务执行记录和状态。

### 主要字段草案

| 字段名 | 类型建议 | 说明 |
| --- | --- | --- |
| `id` | `uuid` | 主键 |
| `source_id` | `uuid` | 关联 `sources.id`，可空 |
| `job_type` | `varchar(50)` | 任务类型 |
| `status` | `varchar(50)` | 状态，如 `pending`、`running`、`success`、`failed` |
| `input_payload` | `jsonb` | 输入载荷 |
| `error_message` | `text` | 错误信息，可空 |

| `finished_at` | `timestamp` | 结束时间，可空 |

| `updated_at` | `timestamp` | 更新时间 |

### 约束方向

- 主键：`id`

## 8. `query_records`

### 表职责

保存轻量查询日志，用于历史追踪和评估。

### 主要字段草案

| 字段名 | 类型建议 | 说明 |
| --- | --- | --- |
| `id` | `uuid` | 主键 |
| `query_type` | `varchar(50)` | 查询类型，如 `search`、`qa`、`summary`、`topic_outline` |
| `query_text` | `text` | 查询文本 |
| `filters_json` | `jsonb` | 过滤条件，可空 |
| `request_context` | `jsonb` | 额外请求上下文，可空 |
| `result_summary` | `text` | 结果摘要，可空 |
| `created_at` | `timestamp` | 创建时间 |

### 约束方向

- 主键：`id`
- 不建议对 `query_text` 加唯一约束

## 9. `generated_artifacts`

### 表职责

保存系统生成的回答、摘要和专题归纳结果。

### 主要字段草案

| 字段名 | 类型建议 | 说明 |
| --- | --- | --- |
| `id` | `uuid` | 主键 |
| `query_record_id` | `uuid` | 关联 `query_records.id` |
| `artifact_type` | `varchar(50)` | 类型，如 `answer`、`summary`、`topic_outline` |
| `title` | `text` | 标题，可空 |
| `content_text` | `text` | 生成文本主内容 |
| `structured_payload` | `jsonb` | 结构化结果 |
| `status` | `varchar(50)` | 状态，如 `active`、`deleted` |
| `created_at` | `timestamp` | 创建时间 |
| `updated_at` | `timestamp` | 更新时间 |
| `deleted_at` | `timestamp` | 逻辑删除时间，可空 |

### 约束方向

- 主键：`id`
- 外键：`query_record_id -> query_records.id`
- `query_record_id` 在第一阶段建议非空
- 即使摘要和专题归纳由系统触发，也应自动创建对应 `query_record`

## 10. `citations`

### 表职责

保存生成结果到证据对象之间的引用关系。

### 主要字段草案

| 字段名 | 类型建议 | 说明 |
| --- | --- | --- |
| `id` | `uuid` | 主键 |
| `generated_artifact_id` | `uuid` | 关联 `generated_artifacts.id` |
| `document_id` | `uuid` | 关联 `documents.id` |
| `chunk_id` | `uuid` | 关联 `chunks.id` |
| `section_id` | `uuid` | 关联 `sections.id`，可空 |
| `citation_label` | `text` | 引用显示标签，可空 |
| `quote_text` | `text` | 被引用片段文本，可空 |
| `is_valid` | `boolean` | 引用是否仍有效 |
| `invalid_reason` | `text` | 失效原因，可空 |
| `created_at` | `timestamp` | 创建时间 |

### 约束方向

- 主键：`id`
- 外键：
  - `generated_artifact_id -> generated_artifacts.id`
  - `document_id -> documents.id`
  - `chunk_id -> chunks.id`
  - `section_id -> sections.id`（可空）
- 第一阶段要求引用目标明确：
  - `document_id` 必填
  - `chunk_id` 建议必填

## 推荐的关系方向总结

### 主关系

- `documents.source_id -> sources.id`
- `sections.document_id -> documents.id`
- `sections.parent_section_id -> sections.id`
- `chunks.document_id -> documents.id`
- `chunks.section_id -> sections.id`
- `document_tags.document_id -> documents.id`
- `document_tags.tag_id -> tags.id`
- `import_jobs.source_id -> sources.id`
- `generated_artifacts.query_record_id -> query_records.id`
- `citations.generated_artifact_id -> generated_artifacts.id`
- `citations.document_id -> documents.id`
- `citations.chunk_id -> chunks.id`
- `citations.section_id -> sections.id`

## 表设计注意事项

### 1. `chunks.content_text` 必须保留

第一阶段必须在 PostgreSQL 中保留 chunk 完整文本，用于：

- 引用展示
- 定位回溯
- FTS 投影
- 生成上下文组装

### 2. 原始文件不进主表

原始 Markdown / PDF 文件本体不直接存入 PostgreSQL 主表，而通过：

- `sources.file_path`
- `documents.file_path`

等字段保留路径引用。

### 3. 生成结果统一用通用表

第一阶段不按回答、摘要、专题归纳拆不同表，而统一落在：

- `generated_artifacts`

通过：

- `artifact_type`
- `structured_payload`

进行区分。

### 4. 引用统一建模

第一阶段所有引用统一落在：

- `citations`

避免不同生成类型各自维护不兼容的引用结构。

## 最终结论

第一阶段 PostgreSQL 主库采用“10 张核心表 + 通用生成结果表 + 通用引用表”的设计方案：

- 关系结构清晰
- 能支撑文档、chunk、标签、任务、查询、生成结果和引用链
- 与 `Qdrant`、`PostgreSQL FTS` 和本地文件系统形成明确分工
