# 4.3 索引与增量更新策略

## 任务目标

定义个人知识库 RAG 项目第一阶段的数据库索引、全文检索、向量索引和增量更新策略，确保文档导入、检索、问答、引用和重建链路具备稳定的数据一致性基础。

本子任务对应路线图中的 `4.3`：

- 设计向量索引字段、embedding 版本字段和重建策略
- 设计全文检索字段，明确标题、正文、标签是否共同参与检索
- 设计文档更新、删除、重导入时的数据一致性方案

## 用户确认结论

基于当前讨论，`4.3` 采用以下正式策略：

- 更新策略采用 `文档级增量检测 + 文档内 chunk 全量重建`
- 文档变更判断以 `content_hash` 为准
- `mtime` 仅作为预筛选依据
- `file_path` 用于来源定位
- 文档逻辑删除后，Milvus 和 BM25 索引中对应内容应同步变为不可检索
- 原始文件默认保留，不随知识库逻辑删除自动删除
- 第一阶段支持 CLI 手动重建索引能力
- 索引设计粒度采用：
  - 字段
  - 原因
  - 建议索引类型

## 第一阶段索引类型总览

第一阶段索引分为 3 类：

1. PostgreSQL 业务索引
2. BM25 全文检索索引
3. Milvus 向量索引

三类索引共同服务混合检索：

- PostgreSQL 业务索引用于过滤、关系查询和状态查询
- BM25 用于关键词和精确匹配
- Milvus 用于语义召回

## PostgreSQL 业务索引策略

### 推荐索引字段

| 表 | 字段 | 原因 | 建议索引类型 |
| --- | --- | --- | --- |
| `documents` | `source_id` | 按来源查询文档、支持导入去重 | B-Tree |
| `documents` | `content_hash` | 判断内容是否变化、支持去重 | B-Tree |
| `documents` | `status` | 过滤 active / deleted / failed 文档 | B-Tree |
| `documents` | `deleted_at` | 过滤逻辑删除文档 | B-Tree |
| `sections` | `document_id` | 快速恢复文档结构 | B-Tree |
| `chunks` | `document_id` | 按文档获取 chunk | B-Tree |
| `chunks` | `section_id` | 按 section 获取 chunk | B-Tree |
| `chunks` | `deleted_at` | 过滤逻辑删除 chunk | B-Tree |
| `tags` | `name` | 标签唯一性与过滤 | Unique B-Tree |
| `document_tags` | `document_id` | 按文档查标签 | B-Tree |
| `document_tags` | `tag_id` | 按标签查文档 | B-Tree |
| `import_jobs` | `status` | 查询待处理、失败任务 | B-Tree |
| `query_records` | `created_at` | 查询历史记录 | B-Tree |
| `generated_artifacts` | `artifact_type` | 按生成类型查询 | B-Tree |
| `generated_artifacts` | `query_record_id` | 查询某次请求的生成结果 | B-Tree |
| `citations` | `generated_artifact_id` | 获取生成结果引用列表 | B-Tree |
| `citations` | `chunk_id` | 反查 chunk 被引用情况 | B-Tree |
| `citations` | `is_valid` | 过滤失效引用 | B-Tree |

### 业务索引设计原则

- 优先服务核心链路
- 优先过滤失效和删除数据
- 不在第一阶段过度设计联合索引
- 联合索引根据真实查询模式在后续阶段补充

## BM25 全文检索策略

### 参与全文检索的数据

第一阶段建议以下内容进入全文检索：

- `documents.title`
- `sections.title`
- `chunks.content_text`
- 可选标签文本

### 字段权重建议

建议在 BM25 检索字段中采用不同权重：

| 内容 | 建议权重 | 原因 |
| --- | --- | --- |
| 文档标题 | 高 | 标题命中通常代表强相关 |
| section 标题 | 中高 | section 标题能帮助定位主题 |
| chunk 正文 | 中 | 正文是主要检索内容 |
| 标签 | 中 | 标签能辅助过滤和主题命中 |

### BM25 设计原则

- 全文检索用于字面匹配和关键词召回
- BM25 不替代向量检索
- BM25 结果应与 Milvus 结果进入混合召回流程
- 逻辑删除的文档和 chunk 不应出现在 BM25 结果中

## Milvus 向量索引策略

### 向量索引对象

第一阶段只对 `Chunk` 建立向量索引。

### Milvus record 建议结构

Milvus 中每条向量记录对应一个 `Chunk`：

| 字段 | 说明 |
| --- | --- |
| `id` | 向量记录主键，建议与 `chunk_id` 保持稳定映射 |
| `vector` | chunk embedding |
| `chunk_id` | chunk id |
| `document_id` | document id |
| `section_id` | section id，可空 |
| `document_type` | 文档类型 |
| `is_active` | 是否可检索 |

### embedding 版本字段

第一阶段建议在 PostgreSQL 的 `chunks` 或后续扩展表中保留：

- `embedding_ref`
- `embedding_model`
- `embedding_version`

当前 `4.2` 草案中已有 `embedding_ref`，后续实现时建议补充：

- `embedding_model`
- `embedding_version`

### 向量索引设计原则

- Milvus 只作为语义检索层，不作为主数据源
- 业务真相以 PostgreSQL 为准
- 删除或更新文档时，Milvus 中旧向量必须同步不可检索

## 增量更新策略

### 采用方案

第一阶段采用：

`文档级增量检测 + 文档内 chunk 全量重建`

### 策略说明

当导入或扫描到一个文档时：

1. 通过 `file_path` 或来源信息定位文档候选
2. 使用 `mtime` 做预筛选
3. 读取文件内容并计算 `content_hash`
4. 如果 `content_hash` 未变化，则跳过重建
5. 如果 `content_hash` 变化，则重建该文档内部结构：
   - `sections`
   - `chunks`
   - BM25 检索投影
   - Milvus 向量索引

### 为什么不做 chunk 级 diff

第一阶段不做 chunk 级增量更新，原因如下：

- chunk 边界变化会引起连锁影响
- 引用定位和 citation 维护会更复杂
- 实现成本高
- 第一阶段更重视一致性和可靠性

## 文档重建流程

当文档确认发生变化时，建议流程如下：

1. 标记旧 chunk 为不可用或逻辑删除
2. 标记旧 BM25 检索投影不可检索
3. 删除或标记 Milvus 中旧向量不可检索
4. 重新解析文档
5. 重建 section
6. 重建 chunk
7. 重新生成 embedding
8. 写入 Milvus
9. 更新 BM25 检索索引
10. 更新 `documents.content_hash`

## 删除同步策略

### 用户删除文档时

第一阶段采用逻辑删除：

- `documents.deleted_at` 标记
- `chunks.deleted_at` 标记
- `citations.is_valid = false`
- 相关 BM25 检索内容不可检索
- Milvus 中相关向量删除或标记 `is_active = false`
- 原始文件默认保留

### 为什么原始文件默认保留

- 原始文件是事实来源
- 用户可能只是从知识库移除，不是删除本地文件
- 避免误删本地资料

### 已有生成结果如何处理

- `GeneratedArtifact` 保留
- 相关 `Citation` 标记失效
- 展示时提示“引用来源已失效”或类似状态

## 手动重建索引能力

### 能力说明

第一阶段建议支持 CLI 手动重建索引能力，用于修复索引不一致、模型变更或导入失败后的恢复。

### 建议命令形态

后续 CLI 可设计为：

- `reindex document <document_id>`
- `reindex all`
- `reindex failed`

### 适用场景

- Milvus 数据丢失或损坏
- BM25 检索投影异常
- embedding 生成失败
- 修改 chunk 切分策略
- 修改 embedding 模型
- 导入任务中断后恢复

### 设计原则

- 手动重建是维护能力，不是日常核心使用能力
- 重建过程应尽量复用导入解析和索引写入逻辑
- 重建结果应更新任务状态或输出明确日志

## 一致性原则

### 1. PostgreSQL 是一致性基准

判断文档、chunk、引用和生成结果状态时，以 PostgreSQL 为准。

### 2. Milvus 和 BM25 是可重建索引

如果 Milvus 或 BM25 索引与 PostgreSQL 不一致，应以 PostgreSQL 为准重建索引。

### 3. 删除优先保护历史

逻辑删除不应直接抹掉历史生成结果。

### 4. 更新优先保护引用正确性

文档更新后旧引用应失效或被标记，而不是继续伪装有效。

## 最终结论

第一阶段索引与增量更新采用“PostgreSQL 作为一致性基准，Milvus 和 BM25 作为可重建索引”的策略：

- 文档级判断是否变化
- 文档变化后重建该文档下所有 section、chunk、BM25 索引和向量索引
- 文档删除后检索索引同步不可用
- 生成结果保留，引用状态失效
- 支持手动 reindex 作为维护能力
