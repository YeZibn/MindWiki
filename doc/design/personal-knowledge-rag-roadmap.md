# 个人知识库 RAG 项目阶段路线图

## 文档目标

本文档用于规划个人知识库管理生成项目的实施步骤，聚焦以下目标：

- 构建可持续演进的个人知识库系统
- 基于 RAG 提供可靠的检索与生成能力
- 支持知识沉淀、专题整理、引用追踪与后续扩展
- 以分步交付的方式推进，优先保证 MVP 可落地

本文档不定义时间周期，只定义实施步骤、阶段目标、主要任务与完成标准。

---

## Step 1. 明确产品目标与范围

### 目标

统一项目目标、边界与第一阶段能力范围，避免在项目初期因范围过大导致设计和实现失控。

### 主要任务

- 明确核心用户场景
  - 导入个人知识材料
  - 搜索已有知识
  - 基于知识库进行问答
  - 生成摘要、专题归纳和相关笔记推荐
- 确定第一阶段支持的数据源
  - 本地 Markdown / Obsidian
  - PDF
  - 网页剪藏
- 明确第一阶段暂不支持的能力
  - 多用户协作
  - 复杂权限系统
  - 大规模知识图谱推理
  - 过多第三方平台接入
- 定义基础成功标准
  - 文档可导入
  - 文档可检索
  - 问答结果带引用
  - 支持基础知识整理能力

### 实施子任务

1.1 梳理 5 到 10 个最核心的用户使用场景，并写成简短用例
1.2 为每个场景定义输入、系统动作、预期输出和成功判断
1.3 列出第一版必须支持的数据源与可延后接入的数据源
1.4 列出第一版必须支持的生成能力与暂不进入 MVP 的高级能力
1.5 形成一份范围边界说明，明确“不做什么”和“为什么不做”
1.6 统一项目关键词定义，例如知识库、文档、chunk、引用、专题、相关笔记

### 产出物

- 产品目标说明
- MVP 范围清单
- 核心用户场景列表
- 非目标清单

### 完成标准

- 团队对第一版支持与不支持的内容达成一致
- 所有后续设计文档都以本阶段范围作为约束

---

## Step 2. 设计整体技术架构

### 目标

将系统拆分为明确的模块与责任边界，形成后续开发的主骨架。

### 主要任务

- 确定系统分层
  - 数据接入层
  - 文档预处理层
  - 文档解析与分块层
  - 知识加工层
  - 存储层
  - 检索编排层
  - 生成服务层
  - 前端交互层
- 明确关键组件职责
  - Importer / Connector
  - Parser
  - Chunker
  - Embedding Processor
  - Retrieval Orchestrator
  - LLM Generator
  - Knowledge Management API
- 明确各层之间的数据流与调用顺序
- 确定异步任务与同步请求的边界

### 实施子任务

2.1 绘制系统上下文图，明确前端、后端、存储、模型和任务系统的关系
2.2 绘制主业务链路图，覆盖导入、解析、索引、检索、生成、展示
2.3 为每个服务定义输入输出接口和职责边界
2.4 标注哪些链路是同步请求，哪些链路必须通过异步任务处理
2.5 明确哪些模块允许后续替换，例如向量库、模型服务、OCR 组件
2.6 补充异常流说明，例如导入失败、解析失败、索引失败时的处理路径

### 产出物

- 系统整体架构图
- 模块职责说明
- 主链路时序图

### 完成标准

- 可以用架构图解释一次完整的“导入 -> 索引 -> 检索 -> 生成 -> 展示”链路
- 各模块边界明确，无明显职责重叠

## Step 3. 设计知识对象模型

### 目标

建立统一的数据抽象，确保文档、段落、分块、标签、引用和关系可以稳定支撑检索与生成。

### 主要任务

- 定义核心实体
  - Document
  - Section
  - Chunk
  - Tag
  - Link
  - Source
  - Job
  - Feedback
- 定义关键字段
  - 标题
  - 来源
  - 内容哈希
  - 摘要
  - 标签
  - 关键词
  - 时间信息
  - 引用位置信息
- 设计文档与 chunk 的映射关系
- 定义引用链路
  - chunk 到 section
  - section 到 document
  - document 到原始文件
- 预留扩展字段
  - rerank score
  - 用户置顶
  - 阅读记录
  - 知识主题归属

### 实施子任务

3.1 输出核心实体清单，并为每个实体写一句职责说明
3.2 为每个实体定义主键、唯一键、外键和必要索引字段
3.3 明确 Document、Section、Chunk 之间的从属关系和删除策略
3.4 定义 chunk 的位置信息结构，确保能回跳到原文位置
3.5 设计标签、关键词、实体、主题之间的边界，避免概念混用
3.6 预留用户反馈、评分、收藏、置顶等行为数据结构

### 产出物

- 数据模型说明
- 实体关系图
- 字段定义草案

### 完成标准

- 模型可以覆盖 Markdown、PDF、网页剪藏三类输入
- 模型可以支撑引用回溯与后续专题整理

## Step 4. 设计存储方案

### 目标

确定关系型数据库、向量检索、全文检索和对象存储的分工。

### 主要任务

- 确定主数据库方案
  - PostgreSQL
- 确定向量存储方案
  - pgvector 作为起步方案
  - 后续可替换或扩展至 Qdrant
- 确定全文检索方案
  - BM25 作为起步方案
- 确定对象存储方案
  - 本地文件系统或 MinIO
- 设计索引策略
  - 文档标题索引
  - 标签索引
  - 时间索引
  - chunk 向量索引
  - 全文检索索引
- 设计增量更新策略
  - 内容哈希比对
  - 文档版本变更检测
  - chunk 重建策略

### 实施子任务

4.1 确定哪些数据进入 PostgreSQL，哪些进入向量索引，哪些进入对象存储
4.2 设计 documents、chunks、tags、links、jobs 等核心表草案
4.3 设计向量索引字段、embedding 版本字段和重建策略
4.4 设计全文检索字段，明确标题、正文、标签是否共同参与检索
4.5 明确原始文件存储路径规则和文件命名策略
4.6 设计文档更新、删除、重导入时的数据一致性方案

### 产出物

- 存储架构设计
- 表结构草案
- 索引策略说明

### 完成标准

- 每类数据都有明确存放位置
- 检索链路不依赖单一向量能力

## Step 5. 构建文档接入与导入流程

### 目标

打通知识进入系统的入口，让系统具备稳定的数据接入能力。

### 主要任务

- 实现文件上传接口
- 实现本地目录扫描能力
- 设计第三方接入适配器接口
- 支持三类初始数据源
  - Markdown
  - PDF
  - Web Clip
- 记录导入任务状态
  - 待处理
  - 处理中
  - 成功
  - 失败
- 设计失败重试与错误日志机制

### 实施子任务

5.1 设计统一导入入口，包括上传、目录扫描和未来连接器模式
5.2 定义导入任务请求结构、状态机和任务追踪字段
5.3 为 Markdown、PDF、Web Clip 分别定义接入适配协议
5.4 确定文件去重策略，例如基于路径、哈希、来源 URI 的联合判断
5.5 设计导入错误分类，例如格式不支持、解析失败、存储失败
5.6 设计失败重试规则和人工介入入口

### 产出物

- 导入接口定义
- 数据源适配规范
- 导入任务流转说明

### 完成标准

- 用户可以把知识文件稳定放入系统
- 系统能记录每次导入的状态与错误原因

## Step 6. 构建文档预处理与清洗能力

### 目标

把原始数据转化为适合解析和检索的标准内容，降低噪音对后续效果的影响。

### 主要任务

- 实现 Markdown 标准化
- 实现 PDF 文本抽取
- 实现网页正文抽取
- 清理无效内容
  - 页眉页脚
  - 重复标题
  - 导航和广告
  - 空段落和乱码
- 设计 OCR 接入点，用于处理扫描件或图片型 PDF
- 设计去重策略
  - 文档级去重
  - chunk 级重复检测

### 实施子任务

6.1 为 Markdown、PDF、Web Clip 分别定义输入标准化流程
6.2 列出常见脏数据类型，并为每类噪音定义清洗规则
6.3 明确正文抽取优先级，区分正文、注释、脚注、附录等内容
6.4 设计 OCR 触发条件，例如文本提取为空或文本密度过低
6.5 定义文档级与 chunk 级去重的阈值和判定标准
6.6 建立预处理结果对照样例，便于后续规则迭代

### 产出物

- 预处理规则说明
- 各类文档清洗策略
- 清洗结果示例

### 完成标准

- 输入内容在进入解析层前已完成基础标准化
- 同一份文档重复导入不会产生明显冗余

---

## Step 7. 构建解析、切分与知识加工流程

### 目标

形成高质量 chunk，补齐摘要、标签、关键词等知识增强信息。

### 主要任务

- 提取文档标题层级
- 将内容拆分为 section
- 设计 chunk 切分策略
  - 按标题切分
  - 按段落边界切分
  - 按语义长度约束切分
- 记录 chunk 与原文位置映射
- 生成 chunk 摘要
- 提取关键词和标签
- 识别实体与潜在关联主题
- 为后续引用展示保留定位信息

### 实施子任务

7.1 定义 section 切分规则，统一标题层级与无标题正文的处理方式
7.2 设计 chunk 大小、重叠长度和特殊内容边界规则
7.3 明确表格、代码块、引用块、列表等结构化内容的切分策略
7.4 设计 chunk 与原文偏移量的记录格式，支持前端高亮定位
7.5 定义摘要、关键词、标签、实体的生成顺序与依赖关系
7.6 为知识加工结果设计置信度或来源标记，便于后续人工校正

### 产出物

- Chunk 设计规范
- 切分策略说明
- 知识加工字段说明

### 完成标准

- chunk 具备语义完整性
- 每个 chunk 可以回溯到原始文档与对应位置
- chunk 数据足以支撑检索与引用展示

## Step 8. 构建索引与召回能力

### 目标

建立以 `chunk` 为统一主对象的混合检索基础设施，让系统既能做 BM25 关键词精确匹配，也能做 embedding 语义召回，并为后续检索编排层提供稳定候选片段。

### 主要任务

- 明确第一阶段统一检索对象
  - `chunk` 作为主检索单元
  - 标题、标签、正文分别通过何种索引信号参与召回
- 设计 embedding 生成方案
  - embedding 模型选择原则
  - embedding 输入构造
  - embedding 版本管理与重建
- 设计 BM25 检索字段
  - 标题字段
  - 标签字段
  - 正文字段
  - 字段权重与命中来源
- 设计统一检索接口
  - query
  - tags
  - source type
  - time range
  - document scope
  - retrieval mode
- 设计基础混合召回策略
  - 向量召回
  - BM25 召回
  - 过滤条件
  - top-k 默认策略
- 设计多路召回结果融合方式
  - 去重规则
  - 双命中优先规则
  - 分层合并
  - RRF 或线性融合等增强路径
- 设计检索验证样例与覆盖标准
  - 标题词查询
  - 标签词查询
  - 正文词查询
  - 语义改写查询
  - 中英混合查询

### 实施子任务

8.1 定义第一阶段检索对象边界，明确 `chunk`、标题、摘要、标签在召回链路中的角色
8.2 定义 embedding 生成方案与版本管理策略，明确模型、升版触发条件、active version 与重建方式
8.3 设计 chunk embedding 输入构造策略，明确标题拼接规则、模板格式与匿名 section 处理方式
8.4 设计 BM25 检索字段与权重，明确标题、标签、正文的纳入范围和命中来源表达
8.5 定义统一检索接口参数与候选结果返回结构
8.6 设计基础混合召回策略，明确向量召回、BM25 召回、过滤条件和 top-k 默认参数
8.7 设计多路召回结果的融合方式，明确去重、双命中优先、分层合并与可选增强策略
8.8 准备典型检索样例与验证标准，用于验证召回覆盖率、命中精度与多语言适配性

### 产出物

- 检索接口设计
- 混合召回策略说明
- 索引构建与版本管理说明
- 检索验证样例集

### 完成标准

- 输入问题后系统能返回以 `chunk` 为单位的基础候选知识片段
- 检索结果同时具备 BM25 精确命中与 embedding 语义召回能力
- 检索接口、索引方案和基础召回策略之间边界清晰

---

## Step 9. 构建检索编排与重排层

### 目标

在 `Step 8` 基础召回结果之上完成 query 拆解、查询扩展、子任务级结果归并、rerank 和上下文构建，提升 RAG 命中质量，避免将低质量上下文直接传入模型。

### 主要任务

- 设计 query 拆解规则
  - 判断原始 query 是否需要拆解
  - 生成可独立检索的 `sub-query`
  - 控制拆解层数与子任务数量
- 实现查询扩展策略
  - 生成 `base_query`
  - 生成 `step-back query`
  - 生成 `HyDE query`
- 设计子任务内检索结果归并策略
  - 合并 `base + bm25`
  - 合并 `base + vector`
  - 合并 `step-back + vector`
  - 合并 `HyDE + vector`
  - 记录命中来源并保留子任务边界
- 引入 rerank 机制
- 设计上下文构建器
  - 按 `sub-query` 分段组织上下文
  - 控制总 token
  - 保留必要引用证据
- 设计引用信息结构
  - 来源标题
  - chunk id
  - section 标题
  - 原文位置
  - 命中来源与跳转信息

### 实施子任务

9.1 定义 query 拆解规则，明确原始 query 的拆解判断与 `sub-query` 生成约束
9.2 设计查询扩展策略，明确 `base_query`、`step-back query` 和 `HyDE query` 的生成方式与使用范围
9.3 设计子任务内检索结果归并策略，明确单个 `sub-query` 内多查询、多通道结果的去重与融合方式
9.4 选择 rerank 方案并定义 rerank 的输入输出结构，明确其在每个 `sub-query` 内独立执行
9.5 设计上下文构建器的截断与拼接策略，明确按 `sub-query` 分段组织上下文并控制 token 预算
9.6 设计引用载荷结构，确保前端可以直接渲染来源与跳转位置

### 产出物

- Retrieval Orchestrator 设计
- Rerank、Result Merger 与 Context Builder 说明
- Query 拆解与查询扩展规则

### 完成标准

- 进入 LLM 的上下文可控、可解释、可回溯
- 检索编排层可以独立演进而不与 `Step 8` 的基础召回层职责重叠

---

## Step 10. 构建生成能力与回答约束

### 目标

让系统输出可靠、可引用、适合个人知识管理的生成结果。

### 主要任务

- 定义生成任务类型
  - 基于知识片段问答
  - 文档摘要
  - 多文档主题归纳
  - 相关笔记推荐
  - 历史观点对比
- 设计提示词模板
- 定义回答约束
  - 仅依据已检索知识回答
  - 必须输出引用来源
  - 无法回答时明确说明
  - 不编造不存在的内容
- 设计输出结构
  - answer
  - sources
  - related notes
  - confidence
- 设计生成失败与降级策略

### 实施子任务

10.1 为问答、摘要、专题归纳、对比分析分别设计 prompt 模板
10.2 统一结构化输出协议，明确字段必填项和可选项
10.3 设计引用插入规则，确保回答段落与来源片段可以对应
10.4 设计无答案场景的输出规范，例如证据不足、召回为空、上下文冲突
10.5 定义生成降级路径，例如仅返回检索结果摘要而不进行深度生成
10.6 预留模型切换配置，区分高质量生成与低成本生成场景

### 产出物

- Prompt 设计说明
- 回答协议定义
- 引用展示结构定义

### 完成标准

- 回答可追溯到知识来源
- 生成结果具备稳定的结构化输出

---

### 目标

将系统核心能力以可用的产品形态呈现出来，形成首个闭环体验。

  - 搜索页
  - 问答页
  - 文档详情页

- 展示相关笔记推荐
- 支持标签与来源过滤
- 支持查看导入任务状态
- 设计基础交互反馈
  - 点赞
  - 点踩
  - 收藏
  - 置顶

### 实施子任务

11.1 绘制核心页面信息架构，明确页面之间的跳转关系
11.2 设计导入、搜索、问答、文档详情四条核心用户路径
11.3 设计搜索结果卡片与回答来源卡片的展示结构
11.4 设计文档详情页中的原文定位、高亮和相邻内容浏览方式
11.5 定义前端状态管理范围，区分会话状态、筛选状态和任务状态
11.6 为点赞、点踩、收藏、置顶设计最小可用交互和数据回写接口

### 产出物

- 页面结构说明
- 前端交互流程图
- 关键组件清单

### 完成标准

- 用户可以完成“导入 -> 搜索 -> 问答 -> 查看来源”的闭环流程
- 前端已经具备 MVP 可演示能力

---

## Step 12. 建立反馈闭环与质量评估

### 目标

让系统可以持续优化，而不是停留在一次性实现。

### 主要任务

- 收集用户反馈
  - 回答是否有帮助
  - 引用是否准确
  - 推荐笔记是否相关
- 设计检索评估指标
  - Recall
  - Precision
  - MRR
- 设计生成评估维度
  - 准确性
  - 可追溯性
  - 完整性
  - 幻觉率
- 建立测试集
  - 常见问题集
  - 多来源问题集
  - 对比类问题集
- 记录失败样例并回流到优化流程

### 实施子任务

12.1 设计反馈埋点，记录查询、召回结果、最终回答和用户操作
12.2 建立首批评测问题集，并覆盖不同来源和不同问题类型
12.3 为检索层定义可量化指标和样本标注方法
12.4 为生成层定义人工评审标准和记录模板
12.5 建立失败案例台账，记录问题、原因、修复方案和验证结果
12.6 设计定期复盘机制，按类型归纳召回问题和生成问题

### 产出物

- 评估指标文档
- 测试数据集设计
- 反馈回流机制说明

### 完成标准

- 团队可以持续判断系统质量是否在提升
- 关键失败案例有记录、有归因、有优化入口

---

## Step 13. 增强个人知识管理特色能力

### 目标

从“通用 RAG 问答”演进到“真正有个人知识库价值的系统”。

### 主要任务

- 支持时间维度的知识回顾
  - 最近新增知识
  - 某阶段学习内容汇总
- 支持主题页自动聚合
- 支持跨文档对比总结
- 支持知识演化分析
  - 我过去如何理解某个主题
  - 某个概念在不同时间的记录变化
- 支持主动生成
  - 本周知识增量摘要
  - 可复习知识点
  - 重复知识合并建议

### 实施子任务

13.1 为时间维度回顾设计数据视图，如近 7 天、近 30 天、指定阶段
13.2 定义主题页的聚合规则，明确自动聚类与人工确认的边界
13.3 设计跨文档对比模板，支持同主题多来源内容对照
13.4 设计知识演化所需的数据结构，例如时间戳、版本、观点快照
13.5 设计主动生成任务列表和触发机制，例如手动触发或定期任务
13.6 为这些特色能力定义独立评估标准，避免只沿用通用问答指标

### 产出物

- 特色能力设计说明
- 主动生成任务清单
- 知识演化展示方案

### 完成标准

- 系统从“能回答”升级为“能整理、能复盘、能演化”
- 产品价值明显区别于通用聊天式 RAG

---

## Step 14. 推进工程化与可维护性建设

### 目标

让项目具备长期开发和迭代基础，降低未来重构成本。

### 主要任务

- 规范项目目录结构
- 规范配置管理
- 建立日志、监控与告警基础能力
- 建立迁移脚本与初始化流程
- 建立单元测试与集成测试
- 设计任务队列监控
- 规范接口文档与错误码
- 预留模型替换与配置能力

### 实施子任务

14.1 规划项目目录结构，区分应用层、领域层、基础设施层和脚本目录
14.2 定义环境变量、配置文件和密钥管理方式
14.3 接入基础日志方案，并区分 API 日志、任务日志和模型调用日志
14.4 设计数据库迁移、初始化种子数据和本地开发脚本
14.5 为导入链路、检索链路、生成链路分别建立基础测试
14.6 设计监控面板最小集合，至少覆盖任务状态、错误率和关键耗时

### 产出物

- 工程规范文档
- 初始化与部署说明
- 测试与监控方案

### 完成标准

- 新开发者可以较快接手项目
- 核心链路有最基本的工程保障

---

## MVP 完成定义

当以下条件全部满足时，可视为 MVP 完成：

- 支持导入 Markdown、PDF、网页剪藏
- 支持预处理、解析、切分与索引
- 支持混合检索
- 支持基于知识库的问答
- 回答带引用来源
- 支持文档浏览、搜索和问答闭环
- 支持基础摘要、标签和相关笔记推荐
