# 15.5 结构化输出校验与修复机制

## 任务目标

为个人知识库 RAG 项目第一阶段定义 LLM 结构化输出的解析、校验与修复机制，明确协议层、schema 层、引用层的校验边界，以及输出不合格时的最小修复路径，确保模型结果在进入上层业务前具备可接受的一致性与可追溯性。

本子任务对应路线图中的 `15.5`：

- 设计结构化输出解析与校验机制，明确 schema 校验、引用字段校验和不合格输出修复策略

## 用户确认结论

基于当前讨论，`15.5` 采用以下正式设计：

- 结构化输出校验采用三层：
  - `protocol_validation`
  - `schema_validation`
  - `citation_validation`
- citation 必须严格来自当前请求的证据集合
- 输出不合格时采用有限修复链路：
  - 本地轻修复
  - 一次结构化修复重试
  - 失败后报错或走上层 fallback
- 不允许无限修复循环

## 一、设计边界

`15.5` 关注的问题是：

- 模型已经返回结果后，平台如何判断它能不能被接收
- 不能直接接收时，平台如何做最小修复

`15.5` 不负责：

- 定义业务 prompt 内容
- 决定是否重新检索
- 决定最终产品级兜底文案

## 二、整体处理阶段

第一阶段建议将一次结构化输出处理拆成 4 步：

1. 原始返回解析
2. 三层校验
3. 有限修复
4. 成功接收或失败退出

对应原则：

- 先尽量本地修，再决定是否回调模型
- 先修结构问题，再判断是否需要 fallback
- 失败路径必须可观测、可终止

## 三、三层校验定义

第一阶段正式采用以下三层校验。

## 1. `protocol_validation`

用于判断 provider 返回是否满足最小协议要求。

建议检查：

- 是否收到有效 response body
- 是否存在目标 `choice`
- `message.content` 是否非空
- `finish_reason` 是否可接受

典型失败场景：

- 返回空内容
- provider 返回成功但无可消费文本
- 返回被异常截断且无法解析

说明：

- 这一层解决“有没有拿到一个可处理的结果”
- 还不判断 JSON 结构是否正确

## 2. `schema_validation`

用于判断模型输出是否满足目标结构。

建议检查：

- 是否能解析为目标 JSON
- 必填字段是否存在
- 字段类型是否正确
- 枚举值是否合法
- 数组结构是否正确
- 不允许出现关键字段缺失或错型

典型失败场景：

- JSON 格式损坏
- `answer` 缺失
- `citations` 不是数组
- `score` 返回成字符串而不是数值

说明：

- 这一层解决“格式对不对”
- 不直接判断 citation 是否可追溯

## 3. `citation_validation`

用于判断输出中的引用是否真实落在当前证据集合内。

第一阶段正式要求：

- citation 只能引用本次请求已提供的证据集合
- 不允许模型生成游离 citation

建议检查：

- `citation_id` 是否存在于当前 `evidence_set`
- `sub_query_id` 是否与当前上下文一致
- `evidence_role` 是否合法
- 被引用 evidence 是否仍在最终可用集合中

典型失败场景：

- 引用了不存在的 chunk
- 引用了未传入的 section
- 引用了其他请求的旧 evidence id
- citation 字段格式正确，但语义上无法回溯

说明：

- 这一层解决“引用能不能落回当前证据”
- 这是 RAG 输出可追溯性的关键校验

## 四、第一阶段建议的输出结构

平台内部建议统一保留以下校验结果结构：

```json
{
  "validation": {
    "protocol_validation": {
      "passed": true,
      "issues": []
    },
    "schema_validation": {
      "passed": true,
      "issues": []
    },
    "citation_validation": {
      "passed": false,
      "issues": [
        {
          "code": "citation_not_in_evidence_set",
          "path": "$.citations[1].citation_id"
        }
      ]
    },
    "final_status": "repairable"
  }
}
```

其中：

- `passed` 表示该层是否通过
- `issues` 记录问题列表
- `final_status` 用于驱动后续修复或失败处理

## 五、修复策略

第一阶段正式采用有限修复链路：

1. 本地轻修复
2. 一次结构化修复重试
3. 失败后报错或 fallback

## 1. 本地轻修复

适用于无需重新请求模型的轻量问题。

建议覆盖：

- 去除代码块包裹符号
- 去除 JSON 前后多余解释文本
- 处理可安全修正的转义问题
- 对可确定的空白字段做标准化

原则：

- 只做确定性修复
- 不做语义猜测
- 不擅自补造 citation

## 2. 一次结构化修复重试

当本地轻修复后仍未通过时，允许发起一次结构化修复调用。

正式约束：

- 最多一次
- 目标是修结构，不是重做业务回答

建议输入：

- 原始任务约束摘要
- 原始模型输出
- 校验失败点列表
- 当前合法 evidence id 集合

建议要求模型：

- 保留原答案语义
- 只修正格式、字段、citation 合法性
- 不新增未给定 evidence 的引用

## 3. 修复失败后的处理

当一次结构化修复后仍失败，第一阶段不继续循环。

正式采用：

- 标记为失败
- 交由 `15.4` 的失败处理机制决定报错或 fallback

不采用：

- 多轮自动 repair loop
- 无限重新生成

## 六、通过 / 可修 / 不可修的判定建议

第一阶段建议将校验结果收敛为三类状态：

- `accepted`
- `repairable`
- `rejected`

## 1. `accepted`

条件：

- 三层校验全部通过

处理：

- 直接进入上层业务结果

## 2. `repairable`

条件：

- 协议层基本可消费
- schema 或 citation 存在可修复问题

处理：

- 先本地轻修复
- 必要时执行一次结构化修复

## 3. `rejected`

条件：

- 返回内容不可解析
- 关键字段缺失严重
- citation 严重越界且无法修复

处理：

- 直接失败
- 进入上层 fallback 或错误返回

## 七、与不同能力接口的关系

第一阶段建议按接口类型决定校验强度。

## 1. `generate_text`

要求最严格：

- 必须做三层完整校验

原因：

- 会直接面向最终用户输出

## 2. `rerank_with_llm`

要求中等：

- 必须做协议层和 schema 层校验
- 若输出含引用字段，再做 citation 校验

## 3. `generate_subqueries` / `generate_step_back_query` / `generate_hyde_query`

要求较轻：

- 重点做协议层和 schema 层
- 通常不要求 citation 校验

## 八、第一阶段不做的事情

当前阶段不建议引入：

- 多轮自动修复链
- 基于语义相似度自动猜 citation
- 输出内容真实性自动判定
- 基于模型投票的结构修复

这些能力可以在后续评估体系稳定后再扩展。

## 九、完成标准

当以下条件满足时，可认为 `15.5` 完成：

- 平台定义了统一的三层校验结构
- citation 合法性具备明确的证据集合约束
- 输出异常具备“轻修复 -> 一次结构化修复 -> 失败退出”的固定路径
- 修复链路可终止，不存在无限循环
