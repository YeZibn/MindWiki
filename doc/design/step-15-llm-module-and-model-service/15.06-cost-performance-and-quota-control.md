# 15.6 成本、性能与配额控制

## 任务目标

为个人知识库 RAG 项目第一阶段定义 LLM 调用的成本、性能与配额控制机制，明确 token 预算、并发限制、简单配额和统计口径，确保模型服务在可控成本下稳定运行，并为后续优化提供统一度量基础。

本子任务对应路线图中的 `15.6`：

- 设计成本与性能控制机制，明确 token 预算、并发限制、配额与调用统计口径

## 用户确认结论

基于当前讨论，`15.6` 采用以下正式设计：

- token 预算按输入 / 输出拆分控制
- 并发限制采用两层：
  - 全局并发上限
  - 接口级并发上限
- 第一阶段采用简单日配额
- 成本统计按：
  - 接口
  - 任务
  - 模型
  三个维度聚合

## 一、设计边界

`15.6` 关注的问题是：

- 一次调用最多能吃多少 token
- 多个调用同时跑时怎么限流
- 每天允许消耗到什么程度
- 后续如何看清成本花在哪里

`15.6` 不负责：

- 选择具体 provider 计费单价
- 定义业务优先级调度系统
- 实现复杂弹性扩缩容

第一阶段先做可落地、易解释的控制规则。

## 二、token 预算控制

第一阶段正式采用输入 / 输出分离预算。

建议字段：

- `max_input_tokens`
- `max_output_tokens`

### 1. 输入预算

输入预算用于控制：

- system prompt
- user 内容
- 文本化检索上下文

正式原则：

- 超预算时优先裁剪上下文
- 不优先挤压系统规则区

这与 `Step 9` 的 context builder 直接衔接：

- `Step 9` 负责按证据优先级构建上下文
- `Step 15` 负责最终卡住模型输入预算

### 2. 输出预算

输出预算用于控制：

- 最终答案长度
- 结构化字段规模
- citation 列表膨胀

正式原则：

- 每个能力接口都应配置独立的默认输出上限
- 不能只用一个全局 `max_tokens` 粗放处理全部场景

## 三、预算控制粒度

第一阶段建议至少按接口类型设置默认预算。

示例策略：

- `generate_subqueries`：
  - 低输入预算
  - 低输出预算
- `generate_step_back_query`：
  - 低输入预算
  - 低输出预算
- `generate_hyde_query`：
  - 中输入预算
  - 中输出预算
- `rerank_with_llm`：
  - 中输入预算
  - 低输出预算
- `generate_text`：
  - 高输入预算
  - 中到高输出预算

第一阶段不要求现在就写死具体数值，但要求：

- 每个接口必须可配置
- 默认值必须独立维护

## 四、并发限制

第一阶段正式采用两层并发控制：

- 全局并发上限
- 接口级并发上限

## 1. 全局并发上限

作用：

- 防止瞬时请求把整个模型服务打满

建议理解：

- 它控制的是“整个 LLM 模块同时允许多少活跃调用”

适用场景：

- 同时有多个检索请求
- 单次请求内部又拆出多个子调用

## 2. 接口级并发上限

作用：

- 防止某一类接口独占全部资源

例如：

- `generate_hyde_query` 不应挤占 `generate_text`
- `rerank_with_llm` 批量运行时不应打爆最终回答链路

建议理解：

- 它控制的是“同一种能力接口最多能并发多少个调用”

## 五、简单配额机制

第一阶段正式采用简单日配额。

建议至少支持：

- `daily_request_limit`
- `daily_token_limit`

### 1. 为什么第一阶段先做日配额

原因很直接：

- 规则简单
- 易于排查
- 对个人知识库 MVP 已足够

第一阶段不优先引入：

- 周配额
- 月配额
- 按用户分层套餐
- 按任务动态价格策略

### 2. 超额后的处理建议

当日配额超限时，建议直接返回明确状态。

例如：

- `quota_exceeded`

第一阶段不建议：

- 静默降级
- 偷偷切更弱模型继续执行

因为这会让成本控制和结果稳定性都变得不可解释。

## 六、成本统计口径

第一阶段正式采用按以下三个维度聚合：

- 接口
- 任务
- 模型

## 1. 按接口统计

例如：

- `generate_subqueries`
- `generate_step_back_query`
- `generate_hyde_query`
- `rerank_with_llm`
- `generate_text`

目的：

- 看清哪一类能力最消耗资源

## 2. 按任务统计

例如：

- `qa`
- `summary`
- `comparison`
- `review`

目的：

- 看清产品层哪类任务成本最高

## 3. 按模型统计

例如：

- 主模型
- 备用模型

目的：

- 看清不同模型的单位成本与稳定性

## 七、建议记录的核心指标

第一阶段建议每次调用至少记录：

- `request_id`
- `task_type`
- `interface_name`
- `model`
- `input_tokens`
- `output_tokens`
- `total_tokens`
- `latency_ms`
- `retry_count`
- `fallback_used`
- `final_status`
- `estimated_cost`

说明：

- `estimated_cost` 允许先按平台配置单价估算
- 不要求第一阶段做到 provider 账单级精确对账

## 八、与 Step 9 / Step 10 的关系

## 1. 与 Step 9 的关系

`Step 9` 决定：

- 要构造多少 sub-query
- 是否做 step-back / HyDE
- 每个子链路喂多少证据

这些设计会直接影响 LLM 成本。

因此：

- `Step 9` 决定成本形状
- `Step 15.6` 负责把成本卡在可控范围内

## 2. 与 Step 10 的关系

`Step 10` 决定：

- 最终生成任务需要什么输出结构
- 输出长度大致有多大

因此：

- `Step 10` 决定输出需求
- `Step 15.6` 负责设置输出预算与运行上限

## 九、第一阶段不做的事情

当前阶段不建议引入：

- 动态竞价式路由
- 基于实时成本的自动模型切换
- 复杂优先级队列调度
- 细粒度租户级多层配额体系
- 自动按成本目标重写 prompt

这些都属于后续优化项，不属于第一阶段最小闭环。

## 十、完成标准

当以下条件满足时，可认为 `15.6` 完成：

- 平台定义了输入 / 输出分离的 token 预算
- 平台定义了全局与接口级两层并发限制
- 平台定义了简单日配额规则
- 平台能够按接口、任务、模型三个维度统计成本与性能
