# 15.3 LLM 能力接口划分

## 任务目标

为个人知识库 RAG 项目第一阶段定义按用途拆分的 LLM 能力接口，明确 query 拆解、step-back、HyDE、rerank 与最终生成等不同调用场景的独立入口，为后续模型绑定、调用控制和成本分层提供稳定的能力边界。

本子任务对应路线图中的 `15.3`：

- 设计模型注册与路由机制，明确不同任务场景下的模型选择、切换与配置策略

结合当前讨论，第一阶段正式收敛为：

- `15.3` 第一阶段不先做复杂模型路由系统
- `15.3` 先定义 LLM 能力接口划分

## 用户确认结论

基于当前讨论，`15.3` 采用以下正式设计：

- 第一阶段先定义 LLM 能力接口划分
- 不先构建复杂模型路由系统
- 第一阶段建议保留 5 个接口：
  - `generate_subqueries`
  - `generate_step_back_query`
  - `generate_hyde_query`
  - `rerank_with_llm`
  - `generate_text`
- 这 5 个接口可分为两类：
  - Query Planning / Retrieval Support
  - Final Generation

## 一、设计边界

### 1. 为什么第一阶段先做接口划分

当前阶段的核心问题不是：

- 做一个完整模型平台

而是：

- 把不同用途的 LLM 调用入口切清楚

如果一开始就直接做复杂路由系统，会带来以下问题：

- 讨论过重
- 业务边界不清
- 模型选择、调用控制、任务语义混在一起

因此，第一阶段建议先做：

- LLM 能力接口划分

### 2. 本任务不深入展开的内容

第一阶段 `15.3` 不深入展开以下内容：

- 动态模型路由规则
- 多模型优先级矩阵
- 复杂 fallback 链
- 跨 provider 智能切换

这些内容可在后续运行策略设计中继续扩展。

## 二、第一阶段建议保留的接口

第一阶段建议至少保留以下 5 个接口。

## 1. `generate_subqueries`

### 职责

- 根据原始 query 生成一组 `sub-query`

### 对应设计阶段

- `9.1`

### 输入

- 原始 query
- 可选拆解约束

### 输出

- `sub_queries[]`
- 拆解理由

## 2. `generate_step_back_query`

### 职责

- 为原 query 或某个 `sub-query` 生成一条更抽象的 `step-back query`

### 对应设计阶段

- `9.2`

### 输入

- `base_query`

### 输出

- `step_back_query`

## 3. `generate_hyde_query`

### 职责

- 为原 query 或某个 `sub-query` 生成一条 HyDE 用的假设性答案文本

### 对应设计阶段

- `9.2`

### 输入

- `base_query`

### 输出

- `hyde_query`

## 4. `rerank_with_llm`

### 职责

- 对 `(sub_query, chunk)` 候选做证据价值判断

### 对应设计阶段

- `9.4`

### 输入

- `sub_query`
- `chunk candidates[]`

### 输出

- `rerank_score`
- `rerank_reason`
- reranked candidates

## 5. `generate_text`

### 职责

- 基于 prompt 与检索上下文生成最终业务结果

### 对应设计阶段

- `10.x`

### 覆盖场景

- 问答
- 摘要
- 对比
- 回顾

### 输入

- `messages`
- `response_format`

### 输出

- 结构化生成结果

## 三、接口分类

第一阶段建议将这 5 个接口分成两类。

## 1. Query Planning / Retrieval Support 类

用于支撑检索过程：

- `generate_subqueries`
- `generate_step_back_query`
- `generate_hyde_query`
- `rerank_with_llm`

### 特点

- 服务于 query planning 与检索编排
- 不直接产生最终业务回答

## 2. Final Generation 类

用于最终业务输出：

- `generate_text`

### 特点

- 面向问答、摘要、对比、回顾等最终生成任务
- 输出会直接进入产品层结果

## 四、设计原则

第一阶段建议采用以下原则：

## 1. 一个接口只承担一种清晰职责

不要将：

- `sub-query`
- `step-back`
- `HyDE`

混成一个泛化的大接口。

## 2. Query Planning 类接口与 Final Generation 类接口分开

原因：

- 输入不同
- 输出不同
- 质量要求不同
- 成本容忍度不同

## 3. 所有接口底层统一走 `15.2` 协议

虽然用途不同，但底层仍统一采用：

- `OpenAI-compatible /chat/completions`

## 五、与后续模型绑定的关系

本任务当前只定义：

- 有哪些能力接口

不在本任务中写死：

- 每个接口背后默认绑定哪个模型
- 每个接口如何 fallback
- 每个接口的成本优先级

这些能力可在后续运行与调用生命周期设计中继续展开。

## 六、最终结论

`15.3` 第一阶段正式采用以下方案：

- 第一阶段先定义 LLM 能力接口划分
- 不先构建复杂模型路由系统
- 第一阶段建议保留 5 个接口：
  - `generate_subqueries`
  - `generate_step_back_query`
  - `generate_hyde_query`
  - `rerank_with_llm`
  - `generate_text`
- 这 5 个接口分为两类：
  - Query Planning / Retrieval Support
  - Final Generation
- 所有接口底层统一走 `15.2` 定义的模型调用协议
