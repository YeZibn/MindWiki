## 一、模块定位

第一阶段建议将 LLM 模块定义为：

- 一个独立的模型服务层

其核心职责不是定义产品任务，而是：

- 将上层标准化请求转换为稳定的模型调用
- 屏蔽不同模型和 provider 的调用差异
- 为上层提供统一的模型服务接口

一句话定义：

- LLM 模块负责“如何稳定地调用模型”
- 不负责“这个业务任务要生成什么内容”

## 二、模块负责的内容

第一阶段 LLM 模块建议负责以下 6 类能力。

## 1. 调用协议适配

负责：

- 接收统一请求对象
- 组织模型输入消息
- 将不同任务的调用请求映射到统一协议

## 2. 模型选择与路由

负责：

- 根据任务类型或调用配置选择模型
- 支持默认模型与备用模型
- 屏蔽不同 provider / model 的实现差异

## 3. 调用执行控制

负责：

- 超时控制
- 重试
- fallback
- 取消
- 幂等策略

## 4. 输出解析与校验

负责：

- 结构化输出解析
- schema 校验
- 引用字段校验
- 不合法输出的修复或拒收

## 5. 成本与性能控制

负责：

- token 预算控制
- 并发限制
- 调用配额
- 成本统计

## 6. 调用日志与观测

负责：

- 记录模型名
- 记录 token 用量
- 记录时延
- 记录错误原因
- 记录输入摘要与结果状态

## 三、模块不负责的内容

为了避免职责漂移，第一阶段正式明确 LLM 模块不负责以下内容：

- 检索与召回
- query 拆解与规划
- step-back / HyDE 生成策略本身
- context builder
- prompt 语义设计本身
- 业务回答正确性判断
- 前端渲染
- 用户权限与会话管理

## 四、与 Step 9 的边界

### Step 9 负责

- query 拆解
- step-back
- HyDE
- rerank
- context builder
- citation payload

### LLM 模块不负责

- 决定检索哪些 chunk
- 决定上下文怎么拼
- 决定哪条证据更相关

### 接口关系

`Step 9` 可以把以下内容传给 LLM 模块：

- 已构建好的上下文包
- 目标任务类型
- 输出 schema

## 五、与 Step 10 的边界

### Step 10 负责

- 问答、摘要、对比、归纳等生成任务类型
- prompt 模板设计
- 回答约束
- 输出协议语义
- 无答案场景与产品级降级策略

### LLM 模块负责

- 将 `Step 10` 定义好的任务转成一次真正的模型调用

简单理解：

- `Step 10` 定义“要模型做什么”
- `Step 15` 定义“怎么把模型调用跑稳”

## 六、与 Step 14 的边界

### Step 14 负责

- 配置管理
- 日志基础设施
- 测试
- 监控面板
- 工程规范

### Step 15 负责

- LLM 相关运行规则

例如：

- 模型路由
- token 预算
- retry / fallback
- schema 校验
- 模型调用观测字段

简单理解：

- `Step 14` 是通用工程底座
- `Step 15` 是模型服务运行层

## 七、建议的最小输入输出接口概念

为了让边界更清晰，第一阶段建议 LLM 模块面向上层提供统一调用接口。

### 建议输入结构

```json
{
  "task_type": "qa",
  "model_profile": "high_quality",
  "messages": [],
  "output_schema": {},
  "timeout_ms": 30000
}
```

### 建议输出结构

```json
{
  "model_used": "example-model",
  "output_text": "...",
  "structured_output": {},
  "usage": {
    "input_tokens": 1200,
    "output_tokens": 280
  },
  "latency_ms": 4200,
  "status": "success"
}
```

说明：

- 该结构只是边界示意
- 真正的统一协议在 `15.2` 中继续细化
