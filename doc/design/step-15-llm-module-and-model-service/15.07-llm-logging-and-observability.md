# 15.7 LLM 调用日志与观测字段

## 任务目标

为个人知识库 RAG 项目第一阶段定义 LLM 调用日志与观测字段，明确记录层级、最小字段集合、聚合统计视角以及调试模式边界，确保模型调用具备基础的排障、评估、审计与调优能力。

本子任务对应路线图中的 `15.7`：

- 设计 LLM 调用日志与观测字段，支持后续排障、评估和模型调优

## 用户确认结论

基于当前讨论，`15.7` 采用以下正式设计：

- 第一阶段先做调用观测最小闭环，不做重型 tracing 平台
- 日志分为两层：
  - `request_log`
  - `attempt_log`
- `context_digest` 除 hash 外，同时记录命中的 `chunk_id` 列表
- 聚合统计至少支持两种视角：
  - 按日统计
  - 按 `接口 x 模型` 统计
- 日志保留时长暂不在第一阶段讨论
- 默认只记录摘要信息
- 可选提供显式调试模式，用于少量请求的增强排障记录

## 一、设计目标

第一阶段的日志与观测主要服务 4 个目标：

1. 排障
2. 评估
3. 审计
4. 调优

对应要回答的问题分别是：

- 这次调用为什么失败
- 哪类接口最慢、最贵、最不稳定
- 这次结果到底使用了哪个模型、哪次重试、哪次 fallback
- 哪组接口、模型和 prompt 版本效果最差

## 二、日志分层

第一阶段正式采用两层日志结构。

## 1. `request_log`

表示：

- 一次上层业务调用的总记录

例如：

- 一次 `generate_text`
- 一次 `rerank_with_llm`
- 一次 `generate_subqueries`

它描述的是：

- 从业务入口视角看，这次调用整体发生了什么

## 2. `attempt_log`

表示：

- 某次调用生命周期内的每一次实际模型请求

例如：

- 首次请求
- 第一次重试
- fallback 后的备用模型请求

它描述的是：

- 从执行链路视角看，每一步实际发给模型的请求发生了什么

## 三、为什么采用两层结构

原因很直接：

- `request_log` 便于从业务层排查问题
- `attempt_log` 便于从执行层分析重试、fallback 和失败原因

这与 `15.4` 中已确定的字段直接对齐：

- `request_id`
- `attempt_id`

因此第一阶段不建议只保留单层日志，否则：

- 无法清晰区分“这次业务调用失败”与“其中某次尝试失败但最终成功”

## 四、第一阶段最小字段集合

第一阶段建议日志字段按以下 5 类组织。

## 1. 身份字段

- `request_id`
- `attempt_id`
- `trace_id`
- `task_type`
- `interface_name`

作用：

- 标识这是谁发起的哪一类调用
- 将上层任务、生命周期和单次请求串起来

## 2. 模型字段

- `provider`
- `model`
- `is_fallback`

作用：

- 标识此次尝试使用的是哪个 provider / model
- 标识是否已进入 fallback 路径

## 3. 输入摘要字段

- `system_prompt_version`
- `user_input_digest`
- `context_digest`
- `context_chunk_ids`
- `context_chunk_count`
- `sub_query_count`

其中：

- `user_input_digest` 用于标识用户输入摘要或 hash
- `context_digest` 用于标识当前上下文的摘要或 hash
- `context_chunk_ids` 明确保留命中的 `chunk_id` 列表

作用：

- 在不默认落完整内容的前提下，仍然保留输入可追踪性
- 支撑后续“这次回答到底基于哪些证据”的排查需求

## 4. 运行字段

- `started_at`
- `finished_at`
- `latency_ms`
- `retry_count`
- `final_status`
- `error_code`

作用：

- 记录调用的起止时间、时延、重试情况和最终结果

其中：

- `final_status` 建议至少支持：
  - `success`
  - `failed`
  - `cancelled`
  - `quota_exceeded`
- `error_code` 用于记录标准化错误类型

## 5. 成本字段

- `input_tokens`
- `output_tokens`
- `total_tokens`
- `estimated_cost`

作用：

- 支撑后续成本、性能和稳定性分析

说明：

- `estimated_cost` 第一阶段允许按平台配置单价估算
- 不要求 provider 账单级精确对账

## 6. 输出结果字段

- `finish_reason`
- `validation_status`
- `repair_applied`
- `citation_count`

作用：

- 记录模型是如何结束的
- 记录结构化校验是否通过
- 记录是否触发过修复
- 记录最终输出里带了多少 citation

## 五、默认记录策略

第一阶段正式采用：

- 默认只记录摘要信息

默认不记录完整内容，包括：

- 完整 `system prompt`
- 完整 `user content`
- 完整检索上下文正文
- 完整模型输出正文

这样做的原因是：

- 成本更低
- 存储更轻
- 默认隐私风险更小
- 对第一阶段排障已经足够

## 六、调试模式

第一阶段允许存在一个可选的显式调试模式。

它的含义是：

- 默认日志策略不变
- 但在少量排障场景下，可以显式打开增强记录

调试模式建议具备以下约束：

- 不是默认开启
- 只用于单次请求或短时排障窗口
- 只能由明确配置触发

调试模式下可额外记录：

- prompt 文本
- context 渲染文本
- 原始输出文本

第一阶段建议同时保留以下原则：

- 调试模式也应支持脱敏
- 不应对全部请求长期常开

## 七、聚合统计视角

第一阶段正式采用两类聚合统计。

## 1. 按日统计

建议至少包含：

- 调用量
- 成功率
- 平均延迟
- token 消耗
- 成本估算
- 失败类型分布

作用：

- 看清每天整体调用健康度和资源消耗

## 2. 按 `接口 x 模型` 统计

建议至少包含：

- 调用量
- 成功率
- 平均延迟
- 平均 token
- 平均成本
- 错误分布

作用：

- 看清哪类接口和哪种模型组合最贵、最慢、最不稳定

## 八、与其他模块的关系

## 1. 与 `15.4` 的关系

`15.4` 定义：

- 生命周期
- 重试
- fallback
- 取消

`15.7` 负责把这些过程记录出来。

也就是说：

- `15.4` 决定调用怎么跑
- `15.7` 决定调用怎么被看见

## 2. 与 `15.5` 的关系

`15.5` 定义：

- 协议校验
- schema 校验
- citation 校验
- 修复链路

`15.7` 负责把这些结果记录为：

- `validation_status`
- `repair_applied`
- `error_code`

## 3. 与 `15.6` 的关系

`15.6` 定义：

- token 预算
- 并发限制
- 配额
- 成本统计口径

`15.7` 负责承接这些运行结果的实际观测数据。

## 九、第一阶段不做的事情

当前阶段不建议引入：

- 重型 tracing 平台
- 自动告警规则设计
- 默认全量保存完整 prompt / output
- 日志保留时长策略
- 复杂可视化运维大盘细节

这些内容可以在后续工程化阶段再展开。

## 十、完成标准

当以下条件满足时，可认为 `15.7` 完成：

- 平台定义了 `request_log` 与 `attempt_log` 两层日志结构
- 平台定义了第一阶段最小字段集合
- 平台明确默认只记录摘要信息
- 平台支持可选调试模式用于增强排障
- 平台明确至少支持按日与按 `接口 x 模型` 的聚合统计
