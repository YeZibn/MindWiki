# 15.2 统一模型调用协议设计

## 任务目标

为个人知识库 RAG 项目第一阶段定义统一的模型调用协议，明确 `LLMRequest` 与 `LLMResponse` 的结构，采用 `OpenAI-compatible /chat/completions` 风格接口，并在不破坏标准消息规则的前提下承载 RAG 场景所需的上下文、结构化输出和运行控制信息。

本子任务对应路线图中的 `15.2`：

- 设计统一模型调用协议，明确输入消息结构、上下文载荷、输出 schema 和调用返回格式

  - `LLMRequest`

## 一、协议总体原则

第一阶段统一模型调用协议建议遵循以下原则：

1. 尽量兼容 `OpenAI-compatible /chat/completions`
2. 模型真正看到的输入仅来自 `messages`
3. `system` 负责行为规则，`user` 负责用户输入与上下文文本
4. 平台运行控制字段与标准消息字段分层
5. 返回结果既保留标准 completion 结构，也保留平台级解析和校验信息

## 二、消息组织规则

### 1. `system`

第一阶段正式采用：

- `system` 放 prompt / 行为规则 / 输出约束

例如：

- 回答范围限制
- 引用要求
- 结构化输出要求
- 无答案场景说明

### 2. `user`

第一阶段正式采用：

- `user` 放用户输入内容与文本化渲染后的结构化上下文

说明：

- RAG 结构化上下文不能以独立模型输入字段直接参与推理
- 结构化上下文必须先被渲染成文本，再进入 `user.content`

例如：

- 用户问题
- sub-query 分段
- 证据 chunk
- 引用辅助信息

## 三、结构化上下文的处理原则

第一阶段正式采用：

- 结构化上下文可以作为平台内部中间对象存在
- 但模型调用协议中不得把它作为独立字段直接交给模型推理

这意味着：

- 平台内部可以先维护：
  - `query`
  - `sub_queries`
  - `evidence_chunks`
  - `citations`
- 但真正调用 `/chat/completions` 前，必须渲染进 `user.content`

## 四、LLMRequest 结构

第一阶段建议 `LLMRequest` 至少包含以下字段：

- `task_type`
- `model`
- `messages`
- `temperature`
- `top_p`
- `max_tokens`
- `response_format`
- `stream`
- `timeout_ms`
- `retry_policy`
- `metadata`

## 1. `task_type`

表示生成任务类型，例如：

- `qa`
- `summary`
- `comparison`
- `review`

## 2. `model`

表示目标模型名。

## 3. `messages`

采用 `OpenAI-compatible /chat/completions` 结构。

第一阶段建议至少支持：

```json
[
  { "role": "system", "content": "..." },
  { "role": "user", "content": "..." }
]
```

## 4. `temperature`

控制采样随机性。

## 5. `top_p`

控制 nucleus sampling。

## 6. `max_tokens`

控制最大输出长度。

## 7. `response_format`

用于约束结构化输出。

第一阶段建议支持：

- `json_schema`

## 8. `stream`

表示是否采用流式输出。

第一阶段可默认：

- `false`

## 9. `timeout_ms`

表示单次调用超时限制。

## 10. `retry_policy`

用于定义调用重试与 fallback 行为。

建议至少支持：

- `max_retries`
- `allow_fallback`

## 11. `metadata`

用于携带平台内部元数据。

例如：

- `request_id`
- `session_id`
- `trace_id`

## 五、LLMRequest 示例

```json
{
  "task_type": "qa",
  "model": "gpt-4.1",
  "messages": [
    {
      "role": "system",
      "content": "你是一个基于检索证据回答问题的助手。只能依据提供的上下文回答，并输出结构化 JSON。"
    },
    {
      "role": "user",
      "content": "用户问题：Step 8 和 Step 9 的区别是什么？\n\n检索上下文：\n[Sub-query 1] Step 8 的职责边界是什么？\n[Evidence 1] ...\n[Sub-query 2] Step 9 的职责边界是什么？\n[Evidence 1] ..."
    }
  ],
  "temperature": 0.2,
  "top_p": 1.0,
  "max_tokens": 1200,
  "response_format": {
    "type": "json_schema",
    "json_schema": {
      "name": "rag_answer",
      "schema": {}
    }
  },
  "stream": false,
  "timeout_ms": 30000,
  "retry_policy": {
    "max_retries": 2,
    "allow_fallback": true
  },
  "metadata": {
    "request_id": "req_001",
    "session_id": "sess_001"
  }
}
```

## 六、LLMResponse 结构

第一阶段建议 `LLMResponse` 同时保留两层信息：

1. 标准 completion 返回
2. 平台扩展字段

建议字段包括：

- `id`
- `object`
- `created`
- `model`
- `choices`
- `usage`
- `parsed_output`
- `validation`
- `timing`
- `status`
- `error`

## 1. 标准字段

尽量保持兼容 `/chat/completions`：

- `id`
- `object`
- `created`
- `model`
- `choices`
- `usage`

## 2. `parsed_output`

表示解析后的结构化结果。

## 3. `validation`

表示结构化输出校验结果。

建议至少包含：

- `schema_valid`
- `citation_valid`

## 4. `timing`

表示调用耗时信息。

建议至少包含：

- `latency_ms`

## 5. `status`

建议支持：

- `success`
- `failed`
- `partial`

## 6. `error`

调用失败或校验失败时返回错误对象。

## 七、LLMResponse 示例

```json
{
  "id": "chatcmpl_xxx",
  "object": "chat.completion",
  "created": 1760000000,
  "model": "gpt-4.1",
  "choices": [
    {
      "index": 0,
      "message": {
        "role": "assistant",
        "content": "{\"answer\":\"...\",\"sources\":[]}"
      },
      "finish_reason": "stop"
    }
  ],
  "usage": {
    "prompt_tokens": 1800,
    "completion_tokens": 320,
    "total_tokens": 2120
  },
  "parsed_output": {
    "answer": "...",
    "sources": []
  },
  "validation": {
    "schema_valid": true,
    "citation_valid": true
  },
  "timing": {
    "latency_ms": 4200
  },
  "status": "success"
}
```
