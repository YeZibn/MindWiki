# 15.4 调用生命周期控制与失败处理

## 任务目标

为个人知识库 RAG 项目第一阶段定义 LLM 调用生命周期控制与失败处理机制，明确超时、重试、fallback、幂等、取消以及错误分类规则，确保模型调用在失败场景下具有可解释、可恢复、可观测的处理路径。

本子任务对应路线图中的 `15.4`：

- 设计调用生命周期控制，明确超时、重试、fallback、幂等与取消策略

## 关联文档

- `15.02-unified-model-invocation-protocol.md`
- `15.03-llm-capability-interface-splitting.md`

## 用户确认结论

基于当前讨论，`15.4` 采用以下正式设计：

- 调用超时采用双层控制：
  - `request_timeout_ms`
  - `overall_deadline_ms`
- 第一阶段最多重试：
  - `2` 次
- 仅对可恢复错误执行重试
- fallback 只在调用失败类场景触发
- 不因主观质量问题触发 fallback
- fallback 必须满足关键能力约束
- 平台内部保留：
  - `request_id`
  - `attempt_id`
- 上层取消后停止后续重试与 fallback
- 错误处理采用三类分类：
  - 环境类错误：可重试
  - 配置类错误：不可重试
  - 输出类错误：优先修复或 fallback，不默认多次重试

## 一、设计边界

### 1. `15.4` 解决的问题

本任务的目标是回答：

- 一次模型调用从发起到结束，中间出问题时该怎么处理

### 2. `15.4` 不解决的问题

本任务不负责：

- 决定调用哪个业务接口
- 定义输出 schema 具体内容
- 定义日志字段集合

这些内容分别在 `15.3`、`15.5`、`15.7` 中继续展开。

## 二、超时控制

第一阶段正式采用双层超时控制：

- `request_timeout_ms`
- `overall_deadline_ms`

## 1. `request_timeout_ms`

表示：

- 单次模型请求的超时上限

## 2. `overall_deadline_ms`

表示：

- 整个调用生命周期的总时限

### 设计原因

一次调用可能经历：

- 首次请求
- 重试
- fallback

因此不能只看单次请求超时，还需要对整次生命周期设置总时限。

## 三、重试策略

第一阶段正式采用：

- 最多重试 `2` 次

### 适用范围

- 仅对可恢复错误执行重试

### 第一阶段不采用

- 无限重试
- 复杂指数退避策略

## 四、fallback 策略

fallback 的职责是：

- 主模型调用失败后，切换到备用模型继续尝试

### 第一阶段正式采用

- fallback 只在调用失败类场景触发

### 第一阶段不采用

- 因回答质量主观不佳而自动 fallback

### 关键约束

- fallback 到的备用模型必须满足关键能力要求
- 例如要求结构化输出时，不能 fallback 到不支持稳定 schema 的模型

## 五、幂等与调用追踪

第一阶段建议至少保证调用层可识别同一请求。

### 平台内部保留

- `request_id`
- `attempt_id`

### 作用

- 追踪一次调用的完整生命周期
- 区分首次调用与重试调用
- 便于日志排查与问题复盘

## 六、取消策略

第一阶段正式采用：

- 上层取消后停止后续重试与 fallback
- 调用状态标记为 `cancelled`

### 第一阶段不强求

- 真正中断 provider 侧已在执行的请求

如果 provider 不支持强制取消，平台侧至少应做到：

- 取消后忽略后续结果
- 停止生命周期继续推进

## 七、错误分类

第一阶段建议将错误分为三类：

1. 环境类错误
2. 配置类错误
3. 输出类错误

## 1. 环境类错误

特点：

- 通常是临时性问题
- 输入本身没有问题

建议包括：

- 请求超时
- provider `5xx`
- 临时网络错误
- 限流类错误
- 上游网关临时不可用

正式策略：

- `retryable = true`
- `fallback_allowed = true`

## 2. 配置类错误

特点：

- 问题在输入或配置
- 不修正请求，重试结果不会改变

建议包括：

- 请求参数非法
- 模型名不存在
- `response_format` 配置非法
- 输出 schema 本身非法
- 权限 / key 配置错误
- 请求内容超过硬限制且未裁剪

正式策略：

- `retryable = false`
- `fallback_allowed = false`

## 3. 输出类错误

特点：

- 模型返回成功，但输出本身不可直接接受

建议包括：

- 输出不符合 schema
- 引用字段缺失
- provider 返回空结果
- `finish_reason` 异常截断

正式策略：

- 优先进入“修复 / fallback”判断
- 不默认多次重试主模型

## 八、错误处理原则

第一阶段正式采用以下简化原则：

- 环境问题：可重试
- 配置问题：不可重试
- 输出问题：优先修复或 fallback，不默认多次重试

## 九、建议的最小错误结构

每次失败建议至少记录：

- `error_type`
- `retryable`
- `fallback_allowed`
- `message`

示例：

```json
{
  "error_type": "timeout",
  "retryable": true,
  "fallback_allowed": true,
  "message": "Model request timed out after 30000 ms."
}
```

- 调用超时采用双层控制：
  - `request_timeout_ms`
  - `overall_deadline_ms`
- 最多重试 `2` 次
- 仅对可恢复错误执行重试
- fallback 只在调用失败类场景触发
- 不因主观质量问题触发 fallback
- fallback 必须满足关键能力约束
- 平台内部保留：
  - `request_id`
  - `attempt_id`
- 上层取消后停止后续重试与 fallback
- 错误处理采用三类分类：
  - 环境类错误：可重试
