# 1.1 核心用户场景梳理

## 任务目标

梳理个人知识库 RAG 项目第一阶段最核心的用户使用场景，并为后续的产品范围、系统架构、数据模型和 MVP 设计提供输入。

本子任务对应路线图中的 `1.1`：

- 梳理 5 到 10 个最核心的用户使用场景，并写成简短用例

## 输出要求

本任务最终需要产出：

- 5 到 10 个核心用户场景
- 每个场景的简短用例描述
- 场景优先级建议
- 是否进入 MVP 的建议标记

## 用户输入结论

基于当前讨论，`1.1` 的核心前提如下：

- 目标内容类型以 `工作资料` 和 `学习资料` 为主
- 当前最核心的问题是：
  - 资料太散乱，缺乏统一整理
  - 希望基于已有知识进行检索增强回答
- 第一阶段优先支持的数据源为：
  - `Markdown`
  - `PDF`
- 第一阶段能力重心为：
  - `检索回答`
- 当前产品定位为：
  - `个人工具`
- 当前优先保留的 MVP 核心场景为：
  - 导入
  - 检索
  - 回答
  - 摘要
  - 专题归纳

## 正式核心场景列表

以下为当前版本确认后的核心用户场景，共 7 个。其中前 5 个进入 MVP 优先范围，后 2 个作为近阶段扩展候选。

### 场景 1. 导入知识资料并完成统一入库

- 编号：`1.1-S01`
- 优先级：`P0`
- MVP：`是`
- 简短用例：
  用户将工作资料或学习资料以 Markdown 或 PDF 形式导入系统，系统完成解析、清洗、切分和索引，为后续检索和回答提供基础。
- 场景价值：
  解决知识分散在不同文件中的问题，是后续所有能力的入口。

### 场景 2. 检索历史资料中的相关内容

- 编号：`1.1-S02`
- 优先级：`P0`
- MVP：`是`
- 简短用例：
  用户输入关键词或自然语言问题，系统从已导入的 Markdown 和 PDF 中找到相关知识片段，并按相关性返回结果。
- 场景价值：
  解决“知道记过，但找不到”的高频问题，是个人知识库最基础的使用方式。

### 场景 3. 基于知识库进行问答并返回引用来源

- 编号：`1.1-S03`
- 优先级：`P0`
- MVP：`是`
- 简短用例：
  用户提出一个问题，例如类似法律助手或医学助手中的知识性问题，系统基于个人知识库中的已有资料生成回答，并附带可追溯的引用来源。
- 场景价值：
  这是该项目的核心差异化能力，重点不只是“答得出来”，而是“依据已有知识回答”。

### 场景 4. 对单篇或多篇资料生成摘要

- 编号：`1.1-S04`
- 优先级：`P1`
- MVP：`是`
- 简短用例：
  用户选择一篇或多篇资料，系统自动提取重点内容，生成结构化摘要，帮助快速理解长文档或多个资料集合。
- 场景价值：
  降低阅读成本，提升工作资料和学习资料的吸收效率。

### 场景 5. 将分散资料自动归纳为专题

- 编号：`1.1-S05`
- 优先级：`P1`
- MVP：`是`
- 简短用例：
  用户围绕某个主题查看系统自动聚合的相关资料、摘要和核心观点，形成一个专题页面或专题总结。
- 场景价值：
  解决资料散乱但难以形成体系的问题，是个人知识管理能力的重要体现。

### 场景 6. 查看与当前问题相关的历史资料

- 编号：`1.1-S06`
- 优先级：`P2`
- MVP：`否`
- 简短用例：
  在问答或检索结果之外，系统额外推荐相关文档、相关 chunk 或相关主题，帮助用户继续横向扩展阅读。
- 场景价值：
  强化知识之间的连接，但不是第一阶段最核心能力。

### 场景 7. 回顾某一主题下的资料积累情况

- 编号：`1.1-S07`
- 优先级：`P2`
- MVP：`否`
- 简短用例：
  用户查看某个主题在一段时间内积累了哪些资料、重点观点和摘要，用于阶段性回顾。
- 场景价值：
  有助于形成知识沉淀，但相对检索和问答优先级较低。

## MVP 场景结论

当前建议纳入 MVP 的 5 个核心场景如下：

1. `1.1-S01` 导入知识资料并完成统一入库
2. `1.1-S02` 检索历史资料中的相关内容
3. `1.1-S03` 基于知识库进行问答并返回引用来源
4. `1.1-S04` 对单篇或多篇资料生成摘要
5. `1.1-S05` 将分散资料自动归纳为专题

## 场景排序结论

按当前产品定位和使用目标，建议优先级排序如下：

1. `1.1-S01` 导入
2. `1.1-S02` 检索
3. `1.1-S03` 回答
4. `1.1-S04` 摘要
5. `1.1-S05` 专题归纳
6. `1.1-S06` 相关资料推荐
7. `1.1-S07` 阶段回顾

## 当前假设

以下假设将在后续子任务中继续沿用，除非后续讨论修改：

- 第一阶段暂不重点支持外部协作场景
- 第一阶段暂不重点支持专业行业知识库的权威校验机制
- 虽然你举了法律助手、医学助手的例子，但当前更偏向“回答方式参考这类工具”，而非一开始就做严格专业合规产品
- 第一阶段的输入源以本地文档为主，而不是第三方 SaaS 平台

## 对后续任务的影响

`1.1` 的结论将直接影响以下子任务：

- `1.2` 场景输入输出与成功标准定义
- `1.3` 第一阶段数据源范围定义
- `1.4` 第一阶段生成能力范围定义
- `2.x` 架构设计中的核心链路取舍
- `11.x` 前端 MVP 页面优先级设计

建议下一步推进 `1.2`，把上述 5 个 MVP 场景进一步拆成：
