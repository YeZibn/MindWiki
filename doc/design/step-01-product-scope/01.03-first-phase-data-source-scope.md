# 1.3 第一阶段数据源范围定义

## 任务目标

明确个人知识库 RAG 项目第一阶段支持的数据源类型、支持边界、接入方式与元数据范围，为后续导入链路、解析流程、数据模型和存储设计提供清晰约束。

本子任务对应路线图中的 `1.3`：

- 列出第一版必须支持的数据源与可延后接入的数据源

## 关联文档

- `01.01-core-user-scenarios.md`
- `01.02-scenario-input-output-and-success-criteria.md`

## 结论概览

第一阶段聚焦两个核心数据源：

1. `Markdown`
2. `PDF`

选择理由如下：

- 与当前用户的实际资料形态高度匹配
- 可以直接支撑导入、检索、问答、摘要、专题归纳这 5 个 MVP 场景
- 能以较低复杂度建立稳定的导入和检索闭环
- 有利于优先保证回答质量，而不是过早扩展接入面

## 第一阶段正式支持的数据源

### 1. `Markdown`

#### 支持目标

支持本地标准 Markdown 文档作为第一阶段核心知识输入源之一。

#### 正式支持范围

- 支持本地 `.md` 文件
- 支持标准 Markdown 结构
  - 标题
  - 段落
  - 列表
  - 引用块
  - 代码块
  - 表格
- 支持 `YAML frontmatter`
- 支持 Obsidian 风格双链 `[[...]]`

#### 第一阶段处理原则

- 标准 Markdown 内容作为主要解析对象
- `YAML frontmatter` 中的字段可作为元数据输入来源
- Obsidian 双链在第一阶段先作为“文本关系线索”保留，不在第一阶段构建完整双链图谱能力

#### 第一阶段暂不深入支持的内容

- 不在第一阶段对图片内容做视觉理解
- 不在第一阶段对附件内容做自动解析
- 图片、附件和外部链接优先保留引用关系，不作为主要知识抽取对象

### 2. `PDF`

#### 支持目标

支持文本型 PDF 作为第一阶段核心知识输入源之一。

#### 正式支持范围

- 支持可直接抽取文本的 PDF
- 支持普通排版文档类 PDF
- 支持后续检索、问答、摘要和专题归纳使用 PDF 中抽取出的文本内容

#### 第一阶段处理原则

- 第一阶段以 `文本型 PDF` 为主支持对象
- 重点保证文本抽取、分段、切分与定位能力稳定可用
- PDF 中的原始页码、段落位置应尽可能保留，以便后续引用定位

#### 第一阶段有限支持的内容

- 扫描版 PDF
- 图片型 PDF
- 需要 OCR 才能读取的 PDF

对于以上内容，第一阶段策略为：

- 可以识别为“有限支持”类型
- 可以保留文件导入能力
- 但不承诺高质量文本抽取与问答效果

#### 第一阶段暂不重点支持的内容

- 不将 OCR 作为第一阶段主路径能力
- 不保证复杂版式、扫描件、图文混排 PDF 的高质量解析结果

## 第一阶段接入方式

第一阶段支持以下两种接入方式：

1. 手动上传文件
2. 指定本地目录批量导入

### 优先级建议

- `P0`：手动上传文件
- `P1`：指定本地目录批量导入

### 设计原因

- 手动上传适合快速验证 MVP 闭环
- 本地目录导入适合个人知识库场景中的长期资料沉淀
- 两种方式可以覆盖“临时导入”和“持续积累”两类使用方式

## 第一阶段统一元数据范围

第一阶段建议统一支持以下元数据字段：

- 标题
- 标签
- 来源备注
- 文件路径
- 文档类型
- 导入时间

### 元数据处理原则

- `标题`
  - 默认自动提取
  - 必要时允许用户后续编辑
- `标签`
  - 允许用户手动补充
  - 后续可扩展自动标签能力
- `来源备注`
  - 允许用户手动补充
  - 用于记录资料背景、用途、来源说明等信息
- `文件路径`
  - 主要用于本地资料追踪和问题排查
- `文档类型`
  - 用于区分 Markdown、PDF 等来源类型
- `导入时间`
  - 用于后续回顾与任务追踪

### 第一阶段不强依赖的元数据

- 原始创建时间
- 原始修改时间
- 作者信息
- 外部系统来源 ID

原因如下：

- 本地文件时间不一定稳定可靠
- 当前阶段以个人工具为主，不需要过早引入复杂来源映射

## 第一阶段明确不纳入的数据源

以下数据源不纳入第一阶段正式支持范围：

- `Notion`
- `飞书`
- `网页剪藏系统直连`
- `代码仓库`
- `音视频转写`
- `图片知识输入`

### 不纳入原因

- 接入复杂度高
- 清洗与解析差异大
- 容易分散 MVP 精力
- 当前阶段的核心目标是建立 Markdown 与 PDF 的稳定闭环

## 第一阶段与后续阶段的边界

### 第一阶段必须做到

- Markdown 可稳定导入和解析
- 文本型 PDF 可稳定导入和解析
- 导入结果可参与检索和问答
- 文档可保留基础元数据
- 检索与问答可回溯到文档内容位置

### 后续阶段可扩展

- OCR 能力
- 扫描版 PDF 深度支持
- Notion / 飞书接入
- 网页剪藏自动同步
- 图片与附件内容理解
- 更强的文档关系建模

## 对后续设计的影响

`1.3` 的结论将直接影响以下设计任务：

- `4.x` 存储结构与元数据表设计
- `5.x` 导入任务流与接入接口设计
- `6.x` Markdown / PDF 清洗和预处理策略
- `7.x` 切分策略与定位信息设计
- `8.x` 索引策略和检索过滤条件设计

## 最终结论

第一阶段以 `Markdown + 文本型 PDF` 为正式支持范围，采用“先做深、再做广”的策略：

- 先把核心数据源做稳定
- 先把定位、引用和可追溯性做好
- 先保障检索和问答质量
- 后续再扩展 OCR、外部系统接入和更复杂的多模态能力
