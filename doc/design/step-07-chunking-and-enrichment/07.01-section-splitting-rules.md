# 7.1 Section 切分规则设计

## 任务目标

定义第一阶段文档解析后的 `section` 切分规则，明确 section 的角色、边界、层级关系和特殊内容处理方式，为后续 chunk 切分、检索归属和引用展示建立稳定的结构骨架。

本子任务对应路线图中的 `7.1`：

- 定义 section 切分规则，统一标题层级与无标题正文的处理方式

## 一、Section 的定义

第一阶段中，`section` 定义为：

- 文档中的结构级内容单元
- 位于 document 与 chunk 之间的中间层
- 用于表达“哪一节在讨论什么”

### Section 的主要作用

`section` 主要服务于：

- 组织文档结构
- 为 chunk 提供归属上下文
- 为引用展示提供“来自哪一节”的定位信息
- 帮助后续检索理解标题和正文关系

### Section 不等于 Chunk

两者职责不同：

- `section` 偏结构组织
- `chunk` 偏检索与模型输入大小控制

同一个 `section` 可以拆成多个 `chunk`。

## 二、Section 切分总体原则

### 1. 标题驱动优先

如果文档中存在稳定标题结构，应优先按标题切分 section。

适用场景：

- Markdown 标题层级
- PDF 识别出的章节标题
- 结构化资料中的显式节标题

### 2. 无标题正文也允许形成 Section

不是所有文档都具备规范标题结构。

因此第一阶段正式允许：

- 连续正文在无标题情况下形成匿名 section

这样可以避免：

- 大段正文没有归属
- 整篇文档被迫塞进一个巨大 section
- 后续 chunk 无法获得结构上下文

### 3. 结构优先于页面边界

section 的边界主要由内容结构决定，而不是由：

- PDF 翻页
- 单纯字符长度
- 单纯空行数量

例如：

- 同一章节跨页延续时，不应因换页自动切成新 section

### 4. 角色感知

section 也应感知内容角色。

例如：

- 正文 section
- supporting_structure section
- supporting_content section

不同角色会影响后续 chunk 和检索权重。

## 三、标题层级切分规则

### 1. Markdown 标题映射

Markdown 标题层级建议直接映射到 section 层级：

- `#` -> level 1 section
- `##` -> level 2 section
- `###` -> level 3 section
- `####` 及以下 -> 可继续保留层级，或在实现中设最大层级限制

### 2. PDF 标题映射

PDF 没有天然 Markdown 标题语法，因此依赖预处理阶段的 `heading` 结构提示。

若能识别：

- 字号显著更大
- 位置独立
- 与上下文存在明显结构边界

则可作为 section 标题候选。

### 3. 层级关系必须保留

第一阶段正式要求：

- section 层记录父子层级

例如：

- 一级标题 section
- 二级标题 section 归属于该一级标题
- 三级标题 section 归属于对应二级标题

这样后续引用可展示为：

```text
文档标题 / 一级节 / 二级节
```

## 四、无标题正文的处理

### 1. 允许匿名 Section

第一阶段允许无标题正文形成匿名 section。

适用场景：

- 无标题笔记
- 扫描整理后的纯文本资料
- PDF 中无法稳定识别标题的正文块

### 2. 匿名 Section 的生成规则

匿名 section 可具备：

- `title = null`
- `display_title = "未命名段落 1"` 之类的自动显示名
- `section_type = anonymous_body`

说明：

- 自动显示名主要用于前端和调试展示
- 不应伪装成原文真实标题

### 3. 匿名 Section 的边界建议

在无标题场景下，可按以下信号形成匿名 section：

- 长段连续正文开始
- 明显结构中断
- 角色切换
- 大块表格或代码块开始

但第一阶段不应把匿名 section 规则做得过于复杂，避免误切。

## 五、特殊内容的 Section 规则

## 1. 目录

目录可形成单独 section，但其类型为：

- `supporting_structure`

正式策略：

- 目录不作为正文主干 section
- 目录可单独保留，便于结构理解和低权检索

## 2. 附录

附录允许独立成 section。

其 section 角色建议为：

- `supporting_content`

说明：

- 附录常有独立标题
- 附录可能包含背景资料、样例、补充说明
- 虽不是正文主干，但具有独立结构价值

## 3. 脚注

脚注通常不单独形成独立正文 section。

正式策略：

- 优先挂靠到其所属正文 section
- 若无法稳定挂靠，可作为附属内容块处理

说明：

- 脚注通常是局部补充，不适合作为整节结构主单元

## 4. 页眉页脚

页眉页脚不应形成正常 section。

若保留：

- 仅可作为 `noise_candidate` 相关结构标记
- 不进入正常正文 section 树

## 六、Section 边界的推荐信号

第一阶段建议综合以下信号判断 section 边界：

### 强信号

- 显式标题出现
- 标题层级变化
- 附录标题出现
- 目录标题出现

### 弱信号

- 长段正文后出现明显结构中断
- 内容角色变化
- 表格或代码块切入
- PDF 中显著段落切换

正式策略：

- 强信号优先决定 section 边界
- 弱信号主要用于无标题正文或标题识别不足场景

## 七、Section 建议数据结构

第一阶段建议 section 至少包含以下字段：

```json
{
  "section_id": "uuid",
  "document_id": "uuid",
  "parent_section_id": "uuid-or-null",
  "level": 1,
  "title": "检索层",
  "display_title": "检索层",
  "section_type": "body | anonymous_body | supporting_structure | supporting_content",
  "content_role": "body | supporting_structure | supporting_content",
  "start_offset": 120,
  "end_offset": 480,
  "page_start": 2,
  "page_end": 3,
  "line_start": 10,
  "line_end": 36
}
```

字段说明：

- `parent_section_id`：层级关系
- `level`：标题层级或匿名层级
- `title`：原始标题，匿名 section 可为空
- `display_title`：展示名称，匿名 section 可自动生成
- `section_type`：section 类型
- `content_role`：该节主要内容角色
- `start_offset` / `end_offset`：标准化文本偏移
- `page_start` / `page_end`：PDF 场景定位
- `line_start` / `line_end`：Markdown 场景定位

## 八、与 Chunk 切分的关系

`section` 是 `chunk` 的上层边界。

第一阶段建议：

- chunk 优先在 section 内切分
- 非必要不跨 section 合并 chunk

这样做的好处：

- 保持 chunk 的语义归属明确
- 引用时能展示所属 section
- 降低跨节拼接导致语义混乱的风险

## 九、与引用展示的关系

section 层将直接影响引用展示体验。

推荐在引用中展示：

- 文档标题
- section 显示名
- 页码或行号

例如：

```text
《RAG 系统设计笔记》 / 检索层 / 第 12 行
```

如果是匿名 section：

```text
《工作记录》 / 未命名段落 2 / 第 3 页
```

## 十、最终结论

`7.1` 正式采用以下 section 切分策略：

- section 是 document 与 chunk 之间的结构单元
- 标题驱动优先，但允许无标题正文形成匿名 section
- section 层记录父子层级，而不是推迟到 chunk 层
- 目录可独立成 `supporting_structure` section
- 附录可独立成 `supporting_content` section
- 脚注通常挂靠到所属正文 section，不单独成为正文主节
- PDF 跨页正文不因翻页自动切 section

该设计将为后续 `7.2` chunk 边界规则、`7.3` 结构化内容切分策略和引用定位能力提供主骨架。
