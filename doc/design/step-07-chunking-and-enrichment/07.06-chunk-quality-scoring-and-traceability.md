# 7.6 Chunk 质量评分、来源标记与可追溯性设计

## 任务目标

为个人知识库 RAG 项目第一阶段中的 `chunk` 建立统一的质量评分、来源标记、依赖追踪与人工复核状态模型，使 `chunk` 具备可解释、可筛选、可追溯、可校正的治理基础。

本子任务对应路线图中的 `7.6`：

- 为知识加工结果设计置信度或来源标记，便于后续人工校正

结合前序设计与本轮对齐，第一阶段正式收敛为：

- `7.6` 的核心对象为 `chunk`
- 第一阶段不对 `keywords`、`entities`、`chunk_summary`、`tag_candidates`、`document_tags` 单独设计评分体系
- 第一阶段优先建立 `chunk-first` 的质量治理模型

基于当前讨论，`7.6` 采用以下正式设计：

- 第一阶段仅对 `chunk` 进行质量评分
- 评分体系采用 `chunk-first`
- `chunk` 质量由 `LLM rubric-based scoring` 完成维度判断
- `LLM` 不直接输出精细小数，而是先对各维度进行分档
- 系统再将档位映射为数值，并计算最终 `chunk_quality_score`

  - `rejected`
  - `edited`

## 一、设计边界

### 1. 为什么 `chunk` 是第一优先级

`chunk` 是后续检索、引用、上下文构建和生成的基础单元。

如果 `chunk` 质量不稳定，常见后果包括：

- 语义被错误截断
- 多个主题被错误拼接
- 检索命中但上下文不可用
- 引用证据不精确
- 下游摘要、标签等结果建立在错误边界之上

因此，第一阶段应优先治理 `chunk` 质量，而不是优先扩展下游 enrichment 结果的评分体系。

### 2. 本任务不覆盖的范围

第一阶段 `7.6` 不深入展开以下内容：

- `keywords` 的独立评分
- `entities` 的独立评分
- `chunk_summary` 的独立评分
- `tag_candidates` 的独立评分
- `document_tags` 的独立评分
- 文档级复杂版本替代策略
- 全量人工校正工作流设计

这些内容可在后续知识管理或质量评估阶段继续扩展。

## 二、Chunk 评分模型总览

第一阶段建议将 `chunk` 质量评分拆为 5 个维度：

1. `boundary_integrity`
2. `semantic_completeness`
3. `retrieval_fitness`
4. `citation_fitness`
5. `size_fitness`

评分总体原则如下：

- `boundary_integrity` 与 `semantic_completeness` 权重最高
- `size_fitness` 不能单独主导总分
- 评分结果既要支持筛选，也要支持后续人工复核

## 三、五个评分维度定义

## 1. `boundary_integrity`

### 定义

评估当前 `chunk` 的切分边界是否合理。

### 关注点

- 是否在自然结构边界上开始和结束
- 是否把应属于同一语义单元的内容错误切开
- 是否把应分开的两个语义单元错误拼接
- 是否错误跨越标题、列表、表格、代码块、引用块等结构边界

### 低分典型情况

- 标题与其解释正文被硬切开
- 一个块前半段与后半段明显属于不同节
- 表格说明与表格本体分离后关联丢失

## 2. `semantic_completeness`

### 定义

评估当前 `chunk` 单独阅读时，语义是否相对完整。

### 关注点

- 是否表达了一个相对完整的意思
- 是否缺少关键前提、限定条件或结论
- 是否只保留过渡句、承接句或残句
- 是否强依赖前后文才能勉强理解

### 低分典型情况

- 仅包含“如下所示”“因此可以看出”等承接性表达
- 只有结论，没有上下文条件
- 长列表被截为不完整片段，单块无法独立理解

## 3. `retrieval_fitness`

### 定义

评估当前 `chunk` 是否适合作为检索召回单元。

### 关注点

- 主题是否集中
- 噪音是否过多
- 长度是否适合检索精度
- 内容是否足以支撑召回后判断相关性

### 低分典型情况

- 单块混合多个主题
- 单块过短，只包含空泛提示语
- 单块过长，相关信息被大段无关内容稀释

## 4. `citation_fitness`

### 定义

评估当前 `chunk` 是否适合作为引用和证据单元。

### 关注点

- 被引用时是否能支撑具体结论
- 定位是否清晰
- 边界是否足够稳定
- 是否因截断或误拼接导致引用失真

### 低分典型情况

- 只有引导语，没有实际证据
- 引用块被截断，原意破坏
- 一个块包含太多无关内容，导致引用不精确

## 5. `size_fitness`

### 定义

评估当前 `chunk` 的大小是否适合作为第一阶段 RAG 系统中的实用单元。

### 关注点

- 是否明显过长
- 是否明显过短
- 是否符合检索、引用和摘要的实用粒度
- 是否属于结构例外块，例如长表格、长代码块

### 低分典型情况

- 只有一句很短的话，但没有独立价值
- 把整节正文塞入一个大块
- 对普通正文过度放大，损害检索和引用精度

## 四、评分权重设计

第一阶段正式采用检索优先的权重方案：

| 维度 | 权重 |
| --- | --- |
| `boundary_integrity` | `0.30` |
| `semantic_completeness` | `0.30` |
| `retrieval_fitness` | `0.20` |
| `citation_fitness` | `0.15` |
| `size_fitness` | `0.05` |

### 设计说明

- `boundary_integrity` 和 `semantic_completeness` 决定 `chunk` 是否为合法基础单元
- `retrieval_fitness` 对 RAG 的召回链路影响直接，因此次高
- `citation_fitness` 重要，但在第一阶段略低于检索适配性
- `size_fitness` 是约束项，而不是主导项

## 五、评分流程设计

第一阶段 `chunk` 评分采用如下流程：

```text
Chunk
  -> 按 rubric 对 5 个维度分别判断
  -> LLM 输出每个维度的分档与理由
  -> 系统将分档映射为数值
  -> 系统按权重计算 chunk_quality_score
  -> 落库评分结果、来源信息和复核状态
```

### 设计原则

- `LLM` 负责做基于 rubric 的分档判断
- 系统负责做数值映射和总分计算
- 不让 `LLM` 直接输出精细小数总分，避免漂移过大

## 六、分档与数值映射

第一阶段每个维度统一采用 5 档：

- `very_high`
- `high`
- `medium`
- `low`
- `very_low`

系统采用以下固定映射：

| 分档 | 数值 |
| --- | --- |
| `very_high` | `1.00` |
| `high` | `0.80` |
| `medium` | `0.60` |
| `low` | `0.35` |
| `very_low` | `0.10` |

### 总分计算方式

建议采用加权求和：

```text
chunk_quality_score =
  boundary_integrity_score * 0.30 +
  semantic_completeness_score * 0.30 +
  retrieval_fitness_score * 0.20 +
  citation_fitness_score * 0.15 +
  size_fitness_score * 0.05
```

第一阶段不额外定义 `chunk_quality_level`，直接保留 `chunk_quality_score`。

## 七、来源标记设计

### 1. `generation_source`

用于表达当前 `chunk` 的生成方式类别。

第一阶段正式采用以下枚举值：

- `rule_only`
- `hybrid`
- `human_edited`

### 含义说明

#### `rule_only`

- `chunk` 由规则递归切分直接形成
- 未经过有效的 LLM 边界调整

#### `hybrid`

- `chunk` 由规则切分与后续语义校正共同形成
- 典型场景包括 merge、re_split 或结构化边界增强

#### `human_edited`

- `chunk` 边界或内容结构经过人工调整

## 八、依赖追踪设计

### 1. `dependency_trace` 的作用

`dependency_trace` 用于记录当前 `chunk` 是如何从上游结构和候选块演化得到的。

它的目标不是记录全部中间细节，而是保证以下能力：

- 能回看当前 `chunk` 的生成依据
- 能在低分或异常时反查问题来源
- 能为后续人工复核提供最小必要上下文

### 2. 第一阶段最小必需字段

第一阶段建议 `dependency_trace` 至少包含以下字段：

- `section_id`
- `candidate_chunk_ids`
- `boundary_signals`
- `generation_path`

### 3. 字段说明

#### `section_id`

表示当前 `chunk` 所属的 `section`。

#### `candidate_chunk_ids`

表示当前最终 `chunk` 由哪些候选块演化而来。

说明：

- 如果最终块直接继承某个候选块，可仅保留一个 ID
- 如果最终块由多个候选块合并而来，应记录多个 ID

#### `boundary_signals`

表示当前 `chunk` 生成时主要参考的边界信号。

建议允许的典型值包括：

- `section_boundary`
- `heading_boundary`
- `paragraph_boundary`
- `list_boundary`
- `table_boundary`
- `code_block_boundary`
- `blockquote_boundary`
- `sentence_boundary`
- `length_fallback`

#### `generation_path`

表示当前 `chunk` 的简化生成路径。

第一阶段建议采用以下枚举值：

- `recursive_only`
- `recursive_plus_merge`
- `recursive_plus_resplit`
- `recursive_merge_resplit`
- `manual_adjusted`

## 九、人工复核状态设计

### 1. 目标

`review_status` 用于表达当前 `chunk` 是否已被人工确认、否定或修改。

该字段的目标包括：

- 区分系统自动产物与人工确认结果
- 为后续人工校正流程预留统一状态位
- 支撑后续质量抽查和治理闭环

### 2. 第一阶段枚举值

第一阶段正式采用以下状态：

- `unreviewed`
- `accepted`
- `rejected`
- `edited`

### 3. 状态说明

#### `unreviewed`

- 系统自动生成
- 尚未经过人工确认

#### `accepted`

- 经人工确认可用

#### `rejected`

- 经人工确认不可用
- 当前块不应继续作为正常高质量 `chunk` 使用

#### `edited`

- 经人工修改
- 当前块已不完全等同于系统原始产物

## 十一、与前序设计的关系

本任务与前序文档的关系如下：

- `07.01` 提供 `section` 结构归属与层级上下文
- `07.02` 提供 `chunk` 的递归切分、LLM 校正和 overlap 主流程
- `07.03` 提供结构化内容的边界与合并例外规则
- `07.04` 提供 `chunk` 的 offset、行号和页码定位基础
- `07.05` 提供后续摘要、标签等下游加工所依赖的 `chunk` 基础单元

因此，`7.6` 的定位不是替代前述设计，而是为 `chunk` 增加一层质量治理与可追溯治理能力。

## 十二、最终结论

`7.6` 第一阶段正式采用以下方案：

- 仅对 `chunk` 建立质量评分体系
- 评分采用 5 维度模型：
  - `boundary_integrity`
  - `semantic_completeness`
  - `retrieval_fitness`
  - `citation_fitness`
  - `size_fitness`
- 权重采用检索优先方案：
  - `0.30 / 0.30 / 0.20 / 0.15 / 0.05`
- `LLM` 负责基于 rubric 分档判断
- 系统负责将分档映射为数值并计算 `chunk_quality_score`
- 第一阶段不额外设计 `chunk_quality_level`
- 第一阶段保留 `generation_source`、`dependency_trace` 和 `review_status`
- 该设计为后续质量抽查、人工校正、检索调优和知识管理增强提供统一基础
