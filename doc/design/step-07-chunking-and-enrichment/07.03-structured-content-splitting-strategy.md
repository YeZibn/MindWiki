# 7.3 结构化内容切分策略设计

## 任务目标

定义第一阶段结构化内容的切分策略，明确表格、代码块、引用块、列表和混合块在 chunk 切分中的默认处理方式、允许拆分的条件，以及与前后正文的关系表达方式。

本子任务对应路线图中的 `7.3`：

- 明确表格、代码块、引用块、列表等结构化内容的切分策略

## 一、设计定位

`7.3` 不是重新定义 chunk 主流程，而是进一步细化：

- 结构化内容是否整体保留
- 何时允许拆分
- 拆分时沿什么边界拆
- 如何表达结构化内容与前后正文的关系
- 检索和引用时如何保持其可读性与可追溯性

## 二、总体原则

第一阶段对结构化内容采用以下总原则：

### 1. 结构完整性优先

不要轻易把天然整体结构切碎。

### 2. 语义可引用优先

如果切开后无法稳定作为证据引用，就不应随意拆分。

### 3. 关联关系必须保留

结构化内容经常依赖前后说明文字，因此即使不合并，也应保留它们之间的关系。

### 4. 默认整体保留，必要时再拆

第一阶段默认先整体保留，只有在超长、明显多组、或检索精度明显受损时才允许拆分。

## 三、覆盖范围

第一阶段重点覆盖以下 5 类结构化内容：

1. 表格
2. 代码块
3. 引用块
4. 列表
5. 混合块

## 四、表格切分策略

### 1. 默认策略

表格优先整体作为一个结构块。

正式策略：

- 表格本体默认单独成为一个 `table` chunk
- 表格前后的说明文字默认不直接并入表格 chunk
- 但应保留说明文字与表格之间的关联关系

### 2. 表格说明文字

例如：

```text
各模型对比如下：

| 模型 | 优点 | 缺点 |
| --- | --- | --- |
```

默认建议：

- `各模型对比如下：` 作为普通正文或混合判断候选
- 表格本体作为独立 `table` chunk
- 两者建立邻接或引用关系

### 3. 何时允许合并

如果说明文字与表格存在强依赖，例如：

- “如下表所示”
- “见下表”
- “对比如下”

则可由 LLM 判断是否将说明文字与表格合并为 `mixed` chunk。

### 4. 长表格处理

正式策略：

- 长表格优先整体保留
- 如果整体过大，则按逻辑行组拆分
- 不按字符长度或机械行数硬切

逻辑行组可以依据：

- 表头分组
- 小节分组
- 明显主题分区

## 五、代码块切分策略

### 1. 默认策略

代码块优先整体保留。

正式策略：

- 代码块本体默认单独成为 `code_block` chunk
- 代码前后的解释文字默认不直接并入代码块

### 2. 说明文字与代码块关系

例如：

```text
如下代码所示：

```python
def search():
    pass
```
```

默认建议：

- “如下代码所示：” 作为普通正文候选块
- 代码本体作为 `code_block` chunk
- 两者保留关联关系

### 3. 何时允许合并

如果说明文字与代码块强依赖，且单独拆开会破坏语义完整性，则可由 LLM 判断：

- 合并为 `mixed` chunk

### 4. 长代码块处理

正式策略：

- 超长代码块不按长度硬切
- 优先按逻辑段拆分

逻辑段可以依据：

- 函数
- 类
- 配置块
- 模块边界
- 注释分段

## 六、引用块切分策略

### 1. 默认定位

引用块默认属于正文体系的一部分，不自动降级为补充内容。

### 2. 默认策略

- 完整引用块可单独成为 `blockquote` chunk
- 若引用块本身语义完整，则优先整体保留

### 3. 与引导句关系

例如：

```text
作者指出：

> 检索质量决定了生成上限。
```

正式策略：

- 引导句和引用块默认可分开
- 如果引导句单独存在语义不足，可由 LLM 判断是否合并为 `mixed` chunk

## 七、列表切分策略

### 1. 默认策略

- 短列表优先整体保留
- 长列表按语义组拆分
- 不把单个列表项从中间切开

### 2. 列表引导句

例如：

```text
主要包括：

1. 检索
2. 重排
3. 生成
```

正式策略：

- 像“主要包括：”“如下几点：”这类引导句，默认可与后面的短列表合并
- 合并后可形成 `list` 或 `mixed` chunk，具体由实现层决定

### 3. 长列表处理

当列表过长时：

- 按语义组拆
- 按子列表拆
- 按逻辑主题拆

而不是：

- 按字符数硬切
- 在单个列表项中间截断

## 八、混合块策略

### 1. 为什么需要 `mixed`

真实文档中经常出现以下情况：

- 一句引导 + 表格
- 一句说明 + 代码块
- 一句总结 + 列表
- 一句引导 + 引用块

如果只允许纯类型块，会导致：

- 很多结构强依赖内容被强行拆散
- 语义完整性下降

### 2. 正式策略

第一阶段明确支持 `mixed` chunk 类型。

适用场景包括：

- 引导句与表格强依赖
- 引导句与代码块强依赖
- 引导句与引用块强依赖
- 短说明与短列表强依赖

### 3. 使用边界

`mixed` 不是默认类型，而是例外增强类型。

正式要求：

- 只有在拆开后明显破坏语义完整性时，才采用 `mixed`
- 不应把大段正文与大型结构块随意混拼

## 九、与 LLM 语义校正的关系

结构化内容仍然服从 `7.2` 的主流程：

- 先递归切分生成候选块
- 再由 LLM 做 `keep / merge / re_split`

在结构化内容场景下，LLM 主要负责判断：

- 说明文字是否要与结构块合并
- 长列表是否要分组
- 长表格是否已超出可检索范围
- 引导句是否单独成块语义不足

## 十、与引用展示的关系

结构化内容在引用展示中应保留类型信息。

例如：

- 表格引用
- 代码块引用
- 引用块引用
- 列表引用

这样做的好处是：

- 用户知道证据来自何种结构
- 前端后续可做差异化渲染

## 十一、建议数据字段

第一阶段建议在 chunk 层补充以下字段：

```json
{
  "chunk_type": "paragraph | list | table | code_block | blockquote | mixed",
  "related_chunk_ids": ["uuid-1", "uuid-2"],
  "structure_group_id": "uuid-or-null",
  "is_structured_content": true
}
```

字段说明：

- `chunk_type`：结构类型
- `related_chunk_ids`：与当前块强关联的说明块或结构块
- `structure_group_id`：同一结构组标识
- `is_structured_content`：是否属于结构化内容

## 十二、最终结论

`7.3` 正式采用以下结构化内容切分策略：

- 表格优先整体保留，说明文字默认分开但保留关联
- 代码块优先整体保留，说明文字默认分开但可强依赖合并
- 引用块默认属于正文体系，可单独保留或与引导句合并
- 列表优先整体或按语义组切分，不切断单个列表项
- 第一阶段明确支持 `mixed` chunk 类型
- 长表格按逻辑行组拆，长代码块按逻辑段拆
- 是否合并说明文字与结构块，由 LLM 在统一语义校正阶段判断

该设计将为后续 `7.4` 定位映射、`8.x` 检索索引和前端结构化引用展示提供基础。
