# 7.2 Chunk 大小、Overlap 与边界规则设计

## 任务目标

定义第一阶段 chunk 切分的完整策略，明确 chunk 的目标、候选块生成方式、递归切分边界优先级、LLM 语义校正流程、`15% overlap` 规则，以及失败回退机制。

- 设计 chunk 大小、重叠长度和特殊内容边界规则

## 一、设计目标

chunk 的目标不是把文档平均切碎，而是形成一组：

- 语义相对完整
- 可稳定检索
- 可准确引用
- 可回溯原文位置

的最小实用单元。

第一阶段正式采用以下核心原则：

- 语义完整性优先于长度整齐
- 结构边界优先于机械长度
- `section` 是 chunk 的第一层边界
- 结构递归切分为基础
- LLM 语义校正为统一增强层

## 二、总体方案

第一阶段 chunk 切分采用：

- 结构递归切分为基础
- LLM 语义校正为增强

完整流程如下：

```text
Document
  -> Section Split
  -> Section 内递归切分，生成全部候选块
  -> 全量候选块进入 LLM 语义校正
  -> 生成最终 chunk
  -> 对最终 chunk 加 15% overlap
  -> 去重与落库
```

说明：

- 不做“只挑高风险候选块进入 LLM”的筛选逻辑
- 全部候选块统一进入 LLM 校正
- 保持流程一致、行为一致、实现一致

## 三、Chunk 与 Section 的关系

`section` 是 chunk 的第一层结构容器。

正式策略：

- chunk 默认不跨 `section`
- 匿名 `section` 也允许正常切分 chunk
- chunk 的结构归属由 `section_id` 明确记录

这样做的原因：

- 防止跨节拼接破坏语义
- 保持引用展示清晰
- 保持检索结果的结构归属稳定

## 四、候选块定义

候选块是：

- section 内基于结构边界、内容类型和长度约束生成的第一轮 chunk 草稿

候选块不是最终 chunk，而是 LLM 语义校正的输入。

其定位是：

- 先保证结构上不乱切
- 再由 LLM 做语义边界修正

## 五、候选块生成方式

### 1. 在 Section 内进行

候选块只在单个 section 内生成，不跨 section 直接切分。

### 2. 采用递归切分

每个 section 内内容先按高优先级边界切分。

若切分后块仍过长，则继续按更低一级边界递归拆分。

### 3. 候选块不追求等长

候选块阶段的目标不是长度统一，而是：

- 结构合理
- 初步可读
- 可作为 LLM 校正输入

## 六、初始递归切分边界优先级

第一阶段从高到低采用以下优先级：

1. `section` 边界
2. 标题 / 子标题边界
3. 表格整体边界
4. 代码块整体边界
5. 引用块整体边界
6. 列表整体或子列表边界
7. 段落边界
8. 句子边界
9. 长度兜底切分

这套优先级的含义是：

- 先守住结构
- 再守住完整语义块
- 最后才使用句子和长度兜底

## 七、递归切分原则

第一阶段递归切分遵循以下规则：

1. 先按高优先级结构边界切
2. 若块仍明显过长，则继续向下一层边界递归
3. 只有最后才使用句子和长度兜底
4. 不追求数学上等长，只追求结构上合理
5. 切分过程必须保留 offset、页码、行号等定位信息

## 八、LLM 语义校正

### 1. LLM 的角色

LLM 不负责从零切分全文。

LLM 只负责对递归切分后的候选块做边界校正。

### 2. LLM 处理对象

第一阶段正式采用：

- 全部候选块统一进入 LLM 语义校正

不再区分：

- 高风险候选块
- 低风险候选块
- 规则直通块

### 3. LLM 可执行动作

LLM 对每个候选块只允许输出以下决策之一：

- `keep`
- `merge_with_prev`
- `merge_with_next`
- `re_split`

说明：

- `keep`：当前块已语义完整，直接保留
- `merge_with_prev`：需要和前一个相邻块合并
- `merge_with_next`：需要和后一个相邻块合并
- `re_split`：当前块内部语义过长或多主题，需要再拆分

### 4. LLM 输入窗口

每次判断时，建议输入给 LLM：

- 当前 `section` 标题
- 前一个候选块
- 当前候选块
- 后一个候选块
- 当前块类型
- 当前块长度信息
- 当前块内容角色

即采用局部窗口判断，而不是全篇重构。

## 九、语义合并触发条件

LLM 在以下情况下应倾向合并：

1. 当前块太短
2. 当前块只有引导句、承接句、过渡句
3. 标题后只有一小句引导文本
4. 出现“如下所示”“见下表”“如下代码”等强依赖相邻块的表达
5. 短列表块、短引用块单独存在时语义不成立
6. 相邻块主题明显连续，拆开后反而破坏语义完整性

### 合并限制

正式限制如下：

- 只允许与相邻块合并
- 默认不跨 `section`
- 默认不同 `content_role` 不合并
- 表格、代码块优先整体保留，不随意与大段正文混拼

## 十、语义再拆分触发条件

LLM 在以下情况下应倾向再拆分：

1. 当前块过长
2. 一个块中包含多个明显子主题
3. 一个长列表内部已经有多个独立语义组
4. 一个长段同时包含定义、方法、结论等多个语义单元
5. 当前块虽然结构完整，但不适合作为精确检索或精确引用单元

### 再拆分原则

正式顺序如下：

1. 优先按子段落拆
2. 其次按句子组拆
3. 最后按长度兜底拆
4. 再拆分后必须保留 offset、页码、行号映射关系

## 十一、结构化内容特殊规则

### 1. 表格

正式策略：

- 默认整体保留
- 若超长，可按逻辑行组拆
- 不建议和普通正文机械拼接

### 2. 代码块

正式策略：

- 默认整体保留
- 不轻易从中间截断
- 若极长，可按逻辑段拆

### 3. 引用块

正式策略：

- 默认整体看待
- 若与上下引导句强依赖，可由 LLM 判断是否合并

### 4. 列表

正式策略：

- 短列表优先整体保留
- 长列表按语义组拆
- 不把单个列表项从中间切开

## 十二、Chunk 大小控制原则

第一阶段不在设计文档中写死唯一数值阈值，但正式要求：

- chunk 大小需要受控
- 不能无限放大“完整语义”
- 不能让单块过长到损害检索精度和引用粒度

因此：

- 长度约束是第二层规则
- 语义完整性是第一层规则
- 两者冲突时先保结构和语义，再做受控再拆分

## 十三、Overlap 规则

### 正式规则

- 最终 chunk 默认采用 `15% overlap`
- overlap 在 LLM 校正完成后生成
- overlap 只作用于相邻 chunk

### 适用范围

主要适用于：

- 普通正文 chunk

不强制机械 적용于：

- 表格
- 代码块
- 极短 chunk
- 已经高度完整的结构块

### 设计目的

- 保住边界上下文
- 避免边界处信息断裂
- 提高检索和回答稳定性

### 说明

overlap 本身会引入部分重复，因此后续必须配合：

- `exact_duplicate`
- `near_duplicate`
- 检索折叠
- 上下文降冗余

但这些属于后续检索编排层，不在 `7.2` 中解决。

## 十四、失败与回退机制

如果 LLM 语义校正失败：

- 不阻断整个导入
- 回退到递归切分结果
- 记录 warning，例如 `semantic_chunk_refine_failed`

这意味着：

- LLM 是增强层
- 不是导入链路的单点阻塞器

## 十五、最终 Chunk 建议字段

第一阶段建议最终 chunk 至少包含以下字段：

```json
{
  "chunk_id": "uuid",
  "document_id": "uuid",
  "section_id": "uuid",
  "chunk_index": 3,
  "content_role": "body",
  "chunk_type": "paragraph | list | table | code_block | blockquote | mixed",
  "text": "chunk正文",
  "start_offset": 1200,
  "end_offset": 1680,
  "page_start": 3,
  "page_end": 4,
  "line_start": 40,
  "line_end": 58,
  "overlap_prev": true,
  "overlap_next": true,
  "generation_path": "recursive_only | recursive_plus_merge | recursive_plus_resplit"
}
```

字段说明：

- `section_id`：chunk 所属 section
- `chunk_index`：section 内顺序
- `content_role`：正文 / 结构 / 补充内容角色
- `chunk_type`：块类型
- `start_offset` / `end_offset`：标准化文本偏移
- `page_start` / `page_end`：PDF 定位
- `line_start` / `line_end`：Markdown 定位
- `overlap_prev` / `overlap_next`：是否与前后块存在 overlap
- `generation_path`：该块是纯递归、递归后合并，还是递归后再拆分

## 十六、最终结论

`7.2` 正式采用以下 chunk 切分策略：

- chunk 切分采用“结构递归切分 + LLM 语义校正”的混合方案
- `section` 是 chunk 的第一层边界，chunk 默认不跨 `section`
- 初始候选块由 section 内递归切分生成
- 全部候选块统一进入 LLM 语义校正，不再做额外筛选
- LLM 只负责 `keep / merge / re_split` 的边界校正，不负责从零全文切分
- 表格、代码块、引用块优先保持结构完整
- 最终 chunk 默认采用 `15% overlap`
- LLM 校正失败时回退到递归切分结果，不阻断导入

该设计将为后续 `7.3` 结构化内容切分、`7.4` 定位映射格式、`8.x` 检索索引和 `10.x` 引用展示提供核心基础。
