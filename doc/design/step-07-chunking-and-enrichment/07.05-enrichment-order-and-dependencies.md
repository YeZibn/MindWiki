# 7.5 摘要、关键词、标签、实体的生成顺序与依赖关系设计

## 任务目标

定义第一阶段知识加工链路中摘要、关键词、标签、实体的生成顺序、依赖关系和存储定位，避免上游结果不稳定污染下游，同时让加工链路既可落地又便于后续扩展。

本子任务对应路线图中的 `7.5`：

- 定义摘要、关键词、标签、实体的生成顺序与依赖关系

## 一、设计目标

本任务需要回答的核心问题不是“要不要做摘要、关键词、标签、实体”，而是：

- 这些结果按什么顺序生成
- 谁依赖谁
- 哪些是第一阶段必须做的基础信号
- 哪些应保守处理，避免过度生成

## 二、总体原则

第一阶段采用以下总原则：

### 1. 以 Chunk 级加工为主

第一阶段知识加工以 chunk 为基础单元。

原因：

- chunk 是检索和引用的最小实用单元
- chunk 级结果更细粒度、更可追溯
- document 级结果更适合由 chunk 级结果聚合而来

### 2. 先抽取，再归纳

比起先做摘要和标签，第一阶段更适合先做：

- 关键词抽取
- 实体抽取

再进一步做：

- 摘要
- 标签候选

这样可以降低“上来就生成一层不稳定归纳结果”的风险。

### 3. 标签要保守

标签面向用户使用，比关键词更稳定、更少量、更可控。

因此：

- 标签不应直接从原文粗暴生成
- 标签应参考关键词、实体和摘要

### 4. 高层结果由低层结果聚合

section / document 级结果尽量由 chunk 级结果向上聚合，而不是跳过 chunk 直接重做整文生成。

## 三、加工对象分层

第一阶段将知识加工对象分为两层：

1. chunk 级
2. section / document 级

## 1. Chunk 级

处理对象是单个 chunk。

包括：

- chunk 关键词
- chunk 实体
- chunk 摘要
- chunk 标签候选

## 2. Section / Document 级

处理对象是 section 或 document。

包括：

- section 摘要
- document 摘要
- document 标签
- 文档级聚合关键词或实体

第一阶段正式建议：

- 以 chunk 级加工为主
- 高层级结果做轻量聚合
- 不做过重的高级主题建模

## 四、正式生成顺序

第一阶段正式推荐以下顺序：

```text
Chunk Text
  -> 关键词提取
  -> 实体提取
  -> Chunk 摘要
  -> 标签候选生成
  -> Section / Document 聚合
```

## 五、各步骤设计说明

## 1. 关键词提取

### 定位

关键词是第一阶段的轻量抽取结果。

主要作用：

- 提炼 chunk 核心词
- 辅助后续标签生成
- 辅助检索增强
- 作为聚合输入

### 设计建议

- 关键词直接基于 chunk 原文生成
- 不依赖摘要和标签
- 不把关键词直接当最终标签

## 2. 实体提取

### 定位

实体是比关键词更结构化的抽取结果。

可能包括：

- 人名
- 机构名
- 产品名
- 法律术语
- 医学术语
- 技术概念

### 第一阶段策略

- 实体先做抽取
- 不做复杂图谱化
- 不做复杂实体消歧

### 作用

- 为后续跨文档关联提供基础
- 为未来主题聚合或知识图谱预留输入
- 为专业资料理解提供额外信号

## 3. Chunk 摘要

### 定位

chunk 摘要是对单个 chunk 的压缩表达。

### 作用

- 检索结果展示
- 上下文构建辅助
- section / document 聚合输入

### 第一阶段策略

- 摘要依赖 chunk 原文
- 可参考关键词与实体结果
- 摘要不能替代原文
- 摘要不应作为唯一证据

## 4. 标签候选生成

### 定位

标签是面向用户使用的知识组织入口。

与关键词的区别：

- 标签更稳定
- 标签更少量
- 标签更适合筛选和组织

### 第一阶段策略

- 先生成“标签候选”
- 不把每个 chunk 的标签候选直接当最终文档标签
- 标签候选应参考：
  - chunk 原文
  - 关键词
  - 实体
  - chunk 摘要

## 六、依赖关系建议

第一阶段建议的依赖关系如下：

```text
Chunk Text
  -> keywords
  -> entities
  -> chunk_summary
  -> tag_candidates
  -> section_summary / document_summary
  -> document_tags
```

更明确地说：

- `keywords` 依赖 chunk 原文
- `entities` 依赖 chunk 原文
- `chunk_summary` 依赖 chunk 原文，可参考 keywords / entities
- `tag_candidates` 依赖 chunk 原文，并参考 keywords / entities / chunk_summary
- `section_summary` 和 `document_summary` 依赖多个 chunk 结果聚合
- `document_tags` 依赖 chunk 级标签候选聚合

## 七、Section / Document 级聚合策略

### 1. Section 摘要

section 摘要建议基于：

- section 下的关键 chunk
- chunk 摘要
- 聚合后的关键词与实体

### 2. Document 摘要

document 摘要建议基于：

- document 内的关键 section
- 关键 chunk 摘要
- 文档级聚合关键词与实体

### 3. Document 标签

document 标签建议基于：

- chunk 标签候选
- 高频关键词
- 高频实体
- 用户已有标签

说明：

- document 标签是聚合后的稳定结果
- 不应简单等于所有 chunk 标签候选的并集

## 八、候选结果与最终结果

第一阶段建议明确区分：

- 候选结果
- 最终结果

### 候选结果

例如：

- chunk 标签候选
- 原始关键词集合
- 原始实体集合

### 最终结果

例如：

- document 最终标签
- 对外展示摘要

这样做的好处：

- 保留中间信号，便于后续迭代
- 降低一次生成错误对最终输出的污染
- 便于后续人工校正或规则调整

## 九、存储建议

第一阶段正式建议：

- 摘要、关键词、标签、实体四类结果都保留
- 但要区分候选层和最终层

可落在：

- chunk 扩展字段
- 独立 enrichment 表
- 或候选结果表 + 聚合结果表

具体工程实现可后续细化，但设计上应保留这一区分。

## 十、第一阶段默认技术路线

在不锁死具体库和模型名称的前提下，第一阶段建议采用以下默认技术路线。

## 1. 关键词提取

关键词的定位是“偏抽取”，因此第一阶段优先采用：

- 轻量
- 稳定
- 低成本
- 可解释

的技术方案。

正式建议：

- 优先使用规则 / 统计 / 轻量算法方法
- 可选用 LLM 做少量增强或清洗
- 不建议把 LLM 作为关键词提取的唯一主方案

原因：

- 关键词更接近原文抽取任务
- 需要稳定可重复
- 成本需要可控
- 可解释性比表达自然度更重要

## 2. 标签生成

标签的定位是“偏归纳”，因此第一阶段优先采用：

- LLM 生成标签候选
- 再用规则做归一化与收敛

正式建议：

- LLM 负责生成候选标签
- 规则层负责数量控制、去重、归一化、聚合和映射
- 不建议直接把 chunk 原文或词频结果当最终标签

原因：

- 标签是面向用户使用的知识组织入口
- 标签比关键词更需要稳定表达和语义归纳
- 直接从原文硬抽标签容易噪音过多

## 3. 与前面依赖关系的结合

结合本任务前文，第一阶段建议落地为：

- 关键词：偏抽取，优先轻量、稳定、低成本方法
- 实体：偏抽取，保守识别，不做复杂图谱化
- 摘要：基于 chunk 原文生成，可参考关键词与实体
- 标签：偏归纳，优先 LLM 候选生成，再做规则收敛

这样可以在保证稳定性的同时，把更“生成式”的能力集中在标签层，而不是让整条链路都过度依赖 LLM。

## 十一、第一阶段不做的内容

以下内容不在 `7.5` 第一阶段深入展开：

- 复杂知识图谱构建
- 多实体消歧系统
- 自动主题树生成
- 大规模聚类主题发现
- 用户反馈驱动的标签自动修正闭环

这些能力可以在后续增强阶段继续扩展。

## 十二、最终结论

`7.5` 正式采用以下知识加工顺序与依赖关系：

- 第一阶段以 chunk 级加工为主
- 先做关键词提取和实体抽取
- 再做 chunk 摘要
- 再生成标签候选
- 最后向 section / document 层做轻量聚合
- 标签不直接从原文粗暴生成，而应参考关键词、实体和摘要
- 关键词技术路线采用偏抽取、轻量、稳定、低成本的方法
- 实体第一阶段只做抽取，不做复杂图谱化
- 标签技术路线采用“LLM 候选生成 + 规则收敛”的混合方式
- 摘要、关键词、标签、实体四类结果都保留，但要区分候选结果与最终结果

该设计将为后续 `7.6` 置信度与来源标记、`8.x` 检索信号构建和 `13.x` 高阶知识组织能力提供基础。
