# 6.4 OCR 触发条件设计

## 任务目标

定义第一阶段 PDF 导入链路中的 OCR 触发判断规则，明确何时应判定一份文档超出当前文本抽取能力范围，何时直接失败，以及未来 OCR 能力应如何接入预处理流程。

本子任务对应路线图中的 `6.4`：

- 设计 OCR 触发条件，例如文本提取为空或文本密度过低

## 第一阶段定位

第一阶段对 OCR 的定位是：

- 当前不实现 OCR
- 当前只做“是否需要 OCR”的判断
- 一旦确认需要 OCR，则直接失败
- 明确提示用户“当前阶段不支持 OCR，后续会增加”

这意味着第一阶段的重点不是 OCR 识别本身，而是建立稳定、保守、可解释的触发判定机制。

## 设计原则

### 1. OCR 不是 PDF 默认流程

不是所有 PDF 都需要 OCR。

只有在文本抽取结果不足以支撑后续：

- 检索
- 引用定位
- 正文抽取
- chunk 切分

时，才应判定“需要 OCR”。

### 2. 失败优于假成功

如果一份 PDF 实际无法可靠抽取正文，就不应：

- 导入一个空文档
- 导入一个几乎不可检索的文档
- 让后续回答基于错误或缺失文本继续工作

因此第一阶段采用：

- 判定需要 OCR -> 直接失败

### 3. 采用多信号组合判断

OCR 触发判断不应依赖单一阈值。

原因：

- 不同 PDF 类型差异很大
- 单页短文档和长文档分布不同
- 法律、医学、学习资料的版式差异大
- 仅靠字符数容易误判

因此第一阶段采用：

- 多信号组合判断
- 区分强信号和弱信号

### 4. 按整份文档判定

第一阶段不做“部分页 OCR、部分页直接导入”的混合处理。

正式策略：

- OCR 需求按整份文档判定
- 一旦整份文档被认定需要 OCR，则整份导入失败

这样做的原因：

- 实现简单
- 行为一致
- 避免部分页缺失导致引用链断裂
- 避免知识库中出现半可用文档

## OCR 触发信号

第一阶段建议从以下信号判断一份 PDF 是否需要 OCR。

## 1. 强信号 `strong_signal`

出现强信号时，通常可直接判定该文档需要 OCR。

### 1.1 文本抽取结果为空

表现：

- 整份 PDF 抽取后无正文文本
- 所有页面文本均为空或仅有极少控制字符

处理：

- 直接判定需要 OCR
- 导入失败

### 1.2 大多数页面无可用文本

表现：

- 空文本页占比极高
- 绝大多数页面几乎没有正文字符

处理：

- 直接判定需要 OCR
- 导入失败

### 1.3 文档明显为扫描图片页

表现：

- 页面主要由整页图片构成
- 抽取层几乎无法识别文本对象
- 页面看起来是扫描件而非数字文本 PDF

处理：

- 直接判定需要 OCR
- 导入失败

### 1.4 标准化后正文几乎为空

表现：

- 仅剩页码、页眉页脚、少量噪音字符
- 无法形成稳定正文主干

处理：

- 直接判定需要 OCR 或不可用
- 导入失败

## 2. 弱信号 `weak_signal`

弱信号单独出现时不一定直接失败，但应参与组合判断。

### 2.1 平均每页字符数过低

表现：

- 文本量明显低于正常正文型 PDF
- 仅有零散短词或短句

说明：

- 对封面页、目录页较多的文档不能单独用此信号定死
- 应结合页数、正文页分布等一起判断

### 2.2 异常字符比例过高

表现：

- 大量乱码
- 控制字符异常集中
- 文本可读性明显受损

说明：

- 这可能是编码问题，也可能是 OCR 需求问题
- 不应仅凭这一项直接判定

### 2.3 提取文本分布极不稳定

表现：

- 少数页面有正常文本，多数页面接近空白
- 页面之间文本密度差异极大

说明：

- 可能意味着混合文档
- 第一阶段仍按整份文档处理

### 2.4 可抽取内容主要是边缘信息

表现：

- 主要抽到页码、页眉、版权、模板内容
- 正文主干缺失

说明：

- 这是 OCR 需求的常见前置信号
- 应结合正文抽取结果共同判断

## 第一阶段判定策略

### 正式策略

第一阶段采用以下判定逻辑：

1. 若命中任一高置信度强信号，则直接判定 `ocr_required_but_not_supported`
2. 若未命中强信号，但多个弱信号共同出现，则升级为需要 OCR
3. 若仅出现个别弱信号，则不直接失败，但记录 warning
4. 若文本抽取能够支撑正文、位置映射和后续引用，则不触发 OCR 判定

### 推荐判定顺序

```text
PDF 文本抽取
  -> 计算基础质量指标
  -> 判断强信号
  -> 若无强信号，判断弱信号组合
  -> 形成 OCR 判定结果
  -> 成功继续 / 失败退出
```

## 建议采集的质量指标

为了支持 OCR 判断，第一阶段建议至少采集以下指标：

- `page_count`
- `text_extracted_page_count`
- `empty_text_page_count`
- `average_chars_per_page`
- `median_chars_per_page`
- `garbled_char_ratio`
- `body_like_page_count`
- `header_footer_only_page_count`
- `image_only_page_count`，如底层库可识别

说明：

- 第一阶段不要求这些指标全部做到高精度
- 但至少应具备支撑“强信号 / 弱信号”判断的基础数据

## 输出结果建议

OCR 判断结果建议统一为结构化输出，便于后续错误处理和人工排障。

```json
{
  "ocr_decision": "not_needed | weak_signal | required",
  "decision_reason_codes": [
    "pdf_text_extraction_empty",
    "empty_page_ratio_high"
  ],
  "signal_level": "weak | strong",
  "metrics": {
    "page_count": 20,
    "empty_text_page_count": 18,
    "average_chars_per_page": 12
  }
}
```

字段说明：

- `ocr_decision`：OCR 决策结果
- `decision_reason_codes`：触发该结果的原因码
- `signal_level`：当前以强信号还是弱信号组合得出
- `metrics`：便于 CLI 展示和排障的辅助指标

## 与导入状态机的关系

当 OCR 判定为 `required` 时：

- 导入任务状态进入 `failed`
- reason code 使用 `ocr_required_but_not_supported`
- CLI 需明确提示用户当前阶段不支持 OCR

当 OCR 判定为 `weak_signal` 时：

- 不直接失败
- 允许继续进入标准化与正文抽取
- 同时记录 warning，提醒后续链路保持保守

## 与 warning / error 分类的关系

建议使用以下原因码或 warning：

- `ocr_required_but_not_supported`
- `pdf_text_extraction_empty`
- `low_text_density`
- `garbled_text_candidate`
- `header_footer_only_content`

其中：

- `ocr_required_but_not_supported` 应作为阻断错误
- 其他项可作为判定 OCR 的辅助依据或 warning

## 与后续 OCR 扩展的关系

未来接入 OCR 后，推荐流程为：

```text
PDF 输入
  -> 常规文本抽取
  -> OCR 判定
  -> 若 not_needed，直接进入标准化
  -> 若 required，进入 OCR fallback
  -> OCR 结果重新进入标准化
```

这样设计的好处是：

- 当前阶段可以先把判断逻辑做稳定
- 后续增加 OCR 时不需要推翻预处理主链路
- OCR 被视为“抽取失败后的补救分支”，而不是默认主路径

## 第一阶段明确不做的事情

以下内容不在 `6.4` 第一阶段范围内：

- OCR 引擎选型
- OCR 模型质量评估
- 部分页 OCR、部分页直抽的混合导入
- 用户手动确认是否对单份文件执行 OCR
- 导入空文档后等待未来补 OCR

## 最终结论

`6.4` 正式采用以下设计：

- 第一阶段不实现 OCR，只实现 OCR 需求判定
- OCR 需求按整份文档判断，不做分页混合策略
- 一旦判定需要 OCR，则导入失败
- 失败原因明确为 `ocr_required_but_not_supported`
- 采用“强信号 + 弱信号组合”的多信号判断机制
- 未来将 OCR 作为 PDF 文本抽取失败后的 fallback 分支接入

该设计将为后续 OCR 扩展、导入失败提示、质量监控和人工介入流程提供统一基础。
