# 6.1 输入标准化流程设计

## 任务目标

定义 Markdown、PDF 和预留 Web Clip 的输入标准化流程，将原始文件内容转换为后续解析、切分、索引和生成链路可以稳定消费的标准化内容。

本子任务对应路线图中的 `6.1`：

- 为 Markdown、PDF、Web Clip 分别定义输入标准化流程

## 关联文档

- `05.01-unified-import-entry.md`
- `05.03-markdown-and-pdf-adapter-protocol.md`
- `05.05-import-error-and-warning-taxonomy.md`
- `05.06-retry-and-manual-intervention.md`
- `07.01-section-splitting-rules.md`，后续补充
- `07.02-chunk-size-overlap-and-boundary-rules.md`，后续补充

## 设计定位

输入标准化层位于：

```text
导入入口
  -> 接入适配器
  -> 输入标准化
  -> 文档解析与 section 切分
  -> chunk 切分
  -> 知识加工
  -> 索引构建
```

输入标准化层的核心职责是：

- 将不同来源的原始内容转换为统一的文本表达
- 保留来源、位置、结构提示和质量信息
- 尽量清除明显噪音
- 为后续 section / chunk 流程提供稳定输入

输入标准化层不负责：

- 不负责最终 section 切分
- 不负责 chunk 切分
- 不负责 embedding
- 不负责向量索引
- 不负责生成摘要或回答
- 不负责深度语义理解

## 总体原则

### 0. 原始内容与标准化内容同时保留

标准化后的文本用于后续解析、切分、检索和生成，但不能替代原始文件。

系统应同时保留：

- 原始文件
- 原始文件哈希
- 标准化后的文本
- 标准化文本与原文位置的映射
- 标准化过程中产生的 warning

这样做的原因是：

- 标准化规则可能误删或误合并内容
- 后续引用需要回跳原始文件
- 后续规则升级后可能需要重新标准化
- 用户需要知道回答依据来自原始资料的具体位置

### 1. 保守标准化

标准化阶段只处理确定性较强的问题，例如：

- 编码统一
- 换行统一
- 空白字符规整
- 明显乱码标记
- 明显重复空段落移除
- 基础结构提示提取

对于不确定内容，不应直接删除，而应：

- 保留原文
- 增加 warning
- 交给后续解析或人工判断

### 2. 可追溯

标准化后的内容必须能追溯到原始来源。

至少需要保留：

- 原始文件路径
- source id
- document id，若已生成
- source type
- 文件哈希
- 页码或行号等位置线索

### 3. 不制造内容

标准化层不得补写原文中不存在的事实内容。

允许做：

- 格式归一
- 结构提示
- 噪音标记
- 文本质量判断

不允许做：

- 自动扩写
- 自动总结
- 自动解释
- 用模型补全缺失段落

### 4. 结构优先于美化

标准化输出应优先服务检索、引用和回跳，而不是追求文本排版美观。

例如：

- PDF 中的页码必须保留
- Markdown 中的标题层级必须保留
- 表格、代码块、引用块应保留结构提示
- 链接应保留原始目标

## 标准化输入

标准化流程的输入来自导入适配器，建议结构如下：

```json
{
  "source_id": "uuid",
  "import_job_id": "uuid",
  "source_type": "markdown | pdf | web_clip",
  "original_path": "/absolute/path/to/file",
  "content_hash": "sha256",
  "mime_type": "text/markdown",
  "file_name": "example.md",
  "raw_content_ref": "local-file-or-extracted-content-ref",
  "adapter_metadata": {
    "encoding": "utf-8",
    "file_size": 12345
  }
}
```

说明：

- `raw_content_ref` 可以指向原始文件，也可以指向适配器抽取后的临时内容
- Markdown 通常直接读取原始文本
- PDF 通常需要先完成文本抽取，再进入标准化
- Web Clip 第一阶段只保留协议，不作为核心实现

## 标准化输出

标准化流程的输出建议统一为 `StandardizedDocumentInput`：

```json
{
  "source_id": "uuid",
  "import_job_id": "uuid",
  "source_type": "markdown | pdf | web_clip",
  "original_path": "/absolute/path/to/file",
  "content_hash": "sha256",
  "raw_content_ref": "local-original-file-ref",
  "standardized_content_ref": "local-standardized-text-ref",
  "title_candidates": [
    {
      "value": "文档标题",
      "source": "frontmatter | h1 | filename | pdf_metadata | first_heading",
      "confidence": 0.9
    }
  ],
  "standardized_text": "标准化后的全文",
  "metadata": {
    "encoding": "utf-8",
    "language_hint": "zh",
    "source_note": "用户导入时填写的来源备注"
  },
  "structure_hints": [],
  "position_map": [],
  "warnings": []
}
```

核心字段说明：

- `raw_content_ref`：原始文件或原始抽取内容的引用
- `standardized_content_ref`：标准化后文本的持久化引用
- `title_candidates`：标题候选，不在本阶段强行决定最终标题
- `standardized_text`：后续解析和切分使用的主文本
- `metadata`：来源、编码、语言、用户备注等补充信息
- `structure_hints`：标题、表格、代码块、页码等结构线索
- `position_map`：标准化文本与原文位置的映射
- `warnings`：标准化过程发现但不一定阻断导入的问题

## Markdown 标准化流程

### 输入特点

Markdown 的优势是结构天然较清晰，通常包含：

- YAML frontmatter
- 标题层级
- 段落
- 列表
- 表格
- 代码块
- 引用块
- 链接
- 图片引用
- Obsidian 双链

Markdown 标准化应尽量保留这些结构信息。

### 处理步骤

```text
读取文件
  -> 检测编码
  -> 统一编码为 UTF-8
  -> 统一换行符
  -> 解析 frontmatter
  -> 提取标题候选
  -> 保留 Markdown 结构
  -> 规整空白与空段落
  -> 提取结构提示
  -> 生成位置映射
  -> 输出标准化结果
```

### 编码与换行

Markdown 文件统一处理为：

- 编码：`UTF-8`
- 换行：`\n`
- 移除 UTF-8 BOM
- 连续多个空行可规整，但不应破坏代码块内部格式

如果编码无法可靠识别：

- 尝试按 UTF-8 读取
- 失败则记录 `encoding_detection_failed`
- 该文件可进入失败或人工介入路径

### Frontmatter 处理

YAML frontmatter 不直接并入正文主文本，但应解析为 metadata。

示例：

```markdown
---
title: RAG 系统设计
tags:
  - retrieval
  - llm
---
```

处理方式：

- `title` 进入 `title_candidates`
- `tags` 可进入后续标签候选
- 其他字段进入 `metadata.frontmatter`
- 原始 frontmatter 可选择保留在 metadata 中，便于追溯

如果 frontmatter 解析失败：

- 不阻断导入
- 将原始 frontmatter 作为普通文本或 metadata raw 保留
- 记录 `frontmatter_parse_failed`

### 标题候选提取

Markdown 标题候选优先级建议为：

1. frontmatter.title
2. 第一个 H1
3. 第一个 H2
4. 文件名

注意：

- 标准化阶段只产生候选，不强行决定最终标题
- 最终标题归属可在文档解析或 Document 创建阶段确定
- 这里的标题主要用于文档列表展示、检索结果展示、引用展示、标题权重检索和后续知识整理
- 这里的标题是 `Document.title` 的候选来源，不等同于 section 标题或 chunk 标题

### 链接处理

Markdown 链接应保留原始表达，同时抽取链接结构提示。

支持：

- 标准 Markdown 链接：`[文本](url)`
- 图片链接：`![alt](path)`
- Obsidian 双链：`[[目标页面]]`
- Obsidian 带别名双链：`[[目标页面|显示文本]]`

处理原则：

- 链接文本保留在正文中
- 链接目标进入 `structure_hints`
- 图片不做 OCR，不抽取图片内容
- 图片路径作为附件引用保留

### 代码块处理

代码块必须保留原始格式。

处理原则：

- 不规整代码块内部缩进
- 不删除代码块内部空行
- 记录 `code_block` 结构提示
- 可记录语言标记，例如 `python`、`sql`、`bash`

### 表格处理

Markdown 表格第一阶段保留原始 Markdown 表达。

处理原则：

- 不强行转换为自然语言句子
- 记录 `table` 结构提示
- 后续 chunk 切分时尽量保持表格完整

### Markdown 输出示例

```json
{
  "source_type": "markdown",
  "title_candidates": [
    {
      "value": "RAG 系统设计",
      "source": "frontmatter",
      "confidence": 0.95
    }
  ],
  "structure_hints": [
    {
      "type": "heading",
      "level": 1,
      "text": "整体架构",
      "line_start": 12,
      "line_end": 12
    },
    {
      "type": "code_block",
      "language": "python",
      "line_start": 30,
      "line_end": 45
    }
  ],
  "position_map": [
    {
      "standardized_start": 0,
      "standardized_end": 120,
      "line_start": 1,
      "line_end": 6
    }
  ],
  "warnings": []
}
```

## PDF 标准化流程

### 输入特点

PDF 的复杂度高于 Markdown，主要问题包括：

- 文本抽取顺序可能错乱
- 页眉页脚重复
- 断行严重
- 连字符断词
- 表格结构丢失
- 页码和段落位置需要额外记录
- 扫描版 PDF 可能无法抽取文本

第一阶段重点支持文本型 PDF。

扫描版 PDF 和图片型 PDF：

- 可检测
- 可记录 warning 或 error
- 当前阶段提示不支持 OCR
- 当前阶段导入失败，不生成不可检索的空文档
- 后续阶段一定预留 OCR 接入能力

### 处理步骤

```text
读取 PDF
  -> 检测是否可抽取文本
  -> 按页抽取文本
  -> 记录页码与基础位置
  -> 合并页内文本块
  -> 规整段落与断行
  -> 标记可能的页眉页脚
  -> 提取标题候选
  -> 生成 position_map
  -> 生成质量 warning
  -> 输出标准化结果
```

### 文本抽取质量检测

PDF 标准化需要记录文本质量指标：

- 总页数
- 成功抽取文本页数
- 空文本页数
- 平均每页字符数
- 异常字符比例
- 重复行比例
- 是否疑似扫描件

当抽取文本为空或文本密度极低时：

- 记录 `pdf_text_extraction_empty` 或 `low_text_density`
- 如果当前 OCR 未启用，则导入失败并提示“当前不支持 OCR”
- 不允许导入成功但生成不可检索的空文档

### 页码保留

PDF 标准化必须保留页码信息。

原因：

- 回答引用需要定位到具体页
- 用户需要回跳原始 PDF
- 后续引用展示不能只显示文档级来源

`position_map` 至少应支持：

```json
{
  "standardized_start": 1024,
  "standardized_end": 1450,
  "page_number": 3,
  "source_block_index": 8,
  "page_block_index": 2
}
```

第一阶段最低定位粒度为：

- PDF 页码
- 页内文本块序号

这样引用至少可以定位到“第几页的第几个文本块附近”，避免只给出文档级引用。

如果底层 PDF 解析库可以提供坐标，则可扩展：

```json
{
  "page_number": 3,
  "bbox": {
    "x1": 72,
    "y1": 120,
    "x2": 520,
    "y2": 180
  }
}
```

第一阶段不强制依赖坐标，但应预留字段。

### 断行与段落规整

PDF 常见问题：

```text
这是一个很长的句子，
因为 PDF 排版原因被强制换行。
```

标准化时可以尝试合并为：

```text
这是一个很长的句子，因为 PDF 排版原因被强制换行。
```

但需要保守处理：

- 标题行不强行合并
- 列表项不强行合并
- 表格疑似内容不强行合并
- 每次合并应尽量保留位置映射

### 连字符断词

英文 PDF 可能出现：

```text
retriev-
al augmented generation
```

可规整为：

```text
retrieval augmented generation
```

但中文资料中该问题较少，第一阶段可作为轻量规则处理。

### 页眉页脚处理

第一阶段不建议激进删除页眉页脚。

建议策略：

- 标记重复出现的页眉页脚候选
- 对高置信度重复页眉页脚可从主文本中移除
- 对低置信度内容保留并记录 warning

常见候选：

- 每页顶部重复文档标题
- 每页底部页码
- 公司名或资料来源
- 固定版权声明

### PDF 标题候选提取

PDF 标题候选优先级建议为：

1. PDF metadata title
2. 第一页高置信度大标题
3. 文件名

如果 PDF metadata title 明显为空、乱码或默认值：

- 降低置信度
- 不作为强标题
- 记录 `pdf_metadata_title_unreliable`

### PDF 输出示例

```json
{
  "source_type": "pdf",
  "title_candidates": [
    {
      "value": "2025 年学习资料汇总",
      "source": "pdf_metadata",
      "confidence": 0.7
    }
  ],
  "metadata": {
    "page_count": 20,
    "text_extracted_page_count": 20,
    "empty_text_page_count": 0,
    "average_chars_per_page": 1200
  },
  "structure_hints": [
    {
      "type": "page",
      "page_number": 1,
      "text_start": 0,
      "text_end": 1800
    }
  ],
  "warnings": []
}
```

## Web Clip 标准化流程

### 第一阶段定位

Web Clip 在路线图中保留，但不是当前第一阶段的核心输入重点。

当前第一阶段重点是：

- Markdown
- 文本型 PDF

Web Clip 先定义协议位置，暂不展开完整实现。

### 未来输入特点

Web Clip 可能包含：

- URL
- 标题
- 正文
- 作者
- 发布时间
- 抓取时间
- 网页正文 HTML
- Markdown 化正文
- 图片和附件链接

### 预留处理步骤

```text
读取 Web Clip
  -> 识别来源 URL
  -> 抽取正文
  -> 清理导航、广告、推荐内容
  -> 保留标题、作者、发布时间
  -> 保留原始 URL
  -> 输出标准化结果
```

### 暂不深入实现的原因

- 当前用户主要资料类型是工作资料和学习资料
- 第一阶段已明确优先支持 Markdown 和 PDF
- Web 内容清洗规则复杂，容易引入额外范围膨胀
- 可以在后续连接器能力成熟后补充

## 统一结构提示类型

`structure_hints` 建议支持以下类型：

| 类型 | 说明 | 常见来源 |
| --- | --- | --- |
| `heading` | 标题 | Markdown / PDF / Web Clip |
| `paragraph` | 段落 | Markdown / PDF / Web Clip |
| `list` | 列表 | Markdown / Web Clip |
| `table` | 表格 | Markdown / PDF / Web Clip |
| `code_block` | 代码块 | Markdown |
| `blockquote` | 引用块 | Markdown / Web Clip |
| `link` | 链接 | Markdown / Web Clip |
| `image_ref` | 图片引用 | Markdown / Web Clip |
| `page` | PDF 页 | PDF |
| `footnote` | 脚注候选 | PDF / Web Clip |
| `header_footer_candidate` | 页眉页脚候选 | PDF |

## 统一 warning 类型

本阶段可能产生以下 warning：

| warning | 说明 | 是否阻断 |
| --- | --- | --- |
| `encoding_detection_failed` | 编码识别失败 | 可能阻断 |
| `frontmatter_parse_failed` | Markdown frontmatter 解析失败 | 不阻断 |
| `markdown_structure_unusual` | Markdown 结构异常 | 不阻断 |
| `pdf_text_extraction_empty` | PDF 文本抽取为空 | 通常阻断 |
| `low_text_density` | PDF 文本密度过低 | 可能阻断 |
| `ocr_required_but_not_supported` | 文档疑似扫描件，需要 OCR，但当前阶段不支持 | 阻断 |
| `pdf_metadata_title_unreliable` | PDF metadata 标题不可靠 | 不阻断 |
| `position_map_incomplete` | 无法完整建立位置映射 | 不阻断，但影响引用定位 |
| `possible_header_footer_noise` | 疑似页眉页脚噪音 | 不阻断 |
| `table_structure_may_be_lost` | 表格结构可能丢失 | 不阻断 |

如果 warning 会影响引用准确性，必须在后续回答或检索结果中保持保守。

## 与后续 section / chunk 的边界

标准化层输出结构提示，但不直接决定最终 section 和 chunk。

示例：

- Markdown 的 `# 一级标题` 会进入 `structure_hints.heading`
- PDF 的页码会进入 `structure_hints.page`
- 表格会进入 `structure_hints.table`

后续 `7.1` 和 `7.2` 再基于这些提示决定：

- section 如何切分
- chunk 如何切分
- 表格是否整体保留
- 代码块是否整体保留
- 是否需要相邻段落合并

## 与引用准确性的关系

输入标准化必须服务于引用准确性。

因此：

- Markdown 应尽量保留行号，并将行号作为第一阶段主要定位方式
- PDF 应至少保留页码和页内文本块序号
- 如果可能，应保留标准化文本偏移量
- 如果位置映射不完整，应记录 warning
- 如果无法定位到具体部分，后续回答应降低置信度或拒绝生成确定性回答

这与项目原则一致：

- 检索成功必须定位到具体部分
- 回答必须带引用
- 宁可保守，不要幻觉

## 最终结论

`6.1` 将输入标准化定义为一个保守、可追溯、结构优先的预处理层。

第一阶段正式支持：

- Markdown 标准化
- 文本型 PDF 标准化

第一阶段预留但不深入实现：

- Web Clip 标准化
- OCR 能力

统一输出为：

- 原始内容引用
- 标准化正文
- 标准化内容引用
- 标题候选
- metadata
- structure hints
- position map
- warnings

该设计为后续 `6.2` 脏数据清洗规则、`7.1` section 切分、`7.2` chunk 切分和引用定位能力提供基础。
