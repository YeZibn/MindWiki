# 6.6 预处理结果对照样例

## 任务目标

为第一阶段预处理链路建立一组可复用的输入输出对照样例，帮助后续开发、联调、测试和验收时快速判断：

- 标准化是否符合预期
- 清洗是否过度或不足
- 正文角色划分是否合理
- OCR 判定是否符合预期
- 去重标记是否正确

本子任务对应路线图中的 `6.6`：

- 建立预处理结果对照样例，便于后续规则迭代

## 使用方式

本文件中的样例用于：

- 开发阶段核对预处理输出结构
- 测试阶段构造基准用例
- 规则调整后做回归比对
- 排查“为什么这份文档处理结果不对”时作为参考

每个样例包含：

- 输入特征
- 预处理前内容摘要
- 预处理后核心输出
- 关键 warning / error
- 预期判定结果

## 样例 1：带 Frontmatter 的 Markdown 正常文档

### 输入特征

- 来源类型：`markdown`
- 文件特点：包含 YAML frontmatter、H1/H2、普通段落、代码块
- 目标：验证标题候选、结构保留、正文角色划分

### 预处理前

```markdown
---
title: RAG 系统设计笔记
tags:
  - rag
  - retrieval
source: 学习资料
---

# 整体架构

RAG 系统由检索层和生成层组成。

## 检索层

检索层负责召回与问题相关的知识片段。

```python
def retrieve(query: str):
    pass
```
```

### 预处理后核心输出

```json
{
  "title_candidates": [
    {"value": "RAG 系统设计笔记", "source": "frontmatter", "confidence": 0.95},
    {"value": "整体架构", "source": "h1", "confidence": 0.80}
  ],
  "metadata": {
    "frontmatter": {
      "tags": ["rag", "retrieval"],
      "source": "学习资料"
    }
  },
  "structure_hints": [
    {"type": "heading", "level": 1, "text": "整体架构"},
    {"type": "heading", "level": 2, "text": "检索层"},
    {"type": "code_block", "language": "python"}
  ],
  "warnings": []
}
```

### 预期判定结果

- `frontmatter.title` 进入标题候选
- frontmatter 不进入正文主文本
- 标题进入 `supporting_structure`
- 正常段落进入 `body`
- 代码块保留结构提示

## 样例 2：含导航块的 Markdown 文档

### 输入特征

- 来源类型：`markdown`
- 文件特点：含自动目录、模板导航、正文内容
- 目标：验证“导航块保留但不激进删除”

### 预处理前

```markdown
# 检索策略总结

[TOC]

> 模板说明：此处用于记录检索实验。

## 向量召回

向量召回适合处理语义相近但关键词不完全一致的问题。
```

### 预处理后核心输出

```json
{
  "structure_hints": [
    {"type": "heading", "level": 1, "text": "检索策略总结"},
    {"type": "navigation_block_candidate", "text": "[TOC]"},
    {"type": "blockquote", "text": "模板说明：此处用于记录检索实验。"},
    {"type": "heading", "level": 2, "text": "向量召回"}
  ],
  "content_roles": [
    {"text": "[TOC]", "role": "supporting_structure"},
    {"text": "模板说明：此处用于记录检索实验。", "role": "body"},
    {"text": "向量召回适合处理语义相近但关键词不完全一致的问题。", "role": "body"}
  ],
  "warnings": ["navigation_block_candidate"]
}
```

### 预期判定结果

- 自动目录块不删除
- 自动目录块标记为 `supporting_structure`
- 引用块默认按 `body` 处理
- 正文段落正常进入主文本

## 样例 3：文本型 PDF 正常导入

### 输入特征

- 来源类型：`pdf`
- 文件特点：可稳定抽取文本、页码清晰、少量页眉页脚
- 目标：验证页码映射、页眉页脚处理、正文抽取

### 预处理前

```text
第 1 页顶部重复：项目资料汇编
第 1 页正文：RAG 系统需要同时支持关键词检索和向量检索。
第 1 页底部：1

第 2 页顶部重复：项目资料汇编
第 2 页正文：混合检索可以提高召回覆盖率。
第 2 页底部：2
```

### 预处理后核心输出

```json
{
  "metadata": {
    "page_count": 2,
    "text_extracted_page_count": 2,
    "empty_text_page_count": 0
  },
  "structure_hints": [
    {"type": "page", "page_number": 1},
    {"type": "page", "page_number": 2},
    {"type": "header_footer_candidate", "text": "项目资料汇编"}
  ],
  "position_map": [
    {"page_number": 1, "page_block_index": 1, "standardized_start": 0, "standardized_end": 28},
    {"page_number": 2, "page_block_index": 1, "standardized_start": 29, "standardized_end": 46}
  ],
  "warnings": ["possible_header_footer_noise"]
}
```

### 预期判定结果

- PDF 进入正常标准化流程
- 页码必须保留
- 高置信度页眉页脚可移除或降权
- 正文进入 `body`
- 后续引用可定位到页码和页内块序号

## 样例 4：扫描版 PDF 触发 OCR 失败

### 输入特征

- 来源类型：`pdf`
- 文件特点：页面几乎全为扫描图片、文本抽取为空
- 目标：验证 OCR 触发条件和失败路径

### 预处理前

```text
共 18 页，文本抽取结果为空，页面仅识别到整页图片对象。
```

### 预处理后核心输出

```json
{
  "ocr_decision": "required",
  "signal_level": "strong",
  "decision_reason_codes": [
    "pdf_text_extraction_empty",
    "ocr_required_but_not_supported"
  ],
  "metrics": {
    "page_count": 18,
    "text_extracted_page_count": 0,
    "empty_text_page_count": 18
  }
}
```

### 预期判定结果

- 判定整份文档需要 OCR
- 第一阶段直接导入失败
- 不允许生成空 document
- CLI 应提示“当前阶段不支持 OCR，后续会增加”

## 样例 5：目录页保留但降权

### 输入特征

- 来源类型：`pdf`
- 文件特点：第一页为目录，后续为正文
- 目标：验证目录页不硬删，但不算正文主干

### 预处理前

```text
目录
1. 系统背景........1
2. 检索策略........3
3. 生成约束........8
```

### 预处理后核心输出

```json
{
  "structure_hints": [
    {"type": "heading", "text": "目录"},
    {"type": "toc_candidate", "text": "1. 系统背景........1"}
  ],
  "content_roles": [
    {"text": "目录", "role": "supporting_structure"},
    {"text": "1. 系统背景........1", "role": "supporting_structure"}
  ],
  "warnings": ["toc_candidate"]
}
```

### 预期判定结果

- 目录内容保留
- 目录内容不算 `body`
- 检索时可低权参与，但不应优先作为回答证据

## 样例 6：脚注与附录作为补充证据

### 输入特征

- 来源类型：`pdf`
- 文件特点：正文后带脚注说明和附录
- 目标：验证 `supporting_content` 的归类

### 预处理前

```text
正文：药物 A 对症状缓解有效。
脚注：样本量较小，结论需谨慎解释。
附录：试验入组标准与排除标准。
```

### 预处理后核心输出

```json
{
  "content_roles": [
    {"text": "药物 A 对症状缓解有效。", "role": "body"},
    {"text": "样本量较小，结论需谨慎解释。", "role": "supporting_content", "subtype": "footnote_candidate"},
    {"text": "试验入组标准与排除标准。", "role": "supporting_content", "subtype": "appendix_candidate"}
  ],
  "warnings": ["footnote_candidate", "appendix_candidate"]
}
```

### 预期判定结果

- 脚注和附录不算正文主干
- 但可作为补充证据参与引用
- 生成回答时优先级低于正文

## 样例 7：同内容多来源的文档级重复

### 输入特征

- 来源类型：`markdown`
- 文件特点：两个不同目录下的文件内容哈希完全相同
- 目标：验证“不同来源但相同内容”的文档级处理

### 预处理前

```text
/notes/rag/design.md
/backup/rag/design-copy.md

两者内容完全一致，哈希相同。
```

### 预处理后核心输出

```json
{
  "document_dedup_result": "same_content_different_sources",
  "sources": [
    "/notes/rag/design.md",
    "/backup/rag/design-copy.md"
  ],
  "action": "keep_both_sources_linked"
}
```

### 预期判定结果

- 不抹平来源关系
- 保留不同 `Source`
- 标记为相同内容的多来源出现

## 样例 8：Chunk `exact_duplicate` 折叠展示

### 输入特征

- 来源类型：`markdown + pdf`
- 文件特点：不同文档中有完全相同的结论段
- 目标：验证检索折叠和多来源引用

### 预处理前

```text
Chunk A：混合检索可以提高召回覆盖率。
Chunk B：混合检索可以提高召回覆盖率。
Chunk C：混合检索通常能提升召回覆盖率，但依赖索引质量。
```

### 预处理后核心输出

```json
{
  "chunks": [
    {"text": "混合检索可以提高召回覆盖率。", "duplicate_type": "exact_duplicate", "duplicate_group_id": "g1", "primary_in_group": true},
    {"text": "混合检索可以提高召回覆盖率。", "duplicate_type": "exact_duplicate", "duplicate_group_id": "g1", "primary_in_group": false},
    {"text": "混合检索通常能提升召回覆盖率，但依赖索引质量。", "duplicate_type": "near_duplicate", "duplicate_group_id": "g2", "primary_in_group": true}
  ]
}
```

### 预期判定结果

- `exact_duplicate` 默认折叠
- `near_duplicate` 保守降冗余，不等同于完全重复
- 回答引用层可展示“来自多个来源”

## 样例 9：低质量乱码段落不静默删除

### 输入特征

- 来源类型：`pdf`
- 文件特点：主体内容可用，但局部出现乱码段
- 目标：验证乱码保留并 warning

### 预处理前

```text
正文：系统应支持引用回溯。
乱码段：#@!% ��� ��� @@
正文：回答中应区分事实依据与生成归纳。
```

### 预处理后核心输出

```json
{
  "content_roles": [
    {"text": "系统应支持引用回溯。", "role": "body"},
    {"text": "#@!% ��� ��� @@", "role": "noise_candidate"},
    {"text": "回答中应区分事实依据与生成归纳。", "role": "body"}
  ],
  "warnings": ["garbled_text_candidate", "low_text_quality"]
}
```

### 预期判定结果

- 局部乱码不静默删除
- 局部乱码不作为核心证据
- 整篇文档仍可继续进入后续链路

## 验收建议

后续开发和测试时，建议至少用这 9 类样例做验收：

- Markdown 标题与 frontmatter 处理
- Markdown 导航块处理
- 文本型 PDF 正常标准化
- 扫描版 PDF OCR 失败
- 目录页角色判断
- 脚注与附录角色判断
- 文档级重复来源处理
- chunk 重复折叠处理
- 乱码保留与 warning 处理

## 最终结论

`6.6` 建立了一组面向实现和验收的预处理对照样例，用于把 `6.1` 到 `6.5` 的抽象规则转化为可核对、可回归、可复盘的具体案例。

这些样例的核心价值是：

- 让预处理规则不只停留在概念层
- 让后续开发可以有明确的目标输出
- 让规则调整后可以快速检查是否引入回归
