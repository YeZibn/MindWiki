# 3.2 主键、唯一键、外键与索引字段定义

## 任务目标

为个人知识库 RAG 项目第一阶段的核心实体定义统一的主键策略、唯一键策略、外键关系和关键索引字段，为后续数据库表结构设计、关系映射和检索优化提供稳定基础。

本子任务对应路线图中的 `3.2`：

- 为每个实体定义主键、唯一键、外键和必要索引字段

## 用户确认结论

基于当前讨论，`3.2` 采用以下正式原则：

- 所有核心实体统一采用 `UUID` 作为主键
- `Document` 不按标题去重，而是按来源关联和内容标识去重
- `Tag.name` 在第一阶段按全局唯一处理
- `Tag` 与 `Document` 通过关联表建模，而不是直接内嵌
- 本任务先定义键与索引，不深入展开删除策略
- 向量索引和全文索引的具体实现细节后续在 `4.x` 中继续展开

## 第一阶段主键策略

### 统一规则

第一阶段所有核心实体统一采用以下主键规则：

- 主键字段名：`id`
- 主键类型：`UUID`

### 适用实体

- `Source`
- `Document`
- `Section`
- `Chunk`
- `Tag`
- `ImportJob`
- `QueryRecord`
- `GeneratedArtifact`
- `Citation`
- `DocumentTag`

### 设计原因

- 本地运行和异步任务执行下更容易保证唯一性
- 不依赖中心化自增序列分配
- 为未来服务化拆分和分布式写入保留空间
- 不暴露业务语义到主键本身

## 第一阶段唯一键策略

### 1. `Source`

#### 唯一键策略

`Source` 第一阶段不强制使用单字段唯一键，而采用业务组合唯一判断。

#### 推荐业务唯一判断因子

- `source_type`
- `source_uri` 或 `file_path`
- `import_method`

#### 设计说明

- 来源对象的“唯一性”依赖上下文，而不是单一字段
- 为未来扩展本地路径、URI、第三方来源保留空间

### 2. `Document`

#### 唯一键策略

`Document` 不按标题唯一，而优先使用以下业务唯一组合之一：

- `source_id + content_hash`
- `source_id + source_native_key`

#### 设计说明

- 标题不稳定，不能作为可靠唯一标识
- 相同标题文档可能对应不同内容
- 内容哈希更适合第一阶段去重场景

### 3. `Tag`

#### 唯一键策略

第一阶段建议：

- `tag_name` 全局唯一

#### 设计说明

- 当前是个人工具场景
- 全局唯一标签可以降低重复标签、同义标签泛滥的复杂度
- 后续如有需要，再扩展为命名空间或作用域唯一

### 4. `ImportJob`

#### 唯一键策略

- 仅使用主键 `id`
- 不额外强制业务唯一键

#### 设计说明

- 每次导入任务都应视为独立执行记录

### 5. `QueryRecord`

#### 唯一键策略

- 仅使用主键 `id`
- 不额外强制查询文本唯一

#### 设计说明

- 同一个问题可能被重复提问
- 查询记录更像事件日志，而不是唯一资源对象

### 6. `GeneratedArtifact`

#### 唯一键策略

- 仅使用主键 `id`
- 不额外强制生成内容唯一

#### 设计说明

- 同一输入在不同时间、不同模型或不同参数下可能生成多个结果

### 7. `Citation`

#### 唯一键策略

- 仅使用主键 `id`
- 不额外强制引用内容唯一

#### 设计说明

- 一个生成结果可引用多个片段
- 相同片段也可能在不同生成结果中重复被引用

## 第一阶段外键关系定义

### 核心关系清单

建议定义以下核心外键关系：

1. `Document.source_id -> Source.id`
2. `Section.document_id -> Document.id`
3. `Chunk.document_id -> Document.id`
4. `Chunk.section_id -> Section.id`
5. `DocumentTag.document_id -> Document.id`
6. `DocumentTag.tag_id -> Tag.id`
7. `Citation.generated_artifact_id -> GeneratedArtifact.id`
8. `Citation.document_id -> Document.id`
9. `Citation.chunk_id -> Chunk.id`
10. `GeneratedArtifact.query_record_id -> QueryRecord.id`（可空）
11. `ImportJob.source_id -> Source.id`（可空）

### 外键关系说明

#### `Document -> Source`

- 每份文档都应关联来源上下文
- 来源是文档的上游信息容器

#### `Section -> Document`

- `Section` 不能脱离 `Document` 独立存在

#### `Chunk -> Document / Section`

- `Chunk` 必须归属于某个文档
- `Chunk` 原则上应尽量归属于某个 section
- 若极少数场景下无法可靠挂到 section，可允许 `section_id` 可空，但仍必须归属于 document

#### `DocumentTag -> Document / Tag`

- `Tag` 与 `Document` 通过关联表连接
- 第一阶段不建议将标签直接内嵌在文档主表中

#### `Citation -> GeneratedArtifact / Document / Chunk`

- 每条引用必须关联一个生成结果
- 每条引用应可定位到具体文档和尽量具体的 chunk

#### `GeneratedArtifact -> QueryRecord`

- 生成结果在第一阶段可选择关联某次查询记录
- 某些摘要或专题归纳也可能不是由检索查询直接触发，因此保持可空

#### `ImportJob -> Source`

- 导入任务通常与来源对象相关
- 由于任务初始化阶段可能尚未完成来源持久化，因此可允许为空

## `DocumentTag` 关联表定义原则

第一阶段明确采用关联表建模：

- `DocumentTag.id`
- `DocumentTag.document_id`
- `DocumentTag.tag_id`

### 设计原因

- 保持文档主表整洁
- 适合后续扩展标签来源、打标时间、权重等字段
- 比直接数组字段更利于关系管理和查询优化

## 第一阶段关键索引字段

### 推荐优先索引字段

第一阶段优先考虑以下业务索引字段：

- `Document.source_id`
- `Section.document_id`
- `Chunk.document_id`
- `Chunk.section_id`
- `ImportJob.status`
- `QueryRecord.created_at`
- `GeneratedArtifact.type`
- `Citation.generated_artifact_id`
- `Citation.chunk_id`
- `Tag.name`

### 索引设计说明

#### `Document.source_id`

- 用于按来源查找文档
- 支撑导入去重和来源追踪

#### `Section.document_id`

- 用于快速获取某文档下的结构层级

#### `Chunk.document_id`

- 用于按文档范围查询 chunk

#### `Chunk.section_id`

- 用于恢复 section 内的 chunk 关系

#### `ImportJob.status`

- 用于快速查询待处理、处理中、失败任务

#### `QueryRecord.created_at`

- 用于历史查询追踪和后续评估

#### `GeneratedArtifact.type`

- 用于区分回答、摘要、专题归纳等生成结果类型

#### `Citation.generated_artifact_id`

- 用于快速取回某次生成结果对应的全部引用

#### `Citation.chunk_id`

- 用于反查某个 chunk 被哪些生成结果引用

#### `Tag.name`

- 用于标签去重和按标签过滤

## 第一阶段逻辑索引说明

除了数据库常规索引，第一阶段还应明确以下逻辑索引概念：

### 1. `Chunk` 向量索引

用于支持：

- 语义检索
- 相似 chunk 召回

### 2. `Chunk` 全文检索索引

用于支持：

- 关键词匹配
- 标题命中
- 内容定位

### 说明

以上索引在本任务中只做逻辑定义，不展开具体数据库产品和索引实现细节。

## 第一阶段暂不在 `3.2` 深入展开的内容

以下内容在本任务中暂不展开：

- 删除策略
- 级联删除还是逻辑删除
- 外键约束是否全部在数据库层强制执行
- 索引是否采用联合索引还是单列索引
- 向量索引和全文索引的具体选型

这些内容将在后续 `3.3` 与 `4.x` 中继续细化。

## 第一阶段设计原则

### 1. 主键统一

所有核心实体统一使用 `UUID`，降低后续演进复杂度。

### 2. 唯一键服务于真实业务语义

不为“形式统一”强行给每个实体设计业务唯一键，而是根据实体本质决定是否需要。

### 3. 外键优先表达强归属关系

先把最稳定的实体依赖关系建起来，再在后续任务中细化约束策略。

### 4. 索引优先服务核心链路

优先服务：

- 导入任务追踪
- 文档结构恢复
- 检索结果定位
- 生成结果引用回溯

## 对后续任务的影响

`3.2` 的结论将直接影响：

- `3.3` 实体关系与删除策略定义
- `4.x` 数据库表结构和索引设计
- `5.x` 导入去重与任务状态设计
- `8.x` 检索过滤条件和范围查询设计
- `10.x` 引用结构落表方式设计

## 最终结论

第一阶段采用“统一 UUID 主键 + 业务语义驱动唯一键 + 强归属外键关系 + 关键链路优先索引”的设计策略：

- 主键稳定统一
- 唯一键避免误用标题等弱标识
- 外键优先支撑文档、chunk、引用和生成结果关系
- 索引优先服务导入、检索、引用和追踪链路
