# 3.3 实体关系与删除策略定义

## 任务目标

定义个人知识库 RAG 项目第一阶段核心实体之间的关系强弱、依赖方向与删除策略，确保文档、检索、生成结果和引用链在数据层面保持一致性与可追溯性。

## 用户确认结论

基于当前讨论，`3.3` 采用以下正式原则：

- 删除策略采用 `混合策略`
- `QueryRecord` 采用 `轻量日志型`
- `Source` 与 `Document` 采用 `强保护策略`
- `Chunk` 不允许成为孤立对象，必须归属于某个 `Document`
- `Citation` 必须指向明确证据对象
- 即使是摘要和专题归纳，也应当存在对应的 `QueryRecord`，可由系统自动创建
- `GeneratedArtifact` 是 `Citation` 的强父对象

## 第一阶段核心关系定义

### 1. `Source -> Document`

#### 关系性质

- `Source` 是 `Document` 的来源上下文
- `Document` 强依赖 `Source`

#### 语义说明

- 每个文档都应有明确来源
- 来源语义包括文件路径、导入方式、来源备注等
- 删除来源会破坏文档的回溯、去重和来源解释能力

#### 删除策略

- 当某个 `Source` 仍被 `Document` 引用时，不允许物理删除
- 第一阶段优先采用：
  - `强保护`
  - `逻辑删除优先`

### 2. `Document -> Section`

#### 关系性质

- `Section` 强依赖 `Document`
- `Section` 不能脱离 `Document` 独立存在

#### 语义说明

- `Section` 用于保留文档结构层级
- 它是文档内部结构对象，而不是可独立管理的知识对象

#### 删除策略

- 删除 `Document` 时，`Section` 跟随逻辑删除

### 3. `Document -> Chunk`

#### 关系性质

- `Chunk` 强依赖 `Document`
- `Chunk` 不允许成为孤立对象

#### 语义说明

- 每个 chunk 必须始终归属于一个明确文档
- chunk 是最小检索和引用单元，但不是独立文档资源

#### 删除策略

- 删除 `Document` 时，`Chunk` 跟随逻辑删除
- 不允许出现 `document_id` 为空的 chunk

### 4. `Section -> Chunk`

#### 关系性质

- `Chunk` 应优先归属于某个 `Section`
- 这是较强依赖关系，但弱于 `Chunk -> Document`

#### 语义说明

- section 关系用于恢复文档结构上下文
- 某些极端场景下，如果无法可靠挂接到 section，可允许 `section_id` 为空
- 但这不影响 `Chunk` 对 `Document` 的强依赖

#### 删除策略

- 删除 `Section` 时，其下 `Chunk` 不直接物理删除
- 若所属 `Document` 仍然存在，系统应优先进行逻辑级重新挂接或在必要时保留 `section_id = null`
- 第一阶段中，若 `Section` 是因 `Document` 删除而消失，则 `Chunk` 由 `Document` 删除策略统一处理

### 5. `Document <-> Tag` 通过 `DocumentTag`

#### 关系性质

- `Tag` 与 `Document` 是弱关系
- 二者通过 `DocumentTag` 关联表连接

#### 语义说明

- 标签是组织信息，而不是文档存在条件
- 标签变化不应反向破坏文档主体

#### 删除策略

- 删除 `Tag` 时：
  - 删除或逻辑删除相关 `DocumentTag`
  - 不影响 `Document`
- 删除 `Document` 时：
  - 删除或逻辑删除相关 `DocumentTag`
  - 不影响 `Tag`

### 6. `QueryRecord -> GeneratedArtifact`

#### 关系性质

- `QueryRecord` 与 `GeneratedArtifact` 之间为弱到中等依赖关系
- `GeneratedArtifact` 应当关联 `QueryRecord`

#### 语义说明

- 问答场景下，生成结果通常由一次查询触发
- 摘要和专题归纳即使不是用户显式检索触发，也应由系统创建对应 `QueryRecord`
- 这样可以统一保留“生成是因为什么请求而发生”的历史语义

#### 删除策略

- 删除 `QueryRecord` 时，不删除 `GeneratedArtifact`
- 第一阶段建议：
  - 保留 `GeneratedArtifact`
  - 解除关联或将 `query_record_id` 置空
- 若系统希望保留更多审计信息，可在生成结果中保留必要快照字段

### 7. `GeneratedArtifact -> Citation`

#### 关系性质

- `GeneratedArtifact` 是 `Citation` 的强父对象
- `Citation` 不应脱离生成结果单独存在

#### 语义说明

- 引用是“某次生成结果的证据明细”
- 若没有对应生成结果，引用本身就失去独立意义

#### 删除策略

- 删除 `GeneratedArtifact` 时，`Citation` 跟随逻辑删除
- 不保留脱离生成结果的孤立引用

### 8. `Citation -> Document / Chunk`

#### 关系性质

- `Citation` 必须指向明确证据
- 第一阶段要求引用目标必须明确

#### 语义说明

- 引用不能是模糊的“来自某些资料”
- 引用应能对应到具体文档和尽量具体的 chunk
- 第一阶段中，引用目标应视为强证据对象

#### 引用要求

- `Citation` 必须指向明确 `Document`
- `Citation` 必须尽量指向明确 `Chunk`
- 对第一阶段而言，引用应以“可定位、可检查、可回溯”为基本原则

#### 删除策略

- 删除 `Document` 时，不立即物理删除 `Citation`
- 应优先将 `Citation` 标记为：
  - 已失效
  - 引用目标不可用
- 删除 `Chunk` 时，同理处理为引用失效，而不是直接让历史生成结果消失

## 第一阶段删除策略总表

### 1. `Source`

- 若仍被 `Document` 引用：禁止物理删除
- 如需失效处理：优先逻辑删除

### 2. `Document`

- 优先逻辑删除
- 级联影响：
  - `Section` 逻辑删除
  - `Chunk` 逻辑删除
  - `DocumentTag` 删除或逻辑删除
  - `Citation` 标记失效
- 不级联删除：
  - `GeneratedArtifact`
  - `QueryRecord`
  - `Tag`

### 3. `Section`

- 一般不作为独立删除主对象
- 若单独删除：
  - 优先保留其下 `Chunk`
  - 必要时将 `Chunk.section_id` 置空

### 4. `Chunk`

- 一般不建议单独物理删除
- 若确需删除：
  - 相关 `Citation` 标记失效
- 不允许删除后留下无所属文档的业务状态

### 5. `Tag`

- 删除 `Tag` 时，只影响 `DocumentTag`
- 不影响 `Document`
- 不影响历史 `GeneratedArtifact`

### 6. `QueryRecord`

- 可视为较轻量日志对象
- 删除时不影响 `GeneratedArtifact`
- 仅解除弱关联

### 7. `GeneratedArtifact`

- 优先逻辑删除
- 删除时：
  - `Citation` 跟随逻辑删除
- 不反向影响：
  - `Document`
  - `Chunk`
  - `QueryRecord`

### 8. `Citation`

- 不建议独立作为高频删除对象
- 若删除，应只影响引用展示，不影响生成结果主体

## 第一阶段关系强弱总结

### 强依赖关系

- `Document -> Source`
- `Section -> Document`
- `Chunk -> Document`
- `GeneratedArtifact -> Citation`

### 中等依赖关系

- `Chunk -> Section`
- `GeneratedArtifact -> QueryRecord`

### 弱依赖关系

- `Document <-> Tag`
- `Citation -> Chunk`
- `Citation -> Document`

说明：

- 这里“弱依赖”不表示不重要，而是表示删除时不直接要求父对象和子对象同时消失
- 对 `Citation` 来说，关键是保留历史生成结果，但允许引用目标失效

## 第一阶段设计原则

### 1. 优先保护知识主链

文档、chunk、引用和生成结果构成了第一阶段最重要的数据主链，应优先保护其可追溯性。

### 2. 优先保护历史可解释性

即使源文档、chunk 或来源对象发生删除或失效，也应尽量保留历史生成结果和引用失效状态，而不是直接抹除历史。

### 3. 删除策略服务于“可回溯”

删除不是单纯的数据清理，而必须服务于：

- 回答可解释
- 引用可审查
- 生成结果可复盘

### 4. 轻量对象允许更灵活处理

如 `QueryRecord` 和 `DocumentTag` 这类对象，可采用相对轻量的删除与解除关联策略。

## 最终结论

第一阶段实体关系与删除策略采用“知识主链强保护、轻量对象弱关联、引用失效优先于历史删除”的设计思路：

- `Source` 强保护
- `Document`、`Section`、`Chunk` 构成核心知识结构链
- `GeneratedArtifact` 保留历史生成结果
- `Citation` 作为证据链接必须明确且可失效
- `QueryRecord` 作为轻量日志存在，不应反向约束生成结果存续
