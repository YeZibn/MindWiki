# 8.8 检索验证样例与验证标准

## 任务目标

为个人知识库 RAG 项目第一阶段建立一组可重复使用的检索验证样例与验证标准，用于验证 `chunk` 级混合检索在标题命中、标签命中、正文命中、语义召回、多语言适配和范围过滤场景下的基础有效性。

本子任务对应路线图中的 `8.8`：

- 准备典型检索样例与验证标准，用于验证召回覆盖率、命中精度与多语言适配性

## 设计原则

第一阶段验证设计采用以下原则：

- 先覆盖典型查询类型，再补充复杂边界样例
- 先做人可解释的验证，再考虑大规模自动化评测
- 先验证召回链路是否成立，再进入后续编排与重排优化
- 验证对象统一落在 `chunk hit` 层

## 一、第一阶段必须覆盖的查询类型

第一阶段建议至少覆盖以下 6 类查询。

### 1. 标题词查询

样例特征：

- query 直接来自文档标题核心词
- query 直接来自 section 标题核心词

验证目标：

- 验证 `document_title` 权重是否有效
- 验证 `section_title` 权重是否有效
- 验证 BM25 对标题词的命中能力

### 2. 标签词查询

样例特征：

- query 来自 `document_tags`
- query 为专题词、组织词、主题词

验证目标：

- 验证标签进入 BM25 后是否能稳定参与召回
- 验证标签词命中不会被正文噪音淹没

### 3. 正文关键词查询

样例特征：

- query 为正文中明确出现的术语、概念词、专有名词、函数名、产品名

验证目标：

- 验证 BM25 基础字面召回能力
- 验证标题未命中时正文仍可支撑精确查找

### 4. 语义改写查询

样例特征：

- query 与原文表述不同
- 但在语义上与目标 chunk 明显接近

验证目标：

- 验证 embedding 召回是否有效
- 验证 BM25 未明显命中时，混合召回仍能返回合理候选

### 5. 中英混合查询

样例特征：

- query 同时包含中文和英文术语
- query 中文描述中包含英文产品名、模型名、库名或协议名

验证目标：

- 验证单一 embedding 模型对中英混合语料的适配性
- 验证 BM25 对英文术语和中文上下文的配合能力

### 6. 范围受限查询

样例特征：

- query 搭配 `tags`
- query 搭配 `source_types`
- query 搭配 `document_scope`
- query 搭配 `time_range`

验证目标：

- 验证强过滤是否在召回前生效
- 验证限制范围后的候选集是否严格受控

## 二、样例组织建议

第一阶段建议按以下结构组织样例集：

### 1. 每类查询至少准备 5 到 10 个样例

这样可以避免只靠个别幸运样例得出错误结论。

### 2. 每个样例都应绑定目标文档或目标 chunk

建议记录：

- `query`
- `query_type`
- `expected_document_ids`
- `expected_chunk_ids`
- `notes`

### 3. 样例应覆盖多来源类型

至少覆盖：

- `markdown`
- `pdf`

如果后续 `web_clip` 进入第一阶段有效数据集，也应补入。

## 三、建议样例结构

第一阶段建议采用如下结构记录验证样例：

```json
{
  "case_id": "case_001",
  "query": "chunk 质量评分",
  "query_type": "title_term",
  "filters": {
    "tags": [],
    "source_types": ["markdown"],
    "document_scope": [],
    "time_range": null
  },
  "expected_document_ids": ["doc_001"],
  "expected_chunk_ids": ["chunk_123", "chunk_124"],
  "notes": "验证 section 标题词在 BM25 和混合召回中的命中情况。"
}
```

## 四、第一阶段评估标准

第一阶段建议优先采用人工判断为主的验证方式，不急于引入复杂量化评测框架。

### 1. `hit@k`

重点观察：

- `hit@1`
- `hit@3`
- `hit@5`

用于判断目标 chunk 是否进入前部候选。

### 2. `top1` 是否合理

重点观察：

- 排名第一的 chunk 是否与 query 高度相关
- 是否存在明显低相关片段压制高相关片段的情况

### 3. `top3` 是否覆盖目标 chunk

重点观察：

- 预期命中的目标 chunk 是否至少出现在前 3 条内

### 4. 命中来源是否符合预期

重点观察：

- 标题词查询是否主要来自标题字段命中
- 标签词查询是否体现标签字段命中
- 语义改写查询是否体现向量通道贡献

### 5. 过滤结果是否严格生效

重点观察：

- 限定 `document_scope` 后是否仍返回范围外结果
- 限定 `source_types` 后是否混入其他来源
- 限定 `tags` 后是否出现不满足标签条件的结果

## 五、第一阶段建议记录的调试信息

为了支持人工判断，验证时建议额外记录：

- `retrieval_mode`
- `vector_score`
- `bm25_score`
- `rrf_score`
- `final_score`
- `match_sources`
- `document_title`
- `section_title`

这样便于定位问题到底出在：

- BM25
- embedding
- 融合层
- 过滤层

## 六、应直接判定为问题的现象

第一阶段若出现以下情况，应直接记为明显问题：

1. 标题词 query 命不中对应标题 chunk
2. 标签词 query 基本不返回带该标签的 chunk
3. 正文关键词 query 对明确出现的术语命中不稳定
4. 语义改写 query 只能依赖 BM25，向量通道几乎无贡献
5. 中英混合 query 明显偏向单一语种，另一语种内容持续失效
6. 指定 `document_scope` 后仍返回范围外文档
7. 指定 `source_types` 后仍返回范围外来源
8. 双路融合后，高相关结果长期被明显低相关 chunk 压制

## 七、第一阶段不强求的内容

以下内容第一阶段不作为 `8.8` 的必需项：

- 大规模自动化评测基准集
- 学习排序训练集
- 精细 NDCG / MRR 统计体系
- query-aware 动态权重对照实验
- 跨版本 embedding 大规模 A/B 测试

这些能力可以在后续评估阶段继续扩展。

## 八、最终结论

`8.8` 第一阶段正式采用以下方案：

- 验证样例至少覆盖 6 类查询：
  - 标题词查询
  - 标签词查询
  - 正文关键词查询
  - 语义改写查询
  - 中英混合查询
  - 范围受限查询
- 每类查询建议准备 5 到 10 个样例
- 每个样例应绑定预期命中文档或 chunk
- 第一阶段评估以人工判断为主
- 重点观察：
  - `hit@1`
  - `hit@3`
  - `hit@5`
  - `top1` 是否合理
  - `top3` 是否覆盖目标 chunk
  - 命中来源是否符合预期
  - 过滤结果是否严格生效
- 若出现明显误召回、过滤失效或双路融合压制高相关结果等现象，应直接记为问题
