# 8.1 检索对象边界定义

## 任务目标

为个人知识库 RAG 项目第一阶段定义统一的检索对象边界，明确 `chunk`、标题、摘要、标签在召回链路中的角色，避免在基础召回阶段并行维护过多索引对象层级。

本子任务对应路线图中的 `8.1`：

- 定义第一阶段检索对象边界，明确 `chunk`、标题、摘要、标签在召回链路中的角色

## 用户确认结论

基于当前讨论，`8.1` 采用以下正式设计：

- 第一阶段统一检索主对象为 `chunk`
- 第一阶段仅 `chunk` 作为独立 embedding 对象
- 第一阶段不为以下对象建立独立向量索引：
  - `title`
  - `summary`
- 标题信号通过 `chunk` embedding 输入拼接吸收
- `summary` 第一阶段不进入主向量召回链路
- 第一阶段 BM25 主对象同样为 `chunk`
- `document_tags` 不作为独立主检索对象，而是作为 `chunk` 级 BM25 字段参与索引

## 一、设计边界

### 1. 为什么第一阶段统一以 `chunk` 为主对象

第一阶段检索层的首要目标是：

- 保持索引对象单一
- 保持召回行为稳定
- 保持候选结果可直接回到证据片段
- 保持后续引用、上下文构建和生成链路一致

`Step 7` 已经明确：

- `chunk` 是最小实用检索单元
- `chunk` 是基础引用单元
- `chunk` 具备正文、结构与定位上下文

因此，第一阶段正式采用：

- `chunk` 作为统一检索主对象

### 2. 为什么不并行采用 document / section / title / summary 多层主对象

如果在第一阶段同时让以下对象并行承担主召回职责：

- `document`
- `section`
- `title`
- `summary`
- `chunk`

会立即引入以下复杂度：

- 多层对象召回结果融合
- 不同对象分数空间比较
- 命中后回落到证据层的转换成本
- 前端和生成层返回单元不统一

因此，第一阶段不采用多层主对象并行召回。

## 二、各类对象在第一阶段的角色

### 1. `chunk`

正式角色：

- 统一检索主对象
- 向量召回主对象
- BM25 召回主对象
- 后续引用与生成的基础候选单元

### 2. `title`

这里的 `title` 包括：

- `document_title`
- `section_title`

正式角色：

- 不作为独立向量索引对象
- 不作为独立 BM25 主召回对象
- 作为增强信号参与 `chunk` 检索

具体方式：

- 在 embedding 中通过输入拼接吸收
- 在 BM25 中作为字段参与 `chunk` 级检索

### 3. `summary`

正式角色：

- 第一阶段不作为独立召回对象
- 第一阶段不进入主向量召回链路
- 可在后续展示、聚合或重排阶段继续利用

原因：

- 摘要属于派生结果
- 容易引入表达漂移
- 第一阶段应优先围绕原始证据片段建立稳定召回层

### 4. `tags`

正式角色：

- 不作为独立主检索对象
- 在 BM25 中作为 `chunk` 级辅助字段参与索引
- 同时保留过滤信号角色

原因：

- 标签更适合作为组织与辅助匹配信号
- 作为独立主对象的收益不足，且会增加融合复杂度

## 三、第一阶段对象边界结论

第一阶段统一采用以下对象边界：

- `chunk`：主检索对象
- `title`：增强信号，不独立召回
- `summary`：派生展示信号，不进入主召回
- `tags`：辅助匹配和过滤信号，不独立召回

## 五、最终结论

`8.1` 第一阶段正式采用以下方案：

- 第一阶段统一检索主对象为 `chunk`
- 第一阶段仅 `chunk` 作为独立 embedding 对象
- 第一阶段不为 `title`、`summary` 建立独立向量索引
- 标题信号通过 `chunk` embedding 输入拼接吸收
- `summary` 第一阶段不进入主向量召回链路
- 第一阶段 BM25 主对象同样为 `chunk`
- `document_tags` 不作为独立主检索对象，而是作为 `chunk` 级辅助检索字段参与索引
