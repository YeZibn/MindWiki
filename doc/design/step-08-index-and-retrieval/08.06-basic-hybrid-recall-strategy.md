# 8.6 基础混合召回策略

## 任务目标

为个人知识库 RAG 项目第一阶段定义基础混合召回策略，明确向量召回与 BM25 召回的执行方式、过滤条件时机、内部召回参数和接口层截断规则，为后续融合层与编排层提供稳定候选集合。

本子任务对应路线图中的 `8.6`：

- 设计基础混合召回策略，明确向量召回、BM25 召回、过滤条件和 top-k 默认参数

## 关联文档

- `08.02-embedding-generation-and-versioning-strategy.md`
- `08.04-bm25-retrieval-fields-and-weighting.md`
- `08.05-unified-retrieval-interface-and-hit-structure.md`

## 用户确认结论

基于当前讨论，`8.6` 采用以下正式设计：

- 第一阶段所有显式过滤条件都优先在召回前执行
- 向量召回与 BM25 召回必须共享同一套过滤条件
- 第一阶段默认内部召回参数采用：
  - `vector_top_k = 30`
  - `bm25_top_k = 30`
- 接口层 `top_k` 与内部召回参数分离
- 第一阶段不做自动扩查
- `retrieval_mode` 执行规则为：
  - `hybrid`
  - `vector_only`
  - `bm25_only`

## 一、设计边界

### 1. 本任务解决的问题

`8.6` 的重点不是定义多路结果如何融合排序，而是先定义：

- 每一路如何执行基础召回
- 过滤条件在什么时候生效
- 内部默认召回参数如何设定
- 接口层和内部召回参数如何分层

因此，本任务不深入展开：

- 多路结果去重策略
- 双命中优先规则
- 分层合并细节
- RRF 或线性融合公式

这些内容将在 `8.7` 中单独讨论。

## 二、过滤条件执行时机

第一阶段正式采用：

- `召回前过滤`

也就是先应用：

- `tags`
- `source_types`
- `time_range`
- `document_scope`

再执行：

- 向量召回
- BM25 召回

### 设计原因

#### 1. 更符合强过滤语义

如果用户显式指定过滤条件，就应先收缩候选空间，而不是先召回再删掉结果。

#### 2. 召回质量更稳定

先过滤后召回，可以确保每一路的 `top-k` 都是在正确候选空间内取得，而不是让后置过滤破坏结果质量。

#### 3. 成本更可控

对于指定文档集合、指定来源类型等场景，先过滤能显著减少后续检索开销。

## 三、两路召回的过滤一致性

第一阶段正式要求：

- 向量召回与 BM25 召回必须共享同一套过滤条件

不允许出现：

- 一路先过滤，另一路不过滤
- 一路按标签过滤，另一路不按标签过滤

### 设计原因

- 保持结果可解释性
- 避免两路候选空间不一致导致融合困难
- 保持 `retrieval_mode` 切换时的行为一致

## 四、内部召回参数

第一阶段建议采用如下默认内部召回参数：

- `vector_top_k = 30`
- `bm25_top_k = 30`

### 设计原因

- 第一阶段尚未引入复杂 rerank 与 query-aware 编排
- 参数过小容易漏召回
- 参数过大容易引入明显噪音
- `30 / 30` 适合作为可调优的初始基线

## 五、接口层 top_k 与内部 top_k 的关系

第一阶段正式采用：

- `vector_top_k` / `bm25_top_k` 作为内部召回参数
- 接口层 `top_k` 作为融合后的最终截断参数

### 说明

这意味着：

- 每一路先按各自内部参数取候选
- 后续完成去重、融合与排序后
- 再按接口层 `top_k` 返回最终结果

### 设计原因

- 将“候选扩展能力”和“最终返回数量”解耦
- 便于后续在不修改接口语义的情况下调优内部召回规模

## 六、retrieval_mode 执行规则

第一阶段建议 `retrieval_mode` 支持：

- `hybrid`
- `vector_only`
- `bm25_only`

### 1. `hybrid`

- 同时执行向量召回与 BM25 召回

### 2. `vector_only`

- 仅执行向量召回

### 3. `bm25_only`

- 仅执行 BM25 召回

### 通用规则

无论采用哪种 `retrieval_mode`：

- 强过滤都必须先执行

## 七、召回不足策略

第一阶段正式采用：

- 召回不足时不做自动扩查

这意味着：

- 若向量召回不足 `vector_top_k`
- 或 BM25 召回不足 `bm25_top_k`
- 系统直接返回当前可得候选

### 设计原因

- 第一阶段优先保证检索行为简单、稳定、可解释
- 自动扩查会引入额外参数与隐式行为
- 扩查策略可留待后续增强阶段继续设计

## 八、基础执行流程

第一阶段基础混合召回流程建议如下：

```text
Receive Query
  -> Apply strong filters
  -> Check retrieval_mode
  -> Run vector retrieval if enabled
  -> Run BM25 retrieval if enabled
  -> Produce two candidate sets
  -> Pass candidate sets to fusion stage
  -> Apply final interface top_k after fusion
```

说明：

- 本任务只定义到“将候选集交给融合阶段”
- 融合阶段的细则在 `8.7` 中展开

## 九、最终结论

`8.6` 第一阶段正式采用以下方案：

- 所有显式过滤条件都优先在召回前执行
- 向量召回与 BM25 召回必须共享同一套过滤条件
- 默认内部召回参数采用：
  - `vector_top_k = 30`
  - `bm25_top_k = 30`
- `vector_top_k` / `bm25_top_k` 属于内部召回参数
- 接口层 `top_k` 属于融合后的最终截断参数
- `retrieval_mode` 执行规则为：
  - `hybrid`：同时执行向量召回与 BM25 召回
  - `vector_only`：仅执行向量召回
  - `bm25_only`：仅执行 BM25 召回
- 第一阶段召回不足时不做自动扩查，直接返回当前可得候选
