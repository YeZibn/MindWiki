# 8.2 Embedding 生成方案与版本管理策略

## 任务目标

为个人知识库 RAG 项目第一阶段确定统一的 embedding 模型策略、版本管理字段、升版触发条件与生效方式，确保向量索引具备可重建、可替换、可追溯的基础。

本子任务对应路线图中的 `8.2`：

- 定义 embedding 生成方案与版本管理策略，明确模型、升版触发条件、active version 与重建方式

## 用户确认结论

基于当前讨论，`8.2` 采用以下正式设计：

- 第一阶段采用 `单一主 embedding 模型`
- 模型需支持 `中英文混合检索`
- 第一阶段不采用中英文双模型方案
- 第一阶段 embedding 主对象为 `chunk`
- 版本字段采用：
  - `embedding_provider`
  - `embedding_model`
  - `embedding_version`
  - `embedding_dim`
- `embedding_version` 表示 `embedding 生成方案版本`
- 检索层同一时刻只允许一个 `active embedding version` 参与主召回
- 第一阶段采用：
  - `手动升版 + 受控重建`
- 系统可提示可能需要升版，但不自动升版

## 一、模型策略

### 1. 第一阶段为什么采用单模型

第一阶段检索基础设施的首要目标是：

- 保证召回行为稳定
- 保证索引构建简单
- 保证问题排查可解释

如果一开始同时采用中文和英文两套 embedding 模型，会立即带来以下复杂度：

- 两套向量索引
- 两套版本管理
- 多语言文档的模型路由规则
- 多路向量召回结果融合
- 排障时更难判断问题来自模型、索引还是融合策略

因此，第一阶段正式采用：

- 单一主 embedding 模型

### 2. 模型选择标准

第一阶段不在设计文档中写死具体供应商或具体模型名，而是定义模型选择标准。

建议模型至少满足：

- 支持中英文混合语料检索
- 适合段落级或 chunk 级文本语义检索
- 向量维度稳定
- 成本与吞吐适合第一阶段离线索引构建
- 支持批量重建与版本切换

## 二、版本管理字段

第一阶段建议至少显式保留以下字段：

- `embedding_provider`
- `embedding_model`
- `embedding_version`
- `embedding_dim`

### 1. `embedding_provider`

表示 embedding 的供应方或推理来源。

### 2. `embedding_model`

表示所使用的 embedding 模型标识。

### 3. `embedding_version`

表示当前使用的 `embedding 生成方案版本`。

正式说明：

- 它不等同于模型名
- 它不是单纯的供应商版本号
- 它表示一整套 embedding 生成方案的版本

### 4. `embedding_dim`

表示当前模型产出的向量维度。

## 三、哪些变化触发升版

第一阶段建议将“触发 `embedding_version` 升级”的变化分为以下 4 类。

### 1. 模型变化

以下变化应触发升版：

- `embedding_provider` 变化
- `embedding_model` 变化
- `embedding_dim` 变化

### 2. 输入文本构造变化

以下变化应触发升版：

- 原先仅使用 `chunk.text`
- 后续改为 `document_title + section_title + chunk.text`
- 或其他会改变 embedding 输入语义构成的文本拼接方式

### 3. 文本标准化策略变化

以下变化应触发升版：

- 是否保留 Markdown 结构标记发生变化
- 是否保留列表符号发生变化
- 是否保留代码块 fence 或表格结构发生变化
- 清洗规则变化导致输入文本显著不同

### 4. Embedding 前预处理规则变化

以下变化应触发升版：

- 截断规则变化
- 长文本裁剪策略变化
- 特殊结构块序列化方式变化

## 四、哪些变化不触发升版

以下变化第一阶段不应触发 `embedding_version` 升级：

### 1. 单个文档内容更新

- 只重建该文档相关 `chunk` 的 embedding
- 不将其视为全局 embedding 方案变更

### 2. `chunk_quality_score` 变化

- 不因质量评分变化而升级 `embedding_version`

### 3. 检索参数变化

例如：

- `top-k` 调整
- 合并规则调整
- 过滤条件调整

这些属于召回层策略变化，而不是 embedding 生成方案变化。

## 五、Active Version 策略

第一阶段正式采用：

- 同一时刻只允许一个 `active embedding version` 参与主召回

这意味着：

- 数据层可以保留多个历史 embedding 版本
- 但主检索层只使用一个当前生效版本

## 六、升版与重建机制

### 1. 升版机制

第一阶段正式采用：

- `手动升版 + 受控重建`

系统可以提示“当前变更可能需要新的 embedding version”，但不自动创建新版本。

### 2. 重建策略

- 文档内容更新：
  - 只重建该文档相关 `chunk` 的 embedding
- `embedding_version` 升级：
  - 重建该版本下全部目标 `chunk` 的 embedding
- 重建完成并验证后：
  - 显式切换 `active embedding version`

## 七、最终结论

`8.2` 第一阶段正式采用以下方案：

- 第一阶段采用支持中英文混合检索的 `单一主 embedding 模型`
- 第一阶段 embedding 主对象为 `chunk`
- 版本管理采用：
  - `embedding_provider`
  - `embedding_model`
  - `embedding_version`
  - `embedding_dim`
- `embedding_version` 表示一整套 `embedding 生成方案版本`
- 模型变化、输入文本构造变化、文本标准化变化和 embedding 前预处理变化都应触发升版
- 单文档内容更新、`chunk_quality_score` 变化和检索参数变化不触发升版
- 检索层同一时刻只允许一个 `active embedding version` 参与主召回
- 第一阶段采用 `手动升版 + 受控重建`，不自动升版
