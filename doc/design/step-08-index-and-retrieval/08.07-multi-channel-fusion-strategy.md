# 8.7 多路召回结果融合方式设计

## 任务目标

为个人知识库 RAG 项目第一阶段定义多路召回结果的融合方式，明确 `vector` 与 `BM25` 两路候选的去重、RRF 排名融合、加权融合与最终排序规则，使混合检索结果具备统一、可解释、可调优的排序基础。

本子任务对应路线图中的 `8.7`：

- 设计多路召回结果的融合方式，明确去重、双命中优先、分层合并与可选增强策略

## 关联文档

- `08.05-unified-retrieval-interface-and-hit-structure.md`
- `08.06-basic-hybrid-recall-strategy.md`

## 用户确认结论

基于当前讨论，`8.7` 采用以下正式设计：

- 第一阶段 `hybrid` 模式默认采用：
  - `RRF + 加权融合`
- 融合前先按 `chunk_id` 合并两路候选
- RRF 固定常数采用：
  - `k = 60`
- 第一阶段最终融合分建议采用：
  - `0.50 * normalized_rrf_score`
  - `0.20 * normalized_vector_score`
  - `0.20 * normalized_bm25_score`
  - `0.10 * dual_hit_bonus`
- `dual_hit_bonus` 用于表达双路同时命中的结果增强
- `hybrid` 模式下最终按 `final_score` 统一排序
- 第一阶段不将以下信号纳入默认融合公式：
  - `chunk_quality_score`
  - `match_sources boost`
  - query-aware 动态权重
  - rerank 分数

## 一、设计边界

### 1. 本任务解决的问题

`8.6` 已经定义了：

- 每一路如何召回
- 过滤条件何时执行
- 内部 `top-k` 默认值

`8.7` 的重点则是：

- 多路候选如何去重
- 两路召回结果如何融合
- 最终统一排序分数如何计算

因此，本任务不再重新定义：

- `vector_top_k`
- `bm25_top_k`
- 过滤条件时机
- `retrieval_mode` 基础执行逻辑

### 2. 适用范围

本任务的默认融合策略只适用于：

- `hybrid`

以下模式不进入多路融合：

- `vector_only`
- `bm25_only`

这两种模式直接按各自通道排序返回。

## 二、融合输入

融合前已得到两路候选：

- `vector_candidates`
- `bm25_candidates`

根据 `8.6` 的基础策略，第一阶段默认最多分别取：

- `vector_top_k = 30`
- `bm25_top_k = 30`

每个候选至少具备以下信息：

- `chunk_id`
- `vector_score` 或 `bm25_score`
- `rank_vector` 或 `rank_bm25`

## 三、去重与候选合并

第一阶段正式采用：

- 按 `chunk_id` 合并候选

这意味着：

- 同一个 `chunk` 在两路候选中出现时，应合并为同一条融合候选
- 不在本任务中做近似重复折叠、相邻 chunk 合并或语义近重复折叠

### 合并后每个候选应至少记录

- 是否命中 `vector`
- 是否命中 `bm25`
- `vector_score`
- `bm25_score`
- `rank_vector`
- `rank_bm25`

## 四、RRF 融合基础分

第一阶段建议先计算：

- `rrf_score`

定义如下：

```text
rrf_score = vector_rrf_part + bm25_rrf_part
```

其中：

```text
vector_rrf_part = 1 / (60 + rank_vector)   if vector hit else 0
bm25_rrf_part   = 1 / (60 + rank_bm25)     if bm25 hit else 0
```

### 设计说明

- `60` 是固定的 RRF 常数
- rank 从 `1` 开始
- 未命中的通道记为 `0`

### 为什么第一阶段引入 RRF

原因包括：

- `vector_score` 与 `bm25_score` 的原始分布通常不可直接比较
- RRF 基于排名而不是原始分值，更适合作为跨通道融合基础
- RRF 对早期多路召回融合是更稳的基线方案

## 五、归一化策略

第一阶段建议对当前融合候选集分别计算：

- `normalized_rrf_score`
- `normalized_vector_score`
- `normalized_bm25_score`

### 建议方式

采用：

- `min-max normalization`

公式如下：

```text
normalized_x = (x - min_x) / (max_x - min_x)
```

### 特殊规则

- 如果某一路所有值相同，则统一记为 `1.0`
- 未命中的通道分数记为 `0`

## 六、双命中奖励

第一阶段正式保留：

- `dual_hit_bonus`

定义如下：

```text
dual_hit_bonus = 1   if vector and bm25 both hit
dual_hit_bonus = 0   otherwise
```

### 设计原因

- 双路同时命中通常代表更强的相关性信号
- 用显式奖励项比隐式规则更容易解释和调试

## 七、最终融合分公式

第一阶段默认采用以下公式：

```text
final_score =
  0.50 * normalized_rrf_score +
  0.20 * normalized_vector_score +
  0.20 * normalized_bm25_score +
  0.10 * dual_hit_bonus
```

### 各项权重含义

#### `0.50 * normalized_rrf_score`

- 让排序主要受跨通道排名一致性驱动

#### `0.20 * normalized_vector_score`

- 保留语义召回强度

#### `0.20 * normalized_bm25_score`

- 保留关键词命中强度

#### `0.10 * dual_hit_bonus`

- 明确鼓励双通道共同支持的结果

## 八、最终排序规则

`hybrid` 模式下，第一阶段正式采用：

- 所有候选按 `final_score` 降序排序
- 再按接口层 `top_k` 截断

第一阶段不再采用：

- 先双命中层
- 再单命中层

的主排序规则。

### 同分时的打破顺序

若出现同分，建议依次采用以下规则：

1. `dual_hit_bonus` 高者优先
2. `normalized_rrf_score` 高者优先
3. `normalized_vector_score` 高者优先
4. `normalized_bm25_score` 高者优先

## 九、返回结构建议

第一阶段建议在 `score_breakdown` 中至少保留以下字段：

- `vector_score`
- `bm25_score`
- `rrf_score`
- `normalized_rrf_score`
- `normalized_vector_score`
- `normalized_bm25_score`
- `dual_hit_bonus`
- `final_score`

### 示例

```json
{
  "vector_score": 0.79,
  "bm25_score": 0.88,
  "rrf_score": 0.0318,
  "normalized_rrf_score": 0.91,
  "normalized_vector_score": 0.76,
  "normalized_bm25_score": 0.84,
  "dual_hit_bonus": 1,
  "final_score": 0.845
}
```

## 十、第一阶段暂不纳入的融合特征

第一阶段不将以下内容纳入默认融合公式：

- `chunk_quality_score`
- `match_sources` boost
- 标签命中额外 boost
- query-aware 动态调权
- rerank 分数

### 原因

- 避免第一阶段融合层过重
- 保持排序逻辑可解释
- 将 query-aware 和 rerank 留给 `Step 9`

## 十一、可选增强路径

在第一阶段默认方案之外，可作为后续增强方向预留：

### 1. 纯 RRF

仅依赖 `rrf_score` 排序，不叠加业务权重。

### 2. 线性归一化融合

例如：

```text
final_score =
  alpha * normalized_vector_score +
  beta  * normalized_bm25_score
```

### 3. Query-aware 动态融合

按 query 类型或命中模式动态调整权重。

### 4. 融合后 rerank

在多路融合候选集之上引入更强的 rerank 模型。

## 十二、最终结论

`8.7` 第一阶段正式采用以下方案：

- `hybrid` 模式默认采用 `RRF + 加权融合`
- 融合前先按 `chunk_id` 合并两路候选
- RRF 固定常数采用：
  - `k = 60`
- 第一阶段默认融合分公式采用：
  - `0.50 * normalized_rrf_score`
  - `0.20 * normalized_vector_score`
  - `0.20 * normalized_bm25_score`
  - `0.10 * dual_hit_bonus`
- `hybrid` 模式下最终按 `final_score` 统一排序
- `vector_only` 与 `bm25_only` 不进入多路融合
- 第一阶段不将 `chunk_quality_score`、`match_sources boost`、query-aware 动态权重和 rerank 分数纳入默认融合公式
