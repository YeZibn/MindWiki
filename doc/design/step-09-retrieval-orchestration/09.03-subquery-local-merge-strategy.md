# 9.3 子任务内多查询结果归并策略

## 任务目标

为个人知识库 RAG 项目第一阶段定义单个 `sub-query` 内部多查询、多通道检索结果的归并策略，明确如何对 `base_query`、`step_back_query` 和 `hyde_query` 产生的结果进行去重、记录命中来源，并使用融合版 RRF 生成子任务级候选集。

本子任务对应路线图中的 `9.3`：

- 设计检索结果归并与邻接整合策略，明确同文档折叠、相邻 chunk 合并和去冗余规则

结合当前讨论，第一阶段正式收敛为：

- `9.3` 只做单个 `sub-query` 内部归并
- `9.3` 暂不做子任务间全局打平

## 关联文档

- `09.02-step-back-and-hyde-expansion-strategy.md`
- `../step-08-index-and-retrieval/08.07-multi-channel-fusion-strategy.md`

## 用户确认结论

基于当前讨论，`9.3` 采用以下正式设计：

- 每个 `sub-query` 会产生 4 路结果：
  - `base_query + bm25`
  - `base_query + vector`
  - `step_back_query + vector`
  - `hyde_query + vector`
- 单个 `sub-query` 内部先按 `chunk_id` 精确去重
- 采用融合版 RRF 作为子任务内部主归并方法
- 融合版 RRF 参数采用：
  - `k = 60`
  - `base_bm25 = 0.35`
  - `base_vector = 0.30`
  - `step_back_vector = 0.20`
  - `hyde_vector = 0.15`
- 每个 `sub-query` 输出独立候选集
- `9.3` 暂不进行子任务间全局合并，保留子任务边界

## 一、设计边界

### 1. `9.3` 解决的问题

`9.3` 的目标是：

- 对单个 `sub-query` 内部的多查询、多通道结果进行归并

### 2. `9.3` 不解决的问题

本任务不负责：

- 子任务间全局打平
- 全局 rerank
- 相邻 chunk 拼接
- 上下文构建

这些内容分别留给 `9.4` 和 `9.5`。

## 二、单个 `sub-query` 内部的 4 路结果

每个 `sub-query` 产生以下 4 路结果：

1. `base_query + bm25`
2. `base_query + vector`
3. `step_back_query + vector`
4. `hyde_query + vector`

## 三、去重规则

第一阶段正式采用：

- 按 `chunk_id` 精确去重

说明：

- 同一个 chunk 命中多路时，只保留一条候选记录
- 不在本任务中做近似重复折叠或跨 chunk 语义折叠

## 四、命中来源记录

每条候选应保留其命中来源，例如：

- `base_bm25`
- `base_vector`
- `step_back_vector`
- `hyde_vector`

同时保留各路 rank 信息：

- `rank_base_bm25`
- `rank_base_vector`
- `rank_step_back_vector`
- `rank_hyde_vector`

## 五、融合版 RRF

单个 `sub-query` 内部建议采用融合版 RRF 计算候选分数：

```text
fused_rrf_score =
  0.35 * 1 / (60 + rank_base_bm25) +
  0.30 * 1 / (60 + rank_base_vector) +
  0.20 * 1 / (60 + rank_step_back_vector) +
  0.15 * 1 / (60 + rank_hyde_vector)
```

规则说明：

- 未命中的路记为 `0`
- `k = 60`

### 设计原因

- `base_query + bm25` 最精确，权重最高
- `base_query + vector` 次高
- `step-back` 是抽象扩展，权重再低
- `HyDE` 是增强召回，噪音风险最高，权重最低

## 六、单子任务候选集输出

对每个 `sub-query`，按 `fused_rrf_score` 排序后，输出独立候选集。

建议输出结构如下：

```json
{
  "sub_query_id": "sq_1",
  "sub_query_text": "Step 8 的职责边界是什么？",
  "base_query": "Step 8 的职责边界是什么？",
  "step_back_query": "检索召回层的职责边界是什么？",
  "hyde_query": "Step 8 主要负责基础索引、召回接口、BM25 与向量混合召回等能力。",
  "candidates": [
    {
      "chunk_id": "chunk_101",
      "document_id": "doc_001",
      "section_id": "sec_008",
      "hit_sources": [
        "base_bm25",
        "base_vector",
        "step_back_vector"
      ],
      "rank_base_bm25": 2,
      "rank_base_vector": 4,
      "rank_step_back_vector": 1,
      "rank_hyde_vector": null,
      "fused_rrf_score": 0.0213
    }
  ]
}
```

## 七、为什么不立即做子任务间合并

第一阶段正式采用：

- `9.3` 暂不进行子任务间全局合并

原因：

- 不同 `sub-query` 对应不同检索目标
- 过早打平会丢失“该 chunk 服务哪个子任务”的关键信息
- 后续 rerank 和 context builder 需要保留子任务结构

## 八、最终结论

`9.3` 第一阶段正式采用以下方案：

- 对每个 `sub-query` 单独执行结果归并
- 单个 `sub-query` 内部先按 `chunk_id` 精确去重
- 使用融合版 RRF 合并 4 路结果：
  - `base_query + bm25`
  - `base_query + vector`
  - `step_back_query + vector`
  - `hyde_query + vector`
- 融合版 RRF 参数采用：
  - `k = 60`
  - `base_bm25 = 0.35`
  - `base_vector = 0.30`
  - `step_back_vector = 0.20`
  - `hyde_vector = 0.15`
- 每个 `sub-query` 输出独立候选集
- `9.3` 暂不进行子任务间全局合并，保留子任务边界，交由 `9.4` 做子任务级 rerank
