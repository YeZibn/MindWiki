# 9.5 Context Builder 截断与拼接策略

## 任务目标

为个人知识库 RAG 项目第一阶段定义 context builder 的截断与拼接策略，明确如何在多个 `sub-query` 的 rerank 结果基础上构造最终给生成层的上下文包，并在保证覆盖度的同时控制 token 预算。

本子任务对应路线图中的 `9.5`：

- 设计上下文构建器的截断与拼接策略，按不同查询意图优先保证语义完整性

结合当前讨论，第一阶段正式收敛为：

- 上下文按 `sub-query` 分段组织
- 每段下面挂对应证据 chunk

## 关联文档

- `09.04-subquery-level-llm-rerank.md`
- `../step-07-chunking-and-enrichment/07.04-offset-and-position-mapping-format.md`

## 用户确认结论

基于当前讨论，`9.5` 采用以下正式设计：

- 输入为多个 `sub-query` 各自的 reranked `top 5`
- 先保留子任务边界，不一开始完全打平
- 每个 `sub-query` 默认先取前 `2` 条代表证据
- 相邻 chunk 仅在：
  - 同一 `sub-query`
  - 同一文档
  - 语义连续
  的条件下做局部合并
- 上下文构建优先保证每个 `sub-query` 都有覆盖
- token 预算采用显式预算控制，但第一阶段不写死具体数值
- 最终上下文按 `sub-query` 分段组织
- 若有剩余 token 预算，优先补高 rerank 分的额外 chunk

## 一、设计边界

### 1. `9.5` 解决的问题

`9.5` 的目标是：

- 把多个 `sub-query` 的 rerank 结果组织成最终给生成层的上下文包

### 2. `9.5` 不解决的问题

本任务不负责：

- 重新做 rerank
- 重新做 query 扩展
- 定义引用载荷字段

## 二、输入结构

第一阶段输入为：

- 多个 `sub-query` 各自的 reranked `top 5`

每条结果至少包含：

- `chunk`
- `rerank_score`
- `rerank_reason`

## 三、保留子任务边界

第一阶段正式采用：

- 先保留子任务边界

这意味着：

- 不一开始就完全打平所有 `sub-query` 的候选
- 先按 `sub-query` 组织证据

### 设计原因

- 保证每个子任务都有证据覆盖
- 避免某个子任务的强结果挤掉其他子任务

## 四、代表证据选择

第一阶段正式采用：

- 每个 `sub-query` 默认先取前 `2` 条代表证据

说明：

- 这 `2` 条作为该子任务的初始证据骨架
- 不默认把 `top 5` 全塞入上下文

## 五、相邻 chunk 合并规则

第一阶段正式采用：

- 相邻 chunk 仅在以下条件下允许做局部合并：
  - 同一 `sub-query`
  - 同一文档
  - 语义连续

不建议：

- 跨 `sub-query` 合并
- 跨文档合并
- 远距离强拼

## 六、覆盖优先级

第一阶段上下文构建优先级正式采用：

1. 先保证每个 `sub-query` 至少有代表证据
2. 再去补强更高价值子任务
3. 最后再考虑同一子任务下的更多相似 chunk

也就是：

- 覆盖优先于冗余堆叠

## 七、token 预算控制

第一阶段正式采用：

- context builder 维护显式总 token 预算

规则：

- 每加入一条 chunk，都检查预算
- 超预算时优先裁掉低分补充证据，而不是代表证据

说明：

- 第一阶段先定机制
- 不在设计文档中写死具体 token 数值

## 八、剩余预算下的补充策略

若有剩余 token 预算，第一阶段正式采用：

- 优先补高 rerank 分的额外 chunk

也就是：

- 每个 `sub-query` 先拿前 `2`
- 若有剩余预算，再按 rerank 分高低补第 `3`、第 `4`

## 九、最终上下文结构

第一阶段正式采用：

- 最终上下文按 `sub-query` 分段组织
- 每段下面挂对应证据 chunk

说明：

- 不是一坨平铺文本
- 而是一个保留子任务结构的上下文包

## 十、建议流程

第一阶段建议流程如下：

1. 输入多个 `sub-query` 的 reranked `top 5`
2. 每个 `sub-query` 先选前 `2` 条代表 chunk
3. 检查同文档相邻 chunk 是否可局部合并
4. 形成初始上下文骨架，保证每个子任务都有覆盖
5. 在剩余预算内，按 rerank 分补充额外 chunk
6. 输出最终上下文包与引用映射

## 十一、最终结论

`9.5` 第一阶段正式采用以下方案：

- 输入为多个 `sub-query` 各自的 reranked `top 5`
- 先保留子任务边界，不一开始完全打平
- 每个 `sub-query` 默认先取前 `2` 条代表证据
- 相邻 chunk 仅在：
  - 同一 `sub-query`
  - 同一文档
  - 语义连续
  的条件下做局部合并
- 上下文构建优先保证每个 `sub-query` 都有覆盖
- token 预算采用显式预算控制，但第一阶段不写死具体数值
- 最终上下文按 `sub-query` 分段组织
- 若有剩余 token 预算，优先补高 rerank 分的额外 chunk
