# 9.1 Query 拆解规则

## 任务目标

为个人知识库 RAG 项目第一阶段定义原始 query 的拆解规则，识别其是否包含多个独立检索目标，并在需要时生成一组可独立检索的 `sub-query`，为后续查询扩展、结果归并和上下文构建提供稳定输入。

本子任务对应路线图中的 `9.1`：

- 定义查询拆解与意图识别规则，明确原始 query 的拆解判断，以及原始 query 或 sub-query 的意图识别与查询计划类型判断

结合当前讨论，第一阶段正式收敛为：

- `9.1` 只负责 query 拆解
- `9.1` 不负责 step-back、HyDE、rerank 或上下文构建

## 用户确认结论

基于当前讨论，`9.1` 采用以下正式设计：

- 顶层只输出：
  - `decomposition_mode = none | decompose`
- 若为 `decompose`，输出：
  - `sub_queries[]`
- 触发拆解的主要场景包括：
  - 比较类
  - 一问多点类
  - 回顾梳理类
  - 多对象总结类
- 默认不拆解的场景包括：
  - 单点事实查询
  - 单点来源查询
  - 单主题总结查询
- 第一阶段拆解约束包括：
  - 只允许一层拆解
  - `sub-query` 默认不超过 `3`
  - 每个 `sub-query` 必须可独立检索
  - `sub-query` 之间尽量低重叠
- 拆与不拆都成立时，优先：
  - `none`

## 一、设计边界

### 1. `9.1` 解决的问题

`9.1` 的目标是判断：

- 原始 query 是否包含多个独立检索目标

如果是，则将其拆成多个可独立检索的 `sub-query`。

### 2. `9.1` 不解决的问题

本任务不负责：

- 生成 `step-back query`
- 生成 `HyDE query`
- 设计 rerank 方案
- 设计上下文拼接

这些内容分别在后续 `9.2`、`9.4`、`9.5` 中展开。

## 二、顶层输出结构

第一阶段顶层只输出：

- `decomposition_mode`

可选值：

- `none`
- `decompose`

### `none`

- 原始 query 作为一个整体检索任务处理

### `decompose`

- 原始 query 需先拆成多个 `sub-query`

若为 `decompose`，还需输出：

- `sub_queries[]`
- `reason`

## 三、触发拆解的主要场景

第一阶段建议仅在以下 4 类场景触发 `decompose`。

## 1. 比较类

典型特征：

- “区别”
- “对比”
- “比较”
- “各自”
- “分别”

示例：

- “BM25 和 embedding 检索有什么区别？”
- “Step 8 和 Step 9 的边界分别是什么？”

## 2. 一问多点类

典型特征：

- 一个问题中同时要求多个输出点

常见连接词：

- “并且”
- “同时”
- “以及”
- “分别说明”

示例：

- “解释 chunk 打分的作用，并说明它如何影响 rerank。”
- “总结 Step 8，并说明它和 Step 9 的关系。”

## 3. 回顾梳理类

典型特征：

- 想看多个历史讨论点或多条结论

常见表达：

- “之前怎么讨论过”
- “有哪些结论”
- “历次提到”
- “整理一下相关内容”

## 4. 多对象总结类

典型特征：

- 同时总结多个对象或多个阶段

示例：

- “分别总结 Step 8 和 Step 9 的职责。”
- “总结 BM25、embedding 和 rerank 的作用。”

## 四、默认不拆解的场景

第一阶段默认不拆的场景包括：

## 1. 单点事实查询

示例：

- “embedding_version 是什么意思？”
- “chunk 质量评分有哪些维度？”

## 2. 单点来源查询

示例：

- “哪篇文档提到了 BM25？”
- “这个规则定义在哪一节？”

## 3. 单主题总结查询

示例：

- “总结一下 Step 8 的设计思路。”

## 五、拆解约束

第一阶段正式采用以下约束：

## 1. 只允许一层拆解

- 原始 query 可拆成 `sub-query`
- `sub-query` 不再递归拆解

## 2. `sub-query` 默认不超过 `3`

原因：

- 超过 `3` 通常说明拆得过碎
- 会明显增加后续检索与 rerank 成本

## 3. 每个 `sub-query` 必须可独立检索

要求：

- 单独看就能理解
- 不能是残句
- 不能依赖上下文代词才能成立

## 4. `sub-query` 之间尽量低重叠

要求：

- 避免多个子问题召回几乎同一批结果
- 每个 `sub-query` 应尽量承载单一主要检索目标

## 六、合格的 `sub-query` 标准

一个合格的 `sub-query` 应满足：

- 单独看就能理解
- 只承载一个主要检索目标
- 保留必要对象名和约束词
- 不主动做抽象化

说明：

- `9.1` 只负责拆解
- 不负责 step-back

## 七、保守策略

第一阶段正式采用：

- 如果“拆”和“不拆”都讲得通，优先 `none`

原因：

- 不拆更简单
- 噪音更小
- 更适合作为第一阶段默认行为

## 八、建议输出示例

```json
{
  "decomposition_mode": "decompose",
  "sub_queries": [
    "Step 8 的职责边界是什么？",
    "Step 9 的职责边界是什么？",
    "Step 8 和 Step 9 的区别是什么？"
  ],
  "reason": "原始 query 同时包含多个独立检索目标。"
}
```

## 九、最终结论

`9.1` 第一阶段正式采用以下方案：

- 只负责原始 query 的拆解判断
- 顶层输出 `decomposition_mode = none | decompose`
- 若为 `decompose`，输出一组 `sub_queries`
- 触发拆解的主要场景包括：
  - 比较类
  - 一问多点类
  - 回顾梳理类
  - 多对象总结类
- 默认不拆的场景包括：
  - 单点事实查询
  - 单点来源查询
  - 单主题总结查询
- 第一阶段只允许一层拆解，且 `sub-query` 默认不超过 `3`
- 每个 `sub-query` 必须可独立检索，且相互尽量低重叠
