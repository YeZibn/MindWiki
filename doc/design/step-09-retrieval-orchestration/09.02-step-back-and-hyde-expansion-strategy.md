# 9.2 Step-Back 与 HyDE 查询扩展策略

## 任务目标

为个人知识库 RAG 项目第一阶段定义对原始 query 或 `sub-query` 的查询扩展策略，明确 `base_query`、`step_back_query` 和 `hyde_query` 的生成规则、使用方式与约束，为后续多查询召回提供稳定输入。

本子任务对应路线图中的 `9.2`：

- 设计查询扩展策略，明确直接查询、step-back query 和 sub-query 的生成方式

结合当前讨论，第一阶段正式收敛为：

- `9.2` 处理原 query 或每个 `sub-query`
- 每个检索单元固定生成：
  - `base_query`
  - `step_back_query`
  - `hyde_query`

- `09.01-query-decomposition-rules.md`

## 用户确认结论

基于当前讨论，`9.2` 采用以下正式设计：

- 每个检索单元固定生成：
  - `base_query`
  - `step_back_query`
  - `hyde_query`
- 默认策略：
  - `step-back` 开启
  - `HyDE` 开启
- 每个检索单元最多生成：
  - `1` 条 `step_back_query`
  - `1` 条 `hyde_query`
- 检索分工采用：
  - `base_query`：`bm25 + vector`
  - `step_back_query`：`vector`
  - `hyde_query`：`vector`
- `step-back` 和 `HyDE` 都不能替代 `base_query`
- `HyDE` 只用于召回增强，不作为最终证据

## 一、设计边界

### 1. 输入对象

`9.2` 的输入对象有两种：

- 若 `9.1` 输出 `decomposition_mode = none`
  - 输入为原始 query
- 若 `9.1` 输出 `decompose`
  - 输入为每个 `sub-query`

因此，`9.2` 始终只处理：

- 当前待检索单元

### 2. 本任务不解决的问题

本任务不负责：

- 多路检索结果归并
- rerank
- 上下文拼接

这些内容分别在 `9.3`、`9.4`、`9.5` 中展开。

## 二、三类查询的角色

## 1. `base_query`

表示：

- 原始 query 或某个 `sub-query`

作用：

- 保留原始问题的具体对象和约束
- 作为最直接、最精确的检索输入

## 2. `step_back_query`

表示：

- 相比 `base_query` 更抽象、但仍同主题的查询

作用：

- 扩大召回面
- 弥补原 query 太窄、太口语或检索锚点不足时的漏召回

## 3. `hyde_query`

表示：

- 基于 `base_query` 生成的一段假设性答案文本

作用：

- 拉近 query 与目标证据表达之间的距离
- 作为向量召回增强输入

## 三、生成规则

第一阶段正式采用：

- 每个检索单元只生成：
  - `1` 条 `step_back_query`
  - `1` 条 `hyde_query`

### 1. `step_back_query` 生成规则

要求：

- 比 `base_query` 更抽象
- 保留核心对象与主题约束
- 不能只是同义改写
- 不能泛化为无边界大话题

### 2. `hyde_query` 生成规则

要求：

- 生成一段假设性答案风格文本
- 文本应贴近“可能的证据表达”
- 不能脱离 `base_query` 的核心对象与主题

## 四、检索分工

第一阶段正式采用以下检索分工：

## 1. `base_query`

用于：

- `bm25`
- `vector`

## 2. `step_back_query`

用于：

- `vector`

## 3. `hyde_query`

用于：

- `vector`

### 设计原因

- `base_query` 最贴近原始用户表达，兼具精确命中和语义检索价值
- `step_back_query` 更适合做抽象语义扩展
- `hyde_query` 是假设性答案文本，更适合作为向量查询输入

## 五、关键约束

第一阶段正式采用以下约束：

## 1. `step-back` 与 `HyDE` 不替代 `base_query`

说明：

- `base_query` 必须保留
- `step_back_query` 与 `hyde_query` 只作为增强召回输入

## 2. `HyDE` 不作为最终证据

说明：

- `hyde_query` 仅用于召回增强
- 最终证据仍只能来自真实 `chunk`

```json
  }
}
```

## 七、最终结论

`9.2` 第一阶段正式采用以下方案：

- 对原 query 或每个 `sub-query` 做独立扩展
- 每个检索单元固定生成：
  - `base_query`
  - `step_back_query`
  - `hyde_query`
- 每个检索单元最多生成：
  - `1` 条 `step_back_query`
  - `1` 条 `hyde_query`
- 检索分工采用：
  - `base_query`：`bm25 + vector`
  - `step_back_query`：`vector`
  - `hyde_query`：`vector`
- `step-back` 和 `HyDE` 都不能替代 `base_query`
- `HyDE` 只用于召回增强，不作为最终证据
