# 5.4 文件去重策略

## 任务目标

定义个人知识库 RAG 项目第一阶段的文件去重、重复导入、内容更新和来源冲突处理策略，确保导入流程能够正确判断“无需处理、文档更新、不同来源重复内容、不同文档”这几类情况。

本子任务对应路线图中的 `5.4`：

- 确定文件去重策略，例如基于路径、哈希、来源 URI 的联合判断

## 用户确认结论

基于当前讨论，`5.4` 采用以下正式策略：

- `content_hash` 用于判断内容是否变化
- `file_path + source_type + import_method` 用于来源定位
- 相同路径、相同 hash：视为重复导入，任务进入 `skipped`
- 相同路径、同类型、hash 变化：视为同一文档更新
- 不同路径、相同 hash：视为不同来源下的相同内容，不自动合并
- 相同路径但文件类型、标题或结构明显变化：视为不同文档，创建新 `Document`
- 发现潜在重复或来源冲突时记录 warning

## 第一阶段去重判断维度

第一阶段使用以下维度共同判断导入对象：

1. `content_hash`
2. `file_path`
3. `source_type`
4. `import_method`
5. 标题变化
6. 文档结构变化

## 1. `content_hash`

### 作用

用于判断文件内容是否发生变化。

### 设计原则

- `content_hash` 是内容变化的最终判断依据
- `mtime` 可以作为预筛选，但不能替代 `content_hash`
- 只要 `content_hash` 未变化，就不应重建 chunk 和索引

## 2. `file_path`

### 作用

用于定位本地来源。

### 设计原则

- `file_path` 是来源定位依据
- 不应单独用 `file_path` 判断内容是否变化
- 同路径不一定永远代表同一文档

## 3. `source_type`

### 作用

用于区分 Markdown、PDF 等来源类型。

### 设计原则

- 相同路径但 `source_type` 变化时，应高度警惕
- 第一阶段将其视为新文档或来源冲突，而不是普通更新

## 4. `import_method`

### 作用

用于区分单文件导入、目录扫描等来源方式。

### 设计原则

- `import_method` 与路径共同构成来源上下文
- 有助于后续判断导入来源和任务行为

## 典型场景处理规则

### 场景 1. 相同路径 + 相同 hash

#### 判断

这是重复导入或重复扫描。

#### 处理策略

- 不重建文档
- 不重建 section
- 不重建 chunk
- 不重建 Qdrant 向量
- 不重建 FTS
- 任务状态记为 `skipped`
- 跳过原因记为 `content_unchanged`

### 场景 2. 相同路径 + 同类型 + hash 变化

#### 判断

这是同一文档发生内容更新。

#### 处理策略

- 视为文档更新
- 使用新的 `content_hash`
- 走文档级增量更新流程
- 旧 section / chunk 逻辑失效
- 重建 section / chunk
- 重建 Qdrant 向量
- 重建 FTS

### 场景 3. 不同路径 + 相同 hash

#### 判断

这是不同来源下的相同内容。

可能原因：

- 用户复制了同一文件
- 不同目录中存在重复资料
- 用户有意保留多个来源副本

#### 处理策略

- 创建新的 `Source`
- 创建新的 `Document`
- 不自动合并为同一文档
- 记录重复内容 warning

#### 设计原因

- 不同路径意味着不同来源上下文
- 自动合并可能误删用户有意保留的资料
- 第一阶段优先保证来源可追溯，而不是激进去重

### 场景 4. 相同路径 + 文件类型变化

#### 判断

这不是普通内容更新，而是来源冲突或替换。

#### 示例

- 原来是 `.md`
- 后来变成 `.pdf`

#### 处理策略

- 视为不同文档
- 创建新 `Document`
- 记录 source conflict warning
- 不强行覆盖旧文档语义

### 场景 5. 相同路径 + 标题明显变化 + hash 变化

#### 判断

可能已经不是原文档，而是同一路径下的替换文件。

#### 处理策略

- 第一阶段视为不同文档
- 创建新 `Document`
- 记录 possible replacement warning

#### 设计原因

- 标题明显变化会影响用户对文档身份的理解
- 强行视为更新可能破坏历史引用语义

### 场景 6. 相同路径 + 结构明显变化 + hash 变化

#### 判断

可能是文档格式或内容形态发生根本变化。

#### 示例

- 原来是 Markdown 笔记
- 后来变成 PDF 报告
- 原来是结构化章节文档
- 后来变成完全不同内容集合

#### 处理策略

- 视为不同文档
- 创建新 `Document`
- 记录 structure changed warning

## 去重结果类型

第一阶段建议将去重判断结果抽象为以下类型：

1. `new_document`
2. `unchanged`
3. `updated_document`
4. `duplicate_content_new_source`
5. `source_conflict_new_document`

### 类型说明

| 类型 | 含义 | 后续动作 |
| --- | --- | --- |
| `new_document` | 全新文档 | 创建 source / document 并导入 |
| `unchanged` | 内容未变化 | skipped |
| `updated_document` | 同一文档更新 | 重建该文档 |
| `duplicate_content_new_source` | 不同来源相同内容 | 新建 source / document，记录 warning |
| `source_conflict_new_document` | 来源冲突或疑似替换 | 新建 document，记录 warning |

## Warning 记录建议

第一阶段建议记录以下 warning：

- `duplicate_content`
- `source_conflict`
- `possible_replacement`
- `structure_changed`
- `title_changed_significantly`

这些 warning 不应阻断导入，但应进入任务结果或日志，便于用户后续排查。

## 与任务状态的关系

### `skipped`

以下情况进入 `skipped`：

- 内容未变化
- 文件为空
- 文件类型不支持

### `success`

以下情况可进入 `success`：

- 新文档成功导入
- 文档更新成功重建
- 重复内容但作为新来源成功导入
- 来源冲突但作为新文档成功导入

### `failed`

以下情况进入 `failed`：

- 读取失败
- 解析失败
- hash 计算失败
- 写入存储失败

## 第一阶段不做的去重能力

第一阶段不做以下能力：

- 跨路径自动合并文档
- chunk 级重复内容合并
- 语义相似去重
- 自动判断两个不同标题文档是否本质相同
- 复杂人工确认流程

## 设计原则

### 1. 内容变化以 hash 为准

标题、路径和 mtime 都不能替代 `content_hash`。

### 2. 来源上下文优先保护

不同路径下的相同内容不自动合并，避免破坏用户来源语义。

### 3. 明显身份变化视为新文档

当标题、类型或结构明显变化时，不强行将其视为同一文档更新。

### 4. 去重保守优先

第一阶段优先避免误合并，而不是追求极致去重。

## 最终结论

第一阶段去重策略采用“内容 hash 判断变化、来源路径保护上下文、明显身份变化视为新文档”的保守策略：

- 不重复重建未变化文档
- 不自动合并不同来源下的相同内容
- 不把明显替换文件强行当作旧文档更新
- 通过 warning 让用户感知重复和冲突
