# 5.1 统一导入入口设计

## 任务目标

定义个人知识库 RAG 项目第一阶段的统一导入入口，明确用户如何通过 CLI 导入 Markdown / PDF 文件、目录如何批量导入、元数据如何传入，以及不支持文件如何处理。

本子任务对应路线图中的 `5.1`：

## 用户确认结论

基于当前讨论，`5.1` 采用以下正式设计：

- 第一阶段导入入口支持：
  - 单文件导入
  - 目录批量导入
- CLI 命令采用显式子命令：
  - `mindwiki import file <path>`
  - `mindwiki import dir <path>`
- 目录导入默认不递归
- 目录递归导入通过 `--recursive` 开启
- 导入元数据通过 CLI 参数传入：
  - `--tag`
  - `--source-note`
- 标题默认自动提取，不强制用户输入
- 不支持的文件跳过并记录，不导致整个目录导入失败

## 第一阶段导入入口范围

第一阶段导入入口覆盖两类场景：

1. 单文件导入
2. 目录批量导入

## 导入入口 1. 单文件导入

### 命令形态

```bash
mindwiki import file <path>
```

### 支持文件类型

- `.md`
- `.pdf`

### 示例

```bash
mindwiki import file ./notes/contract.md --tag work --source-note "合同学习资料"
```

### 处理规则

- 校验文件是否存在
- 判断文件类型是否支持
- 创建导入任务
- 进入导入解析流程
- 返回任务状态或结果摘要

## 导入入口 2. 目录批量导入

### 命令形态

```bash
mindwiki import dir <path>
```

### 支持文件类型

- `.md`
- `.pdf`

### 默认行为

- 默认只扫描当前目录
- 默认不递归扫描子目录

### 递归导入

通过 `--recursive` 开启：

```bash
mindwiki import dir ./notes --recursive
```

### 示例

```bash
mindwiki import dir ./work-docs --tag work --source-note "工作资料目录"
```

### 处理规则

- 校验目录是否存在
- 扫描支持的文件类型
- 为每个支持文件创建导入任务
- 跳过不支持文件
- 汇总输出导入统计

## CLI 参数设计

### `--tag`

#### 说明

为导入文档附加标签。

#### 支持方式

可重复传入：

```bash
mindwiki import file ./a.md --tag work --tag contract
```

#### 处理规则

- 标签不存在时创建
- 标签存在时复用
- 通过 `document_tags` 建立关联

### `--source-note`

#### 说明

为导入来源补充备注。

#### 示例

```bash
mindwiki import file ./a.pdf --source-note "2026 年学习资料"
```

#### 处理规则

- 写入 `sources.source_note`
- 主要用于来源说明和后续追溯

### `--recursive`

#### 说明

目录导入时递归扫描子目录。

#### 示例

```bash
mindwiki import dir ./knowledge --recursive
```

#### 处理规则

- 仅对 `import dir` 生效
- 不对 `import file` 生效

## 标题处理策略

第一阶段标题默认由系统自动提取。

### Markdown 标题提取优先级

建议顺序：

1. YAML frontmatter 中的 `title`
2. 第一个一级标题
3. 文件名

### PDF 标题提取优先级

建议顺序：

1. PDF 元数据标题
2. 第一页明显标题文本
3. 文件名

### 第一阶段不强制用户传标题

原因：

- 减少 CLI 使用负担
- 与前面产品范围中“标题自动提取为主”保持一致

## 不支持文件处理策略

目录导入时，可能出现不支持的文件类型，例如：

- `.docx`
- `.png`
- `.jpg`
- `.xlsx`
- 临时文件

第一阶段策略：

- 跳过不支持文件
- 在导入结果中记录跳过数量
- 可记录跳过文件路径和原因
- 不因为单个不支持文件导致整个目录导入失败

## 导入结果输出建议

### 单文件导入输出

建议输出：

- 文件路径
- 文件类型
- 导入任务 ID
- 当前任务状态
- 错误信息，可空

### 目录导入输出

建议输出：

- 扫描文件总数
- 支持文件数量
- 跳过文件数量
- 创建任务数量
- 失败数量
- 任务 ID 列表或摘要

## 与存储层的关系

导入入口本身不直接写入全部业务对象，而是创建标准化导入请求，后续由导入解析流程完成：

- `sources` 写入
- `documents` 写入
- `import_jobs` 写入
- `sections` / `chunks` 写入
- Milvus 向量写入
- BM25 检索索引写入

## 与后台任务的关系

导入入口负责：

- 接收 CLI 请求
- 校验输入路径
- 创建导入请求
- 返回导入任务信息

后台任务负责：

- 实际文件读取
- 解析
- 切分
- embedding
- 索引写入

## 第一阶段不支持的导入入口

以下入口不纳入第一阶段：

- Notion 导入
- 飞书导入
- 网页剪藏自动同步
- 图片导入
- 音视频导入
- API 上传入口
- Web 上传入口

## 第一阶段设计原则

### 1. 命令清晰优先

采用显式命令：

- `import file`
- `import dir`

避免用户误判系统行为。

### 2. 默认安全优先

目录默认不递归，避免一次性导入大量无关文件。

### 3. 支持文件优先处理

目录中不支持的文件跳过并记录，不阻断整体导入。

### 4. 元数据轻量传入

第一阶段只支持标签和来源备注，避免导入命令过重。

## 最终结论

第一阶段采用“CLI 显式导入入口”的设计：

- 单文件导入使用 `mindwiki import file <path>`
- 目录导入使用 `mindwiki import dir <path>`
- 目录递归通过 `--recursive` 显式开启
- 标签通过 `--tag` 传入
- 来源备注通过 `--source-note` 传入
- 不支持文件跳过并记录
