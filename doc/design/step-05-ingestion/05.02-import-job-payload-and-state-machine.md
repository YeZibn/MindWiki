## 用户确认结论

基于当前讨论，`5.2` 采用以下正式设计：

- 目录导入采用：
  - `一个总任务 + 每个文件一个子任务`
- 第一阶段导入任务状态包括：
  - `pending`
  - `running`
  - `success`
  - `failed`
  - `skipped`
  - `cancelled`
- 失败任务支持重试
- 第一阶段最多自动或受控重试 `2 次`
- 超过重试次数仍失败时，应明确反馈给用户
- `skipped` 用于表示无需继续处理但非异常失败的情况
- `input_payload` 中保留原始请求信息

## 第一阶段导入任务模型

第一阶段建议将导入任务分为两类：

1. 总任务（Batch Job）
2. 文件子任务（File Job）

### 总任务

用于表示一次目录级导入动作的总体上下文。

适用场景：

- `mindwiki import dir <path>`

作用：

- 记录目录导入整体请求
- 汇总扫描结果
- 汇总子任务状态
- 为 CLI 提供总览信息

### 文件子任务

用于表示具体某个文件的导入执行过程。

适用场景：

- `mindwiki import file <path>`
- 目录导入中扫描出的每个支持文件

作用：

- 记录单文件状态
- 记录失败原因
- 记录跳过原因
- 支撑重试与后续排障

## 任务层级关系

### 单文件导入

对于单文件导入：

- 只创建一个文件任务
- 不强制创建总任务

### 目录导入

对于目录导入：

- 创建一个总任务
- 每个支持文件创建一个子任务
- 子任务通过 `parent_job_id` 关联到总任务

## 状态机定义

第一阶段导入任务状态统一采用以下 6 个状态：

1. `pending`
2. `running`
3. `success`
4. `failed`
5. `skipped`
6. `cancelled`

## 状态说明

### 1. `pending`

表示任务已创建但尚未执行。

适用场景：

- CLI 刚创建导入任务
- 目录导入中的子任务尚未开始处理

### 2. `running`

表示任务正在执行中。

适用场景：

- 正在读取文件
- 正在解析、切分、索引

### 3. `success`

表示任务已成功完成。

适用场景：

- 文件已成功导入并完成必要索引写入

### 4. `failed`

表示任务执行失败，且当前未成功完成。

适用场景：

- 文件读取失败
- 解析失败
- embedding 失败
- 写入 PostgreSQL / Qdrant / BM25 检索索引失败

### 5. `skipped`

表示任务被跳过，不属于异常失败，但也未进入有效导入执行。

适用场景：

- 文件类型不支持
- `content_hash` 未变化，无需重建
- 文件为空
- 被规则显式排除

### 6. `cancelled`

表示任务被取消。

适用场景：

- 用户主动取消
- 父任务取消导致子任务终止

## 状态流转建议

### 单文件任务流转

```text
pending -> running -> success
pending -> running -> failed
pending -> skipped
pending -> cancelled
failed -> pending -> running -> success
failed -> pending -> running -> failed
```

### 目录总任务流转

```text
pending -> running -> success
pending -> running -> failed
pending -> running -> cancelled
```

说明：

- 总任务是否 `success`，取决于整体执行结果
- 即使部分文件 `skipped`，总任务仍可 `success`
- 若存在无法接受的失败比例，后续可扩展更细状态判定

## `skipped` 的正式定义

第一阶段中，以下情况统一视为 `skipped`：

1. 文件类型不支持
2. 文件为空
3. 内容未变化，无需重建
4. 导入规则显式排除该文件

### 为什么不是 `failed`

这些情况并不代表系统处理异常，而是：

- 不需要处理
- 不应处理
- 无变化无需重复处理

因此应和真正失败场景分开统计。

## 失败重试策略

### 第一阶段正式规则

- 失败任务支持重试
- 单个任务最多重试 `2 次`
- 超过重试次数仍失败时，向用户明确报告

### 推荐重试范围

建议仅对可恢复性失败重试，例如：

- 临时写入失败
- 模型调用失败
- 索引写入失败

### 不建议自动重试的场景

例如：

- 文件不存在
- 文件格式不支持
- 文件内容为空

这些场景更适合直接进入 `failed` 或 `skipped`。

## `input_payload` 设计原则

第一阶段导入任务应保留原始请求信息，以便：

- 排障
- 重试
- 审计
- CLI 状态展示

### 推荐保留字段

| 字段 | 说明 |
| --- | --- |
| `path` | 用户传入的文件或目录路径 |
| `import_type` | `file` 或 `dir` |
| `recursive` | 是否递归 |
| `tags` | 用户传入标签 |
| `source_note` | 来源备注 |
| `detected_file_type` | 检测到的文件类型 |
| `parent_job_id` | 父任务 ID，可空 |

### 目录总任务 payload 建议

目录总任务可保留：

- 原始目录路径
- 是否递归
- 标签
- 来源备注
- 扫描配置

### 文件子任务 payload 建议

文件子任务可保留：

- 文件路径
- 文件类型
- 所属父任务
- 原始导入参数快照

## 任务追踪字段建议

结合前面的 `import_jobs` 设计，第一阶段建议重点追踪以下字段：

- `id`
- `parent_job_id`（建议后续加入）
- `job_type`
- `status`
- `input_payload`
- `error_message`
- `retry_count`
- `started_at`
- `finished_at`
- `created_at`
- `updated_at`

### 说明

虽然 `4.2` 当前草案中尚未显式列出 `parent_job_id` 和 `retry_count`，但基于本任务结论，后续建议在实现阶段补充。

## CLI 输出建议

### 单文件导入

建议至少输出：

- 任务 ID
- 文件路径
- 当前状态
- 错误信息或跳过原因

### 目录导入

建议至少输出：

- 总任务 ID
- 扫描文件数
- 支持文件数
- 跳过文件数
- 成功任务数
- 失败任务数
- 是否存在仍可重试的失败项

## 最终结论

第一阶段导入任务采用“总任务 + 子任务”的目录导入模型和统一状态机设计：

- 目录导入有总任务
- 每个文件有独立子任务
- 状态统一为 `pending / running / success / failed / skipped / cancelled`
- 失败最多重试 `2 次`
- 原始请求信息保留在 `input_payload`
- `skipped` 与 `failed` 明确区分
