# 5.6 失败重试与人工介入入口设计

## 任务目标

定义个人知识库 RAG 项目第一阶段导入失败后的重试策略、人工介入条件和 CLI 处理入口，确保导入任务失败后用户能够明确知道失败原因，并具备继续处理的路径。

- 设计失败重试规则和人工介入入口

## 用户确认结论

基于当前讨论，`5.6` 采用以下正式设计：

- 可重试错误采用自动重试
- 最多重试 `2 次`
- 重试仍失败后明确告知用户
- 重试间隔采用固定间隔，不采用复杂退避策略
- 提供 CLI 入口查看失败、查看任务详情和手动重试
- 用户可以查看失败原因、重新执行失败任务、跳过失败任务、修改文件后重新导入

## 第一阶段重试策略

### 适用范围

重试策略仅适用于：

- `retryable = true` 的错误
- 任务状态为 `failed`
- 重试次数未超过上限

### 重试上限

第一阶段单个任务最多重试：

- `2 次`

### 重试触发方式

第一阶段采用：

- 自动重试

即：

1. 任务失败
2. 判断错误是否可重试
3. 如果可重试且未超过次数，则自动重试
4. 如果仍失败，则继续重试直到达到上限
5. 达到上限后仍失败，则告知用户

## 重试间隔策略

### 第一阶段正式策略

采用固定间隔重试。

### 设计原因

- 实现简单
- 行为容易理解
- 适合第一阶段 CLI 工具
- 暂不引入复杂指数退避策略

### 说明

具体间隔时长不在本设计文档中写死，可在实现阶段根据任务类型和运行体验配置。

## 自动重试流程

```text
failed
  -> 判断 retryable
  -> 判断 retry_count < 2
  -> 等待固定间隔
  -> retry
  -> success / failed
```

当 `retry_count >= 2` 后仍失败：

- 停止自动重试
- 保持任务为 `failed`
- 输出明确错误信息
- 提供人工处理建议

## 不进行自动重试的情况

以下情况不进行自动重试：

- `retryable = false`
- 文件类型不支持
- 文件不存在
- 文件为空
- 扫描版 PDF 当前无法抽取文本
- embedding 维度不匹配
- 用户主动取消的任务

## 人工介入条件

以下情况应进入人工介入或用户关注路径：

### 1. 不可重试错误

例如：

- 文件不存在
- 文件类型不支持
- 文件为空
- 扫描版 PDF 无法抽取文本

### 2. 超过重试上限仍失败

例如：

- Milvus 多次写入失败
- embedding 模型多次调用失败
- PostgreSQL 写入多次失败

### 3. 来源冲突类 warning

例如：

- `source_conflict`
- `possible_replacement`
- `title_changed_significantly`
- `structure_changed`

这些 warning 不一定导致任务失败，但应在 CLI 输出中提示用户关注。

### 4. 文本质量类 warning

例如：

- `low_text_quality`

这类 warning 可以允许导入成功，但应提醒用户结果可能影响检索和问答质量。

## CLI 人工介入入口建议

第一阶段建议提供以下 CLI 入口：

### 1. 查看失败任务

```bash
mindwiki import failures
```

用途：

- 查看当前失败任务列表
- 查看是否可重试
- 查看错误原因摘要

### 2. 查看任务详情

```bash
mindwiki import show <job_id>
```

用途：

- 查看任务状态
- 查看输入路径
- 查看错误详情
- 查看 warning
- 查看重试次数

### 3. 手动重试单个任务

```bash
mindwiki import retry <job_id>
```

用途：

- 用户确认问题已修复后重试某个任务
- 可用于超过自动重试上限后的人工重试

### 4. 批量重试失败任务

```bash
mindwiki import retry --failed
```

用途：

- 批量重试当前仍可重试的失败任务

## 用户可执行的处理方式

第一阶段用户可执行以下处理：

1. 查看失败原因
2. 修改文件后重新导入
3. 手动重试失败任务
4. 批量重试失败任务
5. 跳过失败任务

## 跳过失败任务

### 设计说明

某些失败任务用户可能暂时不想处理。

第一阶段可提供逻辑上的跳过能力，例如：

- 标记为已忽略
- 或在 CLI 输出中不再默认展示

### 实现建议

具体实现可后续决定，例如：

- 增加 `ignored` 状态
- 或在任务 payload / 状态扩展字段中记录 `ignored = true`

第一阶段设计上先保留这个能力，不强制立即实现。

## 与任务状态机的关系

### 自动重试前

任务状态通常为：

- `failed`

### 自动重试中

任务可重新进入：

- `pending`
- `running`

### 自动重试成功

任务进入：

- `success`

### 自动重试失败且达到上限

任务保持：

- `failed`

并输出：

- 错误类型
- 原因码
- 已重试次数
- 人工处理建议

## 与错误分类的关系

重试策略依赖 `5.5` 中定义的：

- `retryable`
- `error_type`
- `reason_code`

例如：

- `embedding_model_call_failed` 可重试
- `milvus_write_failed` 可重试
- `file_not_found` 不可重试
- `unsupported_file_type` 不可重试

## 最终结论

第一阶段失败处理采用“可重试错误自动重试，固定间隔，最多 2 次，失败后明确交给用户处理”的策略：

- 系统先自动尝试恢复
- 恢复失败后不沉默失败
- CLI 提供查看、重试、批量重试和跳过入口
- warning 不阻断导入，但需要清晰提示用户
