# 5.3 Markdown 与 PDF 接入适配协议

## 任务目标

定义个人知识库 RAG 项目第一阶段对 Markdown 和 PDF 的接入适配协议，明确输入、解析边界、标题抽取规则、统一中间输出对象与失败处理方式，为后续导入解析模块实现提供统一规范。

## 用户确认结论

基于当前讨论，`5.3` 采用以下正式设计：

- 第一阶段只正式定义：
  - `Markdown` 接入适配协议
  - `PDF` 接入适配协议
- 两类文件最终都必须输出统一的中间文档对象
- Markdown 协议支持：
  - 文件路径输入
  - YAML frontmatter
  - 标题结构解析
  - 正文提取
  - Obsidian 双链保留
  - 图片与附件链接保留但不深度解析
- PDF 协议支持：
  - 文件路径输入
  - 文本型 PDF 主路径
  - 页码保留
  - 页面文本抽取
  - 基础段落切分输入
  - 扫描版 / OCR 仅有限支持
- 标题抽取策略采用：
  - Markdown：`frontmatter.title > 第一个 H1 > 文件名`
  - PDF：`首屏明显标题 > PDF metadata title > 文件名`
- 适配失败处理采用：
  - 支持范围内解析失败：`failed`
  - 明确不支持类型：`skipped`
  - 扫描版 PDF 在第一阶段可记为 `failed`，并标注当前仅有限支持

## 第一阶段统一接入目标

Markdown 与 PDF 的适配协议虽然不同，但在进入后续导入解析模块前，必须统一输出为同一种中间文档对象。

这样做的原因：

- 避免后续切分、索引和生成链路区分文件类型写两套逻辑
- 保持导入解析模块边界清晰
- 让后续可以扩展其他来源时沿用同一协议

## 统一中间文档对象

第一阶段建议统一输出一个中间对象，例如：

```text
NormalizedDocument
```

### 推荐字段组成

| 字段 | 说明 |
| --- | --- |
| `source_type` | 来源类型，如 `markdown`、`pdf` |
| `file_path` | 原始文件路径 |
| `title` | 文档主标题 |
| `title_candidates` | 标题候选列表 |
| `raw_text` | 归一化后的完整文本 |
| `metadata` | 元数据字典 |
| `section_candidates` | 结构分段候选 |
| `page_map` | 页码或偏移信息，可空 |
| `warnings` | 警告信息 |
| `adapter_status` | 适配结果状态 |

### 设计原则

- 这是接入层与后续解析层之间的标准对象
- 不是最终落库结构
- 重点用于标准化不同来源文件的输出

## Markdown 接入适配协议

### 输入

- 本地 `.md` 文件路径

### 第一阶段支持内容

- 标准 Markdown 文本
- YAML frontmatter
- 标题、段落、列表、引用块、代码块、表格
- Obsidian 双链 `[[...]]`

### 第一阶段处理规则

- 读取原始 Markdown 文件
- 解析 YAML frontmatter
- 提取正文文本
- 保留 Markdown 结构信息用于 section 候选生成
- 保留 Obsidian 双链文本，不在第一阶段做深度图谱建模
- 图片和附件链接保留为引用文本，不做内容理解

### Markdown 标题抽取规则

Markdown 的“标题”指文档主标题，即进入 `documents.title` 的标题。

第一阶段建议优先级：

1. `frontmatter.title`
2. 第一个 H1
3. 文件名

### Markdown 输出重点

- 文档主标题
- frontmatter 元数据
- 原始正文文本
- 标题层级候选
- 警告信息

### Markdown 警告示例

- 缺少标题
- frontmatter 格式异常
- 存在无法标准解析的结构

## PDF 接入适配协议

### 输入

- 本地 `.pdf` 文件路径

### 第一阶段主支持对象

- 文本型 PDF
- 可直接抽取文本的普通排版文档

### 第一阶段处理规则

- 读取 PDF 文件
- 尝试提取页面文本
- 保留页码映射信息
- 组织为可供后续切分的统一文本
- 为后续 section / chunk 切分提供基础段落输入

### PDF 标题抽取规则

PDF 的“标题”指文档主标题，即进入 `documents.title` 的标题。

第一阶段建议优先级：

1. 首屏明显标题
2. PDF metadata title
3. 文件名

### PDF 输出重点

- 文档主标题
- 全文文本
- 页码映射
- PDF 元数据
- 首屏标题候选
- 警告信息

### PDF 警告示例

- 文本抽取质量低
- 页码无法稳定映射
- metadata 标题缺失

## 扫描版 PDF 与 OCR 处理

### 第一阶段定位

扫描版 PDF 和图片型 PDF 在第一阶段属于：

- `有限支持`

### 第一阶段处理原则

- 不作为主路径支持对象
- 若无法抽取出有效文本，可直接判定适配失败
- 错误信息中应明确提示：
  - 当前版本仅对 OCR 前文本型 PDF 有较好支持

### 为什么不作为主路径

- OCR 复杂度高
- 错误率高
- 与第一阶段“稳而清晰”的目标不一致

## 适配失败与跳过规则

### 记为 `failed` 的情况

以下情况建议记为 `failed`：

- Markdown 或 PDF 本应支持，但解析失败
- 文件内容损坏
- 编码严重异常
- 文本抽取失败
- PDF 为扫描件且当前无法抽取有效文本

### 记为 `skipped` 的情况

以下情况建议记为 `skipped`：

- 文件类型不支持
- 文件为空
- 文件已判定无变化，无需重建

### 设计原则

- `failed` 表示“应该能处理但没处理成功”
- `skipped` 表示“本来就不应进入处理”

## 统一输出与后续模块的关系

接入适配协议负责：

- 识别文件类型
- 读取文件
- 提取可用文本和结构候选
- 生成统一中间文档对象

后续导入解析模块负责：

- 清洗
- section 构建
- chunk 切分
- embedding
- 入库
- 建索引

## 最终结论

第一阶段接入适配协议采用“Markdown / PDF 各自解析、统一中间对象输出”的策略：

- Markdown 支持 frontmatter、正文和结构候选解析
- PDF 主支持文本型文档和页码映射
- 标题抽取规则按文件类型分别处理
- 统一输出 `NormalizedDocument` 风格中间对象
- 支持范围内失败记为 `failed`
- 明确不支持或无需处理记为 `skipped`
